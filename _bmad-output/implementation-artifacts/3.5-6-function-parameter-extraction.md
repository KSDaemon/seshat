# Story 3.5.6: Function Parameter Extraction & Naming Analysis

Status: done

## Story

As a **developer**,
I want the naming detector to analyze function parameter naming conventions,
so that AI agents follow consistent parameter naming across the project. (ADR-29)

## Acceptance Criteria

1. `Function` struct in `seshat-core/src/ir.rs` extended with `parameters: Vec<String>` field
2. Rust tree-sitter parser extracts parameter names from `function_item` and `method_definition` AST nodes
3. TypeScript tree-sitter parser extracts parameter names from `function_declaration`, `arrow_function`, `method_definition`
4. JavaScript tree-sitter parser extracts parameter names (same node types as TS)
5. Python tree-sitter parser extracts parameter names from `function_definition` (excluding `self`/`cls`)
6. Naming detector analyzes parameter name case patterns per language with language-aware weighting
7. Existing tests updated ÔÇö `make_project_file()` test helper produces `parameters: vec![]` by default
8. New tests: parse fixture files, verify parameter names extracted correctly for each language
9. `cargo test --workspace` passes
10. `cargo clippy --all-targets --all-features -- -D warnings` passes

## Dev Notes

### Tree-sitter Node Types per Language

**Rust AST:**
```
function_item Ôćĺ parameters Ôćĺ parameter Ôćĺ pattern (identifier)
impl_item Ôćĺ function_item Ôćĺ parameters (first param may be self_parameter)
```

**TypeScript/JavaScript AST:**
```
function_declaration Ôćĺ formal_parameters Ôćĺ required_parameter Ôćĺ identifier
arrow_function Ôćĺ formal_parameters Ôćĺ ...
method_definition Ôćĺ formal_parameters Ôćĺ ...
```

**Python AST:**
```
function_definition Ôćĺ parameters Ôćĺ identifier (each param)
                                 Ôćĺ default_parameter Ôćĺ identifier
                                 Ôćĺ typed_parameter Ôćĺ identifier
                                 Ôćĺ typed_default_parameter Ôćĺ identifier
```

### What NOT to Extract

- Local variable names ÔÇö too much noise, not enough signal
- Closure/lambda parameter names ÔÇö only named functions
- Type parameter names (generics) ÔÇö separate convention domain

### Architecture Compliance

- IR change in `seshat-core` ÔÇö must update serialization format (may need migration if existing serialized IR is cached)
- Parser changes in `seshat-scanner` ÔÇö parser crate owns AST extraction
- Detector change in `seshat-detectors` ÔÇö detector crate owns naming analysis
- No new dependencies needed
