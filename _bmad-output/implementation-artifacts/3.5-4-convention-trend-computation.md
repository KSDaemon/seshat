# Story 3.5.4: Convention Trend Computation

Status: done

## Story

As a **developer**,
I want each detected convention to have a trend indicator (Rising/Stable/Declining/Unknown),
so that AI agents know whether to adopt or avoid a pattern. (FR63, ADR-24)

## Acceptance Criteria

1. `Trend` enum defined in `seshat-core/src/knowledge.rs`: `Rising`, `Stable`, `Declining`, `Unknown`
2. For each aggregated convention: P90 percentile of `last_commit_date` computed for files where `follows_convention = true`
3. Trend thresholds configurable in `DetectionConfig`: `trend_rising_days: 90`, `trend_stable_days: 365`
4. Mapping: P90 < `trend_rising_days` → Rising, < `trend_stable_days` → Stable, else → Declining, no dates → Unknown
5. Trend stored in `KnowledgeNode.ext_data` as `{"trend": "rising"|"stable"|"declining"|"unknown"}`
6. `aggregate_findings()` in `seshat-detectors/src/confidence.rs` returns trend alongside confidence
7. Unit tests verify correct trend at threshold boundaries (89 days, 90, 91, 364, 365, 366)
8. `cargo test --workspace` passes
9. `cargo clippy --all-targets -- -D warnings` passes

## Dev Notes

### P90 Percentile Calculation

P90 means "90% of the data points are below this value." For file dates sorted ascending, P90 is the date at index `ceil(N * 0.9) - 1`. This means:
- 10 files: P90 = 9th date (index 8)
- 5 files: P90 = 5th date (index 4) — i.e., the newest
- 1 file: P90 = that file's date

This is robust: if 90% of files following a convention were recently modified, the trend is Rising — even if 10% are ancient legacy files.

### Why P90 and not median or mean?

- **Mean** is skewed by outliers (one very old file drags the average)
- **Median** (P50) is too conservative — half the files could be ancient and it would still read "Stable"
- **P90** captures "the vast majority of adoption is recent" — the right signal for "this pattern is gaining traction"

### Integration with scan pipeline

Current flow:
```
discover_files() → parse_file() → run_all_detectors() → aggregate_findings() → KnowledgeNode
```

After Stories 3.5.3 + 3.5.4:
```
discover_files() → collect_git_file_dates() → parse_file() → run_all_detectors()
  → aggregate_findings(findings, file_dates) → KnowledgeNode with trend in ext_data
```

The `file_dates` HashMap must be passed from orchestrator through to `aggregate_findings()`. This is a signature change to `aggregate_findings()`.

### Architecture Compliance

- `Trend` enum in `seshat-core` — shared type
- Trend computation in `seshat-detectors/confidence.rs` — detection logic stays in detectors
- ext_data merge (not overwrite) — preserves existing ext_data fields

### Dependency on Story 3.5.3

This story requires `last_commit_date` in `files_ir` (from Story 3.5.3). The `file_dates` HashMap is built from storage data or passed directly from the orchestrator.
