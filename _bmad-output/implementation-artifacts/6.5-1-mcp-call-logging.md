# Story 6.5.1: MCP Call Logging for Dogfooding

Status: ready

## Story

As a **developer dogfooding Seshat**,
I want all MCP tool calls logged to a JSONL file with full input and response summary metrics,
so that I can analyze tool usage frequency, call sequences, error rates, and validate the tool API surface.

## Background

Seshat is used as an MCP server in its own development (dogfooding), but there is no visibility into how tools are actually called by AI agents. The existing `tracing::info!` calls in `server.rs` go to stderr and are suppressed at the default `warn` level. A dedicated, purpose-built call log (separate from debug tracing) is needed to:

1. **Usage frequency** -- which tools are used, which are dead weight
2. **API correctness** -- are tools called with valid parameters, what's the error rate
3. **Flow analysis** -- do AI agents follow intended patterns (e.g., `query_project_context` first, then `query_convention`)
4. **Response quality** -- are tools returning useful results (via summary metrics)

## Design Decisions

- **Option B chosen**: Dedicated `CallLogger` JSONL writer, NOT a tracing layer. Purpose-built telemetry with clean schema, no debug noise.
- **Opt-in activation**: `seshat serve --call-log [path]` CLI flag + `[server] call_log` in `seshat.toml`. CLI overrides config.
- **Append-only**: File opened with `OpenOptions::create(true).append(true)`. Multiple sessions write to the same file, distinguished by session ID.
- **Full input, summary output**: Log complete request parameters (small), but only summary scalars for responses (not full payload which can be KBs).
- **Session ID**: Short random string generated once at `McpServer::new()`, attached to every log entry.
- **Sequence counter**: `AtomicU64`, monotonic per session. Simplest ordering mechanism.
- **No rotation for V1**: User manages file lifecycle. Rotation can be added later if needed.
- **Architecture Decision**: ADR-30

## Acceptance Criteria

1. New `CallLogger` module at `crates/seshat-mcp/src/call_logger.rs` with `CallLogger` struct and `CallLogEntry` type
2. `CallLogger::new(path: &Path) -> io::Result<Self>` opens file in append mode, creates parent dirs if needed, generates session ID
3. Each of the 5 MCP tool calls produces one JSONL line in the log file when call logging is enabled
4. Log entry schema:
   ```json
   {
     "ts": "2026-04-03T14:22:01.337Z",
     "session": "a1b2c3",
     "seq": 7,
     "tool": "query_convention",
     "input": {"topic": "error handling", "repo": null, "scope": null},
     "duration_ms": 42,
     "status": "ok",
     "result": {"convention_count": 12, "decision_count": 3}
   }
   ```
5. Error entries include `error_code` instead of `result`:
   ```json
   {
     "ts": "...",
     "session": "a1b2c3",
     "seq": 8,
     "tool": "record_decision",
     "input": {"description": ""},
     "duration_ms": 1,
     "status": "error",
     "error_code": "EMPTY_TOPIC"
   }
   ```
6. Tool-specific `result` summary fields:
   - `query_project_context`: `{language_count, convention_count, golden_file_count}`
   - `query_convention`: `{convention_count, decision_count}`
   - `record_decision`: `{node_id}`
   - `update_decision`: `{node_id}`
   - `remove_decision`: `{node_id}`
7. `seshat serve --call-log` (no value) uses default path: `$XDG_DATA_HOME/seshat/call-log.jsonl`
8. `seshat serve --call-log /path/to/file.jsonl` uses specified path
9. `[server] call_log = "path"` in `seshat.toml` also activates logging; CLI flag overrides config
10. When `--call-log` is not passed and config has no `call_log`, no file is created, no overhead
11. `McpServer` struct has `Option<CallLogger>` field -- `None` when logging disabled
12. Log file write failure does NOT crash the server -- errors logged via `tracing::warn!` and call continues
13. Concurrent appends from multiple `seshat serve` processes are safe (JSONL lines < PIPE_BUF)
14. `seq` counter is monotonically increasing per session (AtomicU64)
15. Session ID is a short random alphanumeric string (8 chars)
16. `seshat serve --help` documents the `--call-log` flag
17. All existing tests pass (`cargo test --workspace`)
18. `cargo clippy --all-targets -- -D warnings` passes

## Technical Notes

- The `dirs` crate is already in the workspace for XDG path resolution
- `serde_json` is already a workspace dependency
- `rand` crate may be needed for session ID generation (or use `std::time` hash for simplicity)
- The `Option<Option<PathBuf>>` clap pattern: `None` = flag not passed, `Some(None)` = flag passed without value, `Some(Some(path))` = flag passed with value
- JSONL format: one JSON object per line, newline-separated, no wrapping array
- `Mutex<BufWriter<File>>` is fine for performance -- tool calls are ~30-120ms, lock contention is negligible
