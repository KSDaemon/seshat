# Story 3.5.7: Heuristic Fallbacks for Unknown Libraries

Status: done

## Story

As a **developer**,
I want detectors to identify unknown libraries via heuristics,
so that new or uncommon packages are still classified rather than silently ignored. (ADR-29)

## Acceptance Criteria

1. Error handling detector: Rust derive(Error)/impl Error heuristic + known libs eyre/snafu/miette/error-stack added
2. Error handling detector: Python inheritance-based exception detection (class inherits from `*Error`/`*Exception`)
3. Logging detector: name-based heuristic (contains `log`/`logger`/`logging`/`trace`/`tracing`/`observ`)
4. Logging detector: API shape heuristic (calls to `.info()`/`.debug()`/`.warn()`/`.error()`/`.fatal()`/`.trace()`)
5. Test patterns detector: config file detection (`jest.config.*` → Jest, `vitest.config.*` → Vitest, `[tool.pytest]` → pytest)
6. Test patterns detector: unknown framework fallback (test dir + test-prefixed functions)
7. Dependency usage detector: name-based domain classification for unrecognized packages
8. Export patterns detector: reads `JavaScriptIR::module_system`, flags mixed ESM/CJS
9. All heuristic findings use `KnowledgeWeight::Weak` or `Info` — never `Strong`/`Rule`
10. Known-library findings always override heuristic findings for the same package
11. Unit tests for each heuristic
12. `cargo test --workspace` passes
13. `cargo clippy --all-targets --all-features -- -D warnings` passes

## Dev Notes

### Three-Level Detection Model (ADR-29)

```
L1 (Known):     "tracing" → Logging (High confidence)
L2 (Heuristic): "my-trace-lib" → Logging (Low confidence, name contains "trace")
L3 (Registry):  crates.io category "development-tools::debugging" → may be Logging (Medium confidence)
```

This story implements L2. L1 already exists from Epic 3. L3 is Story 3.5.2.

### Current IR Limitations

The current IR (`ProjectFile`) does NOT track:
- Method call chains (`.info()`, `.debug()`) — we know what's imported but not how it's called
- Derive macro usage per type — `RustIR.error_types` exists but may not capture all derive annotations
- `pyproject.toml` content — we parse it for dependencies but may not expose `[tool.pytest]` section

For each limitation: implement what's feasible with current IR, document what requires IR enhancement as future work.

### Heuristic False Positive Risk

Name-based heuristics WILL produce false positives:
- `logformat` → incorrectly matched as logging (contains "log")
- `test-data-generator` → incorrectly matched as testing (contains "test")

Mitigations:
1. Low confidence weight (`Weak`/`Info`) — never presented as strong conventions
2. Known-library match always overrides — reduces noise for common packages
3. Future: L3 registry metadata provides authoritative classification
