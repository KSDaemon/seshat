# Story 3.5.5: Wrapper/Facade Convention Detection Enhancement

Status: done

## Story

As a **developer**,
I want the dependency usage detector to detect wrapper/facade patterns structurally,
so that direct usage of wrapped external dependencies is flagged as a convention violation. (FR67, ADR-28)

## Acceptance Criteria

1. For each external dependency D: identify internal modules that import D and are re-imported by other project files (potential wrappers)
2. If >50% of files that need domain D's functionality use wrapper module M instead of D directly: establish wrapper convention
3. Files importing D directly when wrapper M exists: `follows_convention = false` with descriptive finding
4. Convention description auto-generated: "Use `{wrapper_module}` for `{external_dep}` operations"
5. No hardcoded directory names (`shared/`, `utils/`, etc.) — purely import graph analysis
6. Works for all 4 supported languages (Rust, TypeScript, JavaScript, Python)
7. Unit tests with fixture projects demonstrating wrapper patterns for at least 2 languages
8. `cargo test --workspace` passes
9. `cargo clippy --all-targets -- -D warnings` passes

## Tasks / Subtasks

- [ ] Task 1: Design wrapper detection data structures (AC: 1, 2)
  - [ ] In `crates/seshat-detectors/src/dependency_usage.rs`, add:
    ```rust
    struct WrapperCandidate {
        wrapper_file: PathBuf,
        external_dep: String,
        wrapper_imports: Vec<String>,  // what the wrapper re-exports
        consumer_count: usize,         // files that import the wrapper
        direct_users: Vec<PathBuf>,    // files that bypass wrapper and import external dep directly
    }
    ```
  - [ ] Detection requires cross-file analysis — the detector needs access to ALL project files, not just one at a time
- [ ] Task 2: Implement cross-file wrapper detection (AC: 1, 2, 3, 5)
  - [ ] The `ConventionDetector` trait's `detect()` receives one file at a time. Wrapper detection needs the full file set. Two options:
    - **Option A:** Add `fn detect_cross_file(&self, files: &[ProjectFile]) -> Vec<ConventionFinding>` to the trait (new optional method with default empty impl)
    - **Option B:** Implement wrapper detection as a post-processing step in the pipeline (after all per-file detectors run)
  - [ ] **Recommended: Option A** — keeps detection logic in the detector, trait gets a default no-op method that only this detector overrides
  - [ ] Algorithm in `DependencyUsageDetector::detect_cross_file()`:
    1. Build import graph: for each file, record `(file_path, imported_package, is_external)`
    2. Identify external dependencies: packages found in manifests
    3. For each external dep D:
       a. Find files that import D directly: `direct_importers = files.filter(|f| f.dependencies_used.any(|d| d.package == D))`
       b. Find files that import D AND are imported by other project files: these are wrapper candidates
       c. A wrapper W satisfies: W imports D, AND other files import W (check `imports` field with resolved paths)
       d. Consumer files: files that import W (not D directly)
       e. If `consumer_count > direct_importer_count` (majority uses wrapper): establish convention
    4. Generate findings:
       - For each direct importer when wrapper exists: `ConventionFinding { follows_convention: false, description: "Use {W} for {D} operations instead of importing {D} directly" }`
       - For each wrapper consumer: `ConventionFinding { follows_convention: true, description: "Uses wrapper {W} for {D}" }`
- [ ] Task 3: Handle import resolution per language (AC: 6)
  - [ ] **Rust:** `use crate::module::submodule` → resolve relative to crate root. External = in Cargo.toml dependencies. Internal = starts with `crate::`, `super::`, `self::`.
  - [ ] **TypeScript/JavaScript:** `import { foo } from './utils/datetime'` → relative path = internal. `import { DateTime } from 'luxon'` → bare specifier = external.
  - [ ] **Python:** `from shared.datetime_utils import utc_now` → starts with project package name = internal. `from datetime import datetime` → stdlib/external. Distinguish: if module path exists as a file in the project = internal.
  - [ ] Use existing `ProjectFile.imports` (has `module` field) and `ProjectFile.dependencies_used` (has `package` field) to distinguish internal vs external
  - [ ] Internal import detection: check if `import.module` resolves to another file in the project file set
- [ ] Task 4: Integrate into detection pipeline (AC: 1)
  - [ ] Update `ConventionDetector` trait in `crates/seshat-detectors/src/trait_def.rs` (or wherever defined):
    ```rust
    fn detect_cross_file(&self, files: &[ProjectFile]) -> Vec<ConventionFinding> {
        Vec::new() // default: no cross-file detection
    }
    ```
  - [ ] Update `run_all_detectors()` in pipeline:
    1. Run per-file detection (existing)
    2. Run cross-file detection for each detector that overrides it
    3. Merge findings
  - [ ] Only `DependencyUsageDetector` overrides `detect_cross_file()` initially
- [ ] Task 5: Tests (AC: 7, 8, 9)
  - [ ] **Python wrapper test:** Create fixture with:
    - `shared/datetime_utils.py` importing `datetime` and exporting `utc_now()`

    - 2 files importing `from datetime import datetime` directly (violate convention)

  - [ ] **TypeScript wrapper test:** Create fixture with:

## Dev Notes

### The PR #589 Pattern (Real-World Example)

From `walt-chat-backend` PR #589 review:
- `shared/datetime_utils.py` wraps `datetime.datetime.now(tz=UTC)` as `utc_now()`
- 40+ files use `utc_now()` from the wrapper (convention)
- 8 files use `datetime.now(tz=UTC)` directly (violation — caught by human reviewer)
- Seshat should catch this automatically after this story

Similarly: `shared/concurrency.py` wraps `asyncio.get_event_loop()` as `run_with_timeout()`.

### Cross-File Detection Challenge

The `ConventionDetector` trait is designed for per-file detection. Wrapper detection is inherently cross-file. The solution (Option A: new trait method) is the cleanest because:
- Backward compatible (default no-op)
- Keeps detection logic in the detector
- Pipeline already has all files available

### Import Resolution Limitations

We do NOT have a full module resolution system. We rely on:
- `ProjectFile.imports[].module` — the raw import path string
- `ProjectFile.dependencies_used[].package` — the external package name

For wrapper detection, we need to distinguish "internal module" from "external package." Heuristic:
- If `import.module` starts with `.` or `./` or `../` → internal (JS/TS)
- If `import.module` starts with `crate::` or `super::` or `self::` → internal (Rust)
- If `import.module` matches a file path in the project → internal (Python — check against known file paths)
- Otherwise → external

This heuristic won't catch all cases (e.g., Python absolute imports with `src.utils.datetime`) but covers the majority.

### Architecture Compliance

- All code in `seshat-detectors` — detector crate owns detection logic
- Trait extension is backward-compatible — existing detectors unchanged
- No hardcoded paths — pure structural analysis from import data
- Follows ADR-28 decision exactly

### Dependency on Story 3.5.1

Uses unified `DependencyDomain` for domain identification. Wrapper detection groups by domain: "wrapper for HTTP dependency", "wrapper for datetime dependency."
