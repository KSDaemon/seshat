# Story 3.5.2: Package Registry Metadata Integration

Status: ready-for-dev

## Story

As a **developer**,
I want dependency domain classification to use package registry metadata instead of hardcoded name lists,
so that new packages are correctly categorized without code changes. (FR68, ADR-25)

## Acceptance Criteria

1. New `package_metadata` SQLite table stores cached registry metadata per package
2. `PackageRegistryClient` trait defined with implementations for crates.io, npm, PyPI
3. For each dependency during scan: lookup cache first, fetch from registry API on miss, fall back to hardcoded mapping
4. Registry categories/keywords/classifiers mapped to unified `DependencyDomain` (~30 mapping rules)
5. Cache TTL of 30 days — stale entries re-fetched on next scan
6. Offline operation works with graceful fallback to hardcoded mapping (lower confidence)
7. HTTP client: `ureq` (blocking, minimal deps) added to `seshat-scanner`
8. Existing hardcoded `classify_*()` functions preserved as fallback tier
9. `cargo test --workspace` passes (with mock HTTP responses in tests)
10. `cargo clippy --all-targets -- -D warnings` passes

## Dev Notes

### Registry API Details

**crates.io:** `GET https://crates.io/api/v1/crates/{name}`
- Response: `{ "crate": { "categories": ["web-programming::http-client"], "keywords": ["http", "rest"] } }`
- Rate limit: 1 req/sec (User-Agent required)
- Full category list: https://crates.io/categories

**npm:** `GET https://registry.npmjs.org/{name}`
- Response: `{ "keywords": ["http", "rest", "client"] }`
- No auth required for public packages
- Response can be large — only parse `keywords` and `description` fields

**PyPI:** `GET https://pypi.org/pypi/{name}/json`
- Response: `{ "info": { "classifiers": ["Topic :: Internet :: WWW/HTTP"], "keywords": "http,rest,client" } }`
- Keywords is comma-separated string (not array!)
- Classifiers are structured: `"Topic :: Category :: Subcategory"`

### Architecture Compliance

- `PackageRegistryClient` trait in `seshat-scanner` — scanner owns the data pipeline
- `PackageMetadataRepository` in `seshat-storage` — storage owns persistence
- `DependencyDomain` in `seshat-core` — from Story 3.5.1
- HTTP calls only during scan (never during serve) — blocking `ureq` is fine
- No new crate — all code in existing scanner and storage crates

### Dependency on Story 3.5.1

This story requires the unified `DependencyDomain` enum from Story 3.5.1. Implement 3.5.1 first.

### Testing Strategy

Use a trait-based mock for `PackageRegistryClient` in tests:
```rust
struct MockRegistryClient { responses: HashMap<String, PackageMetadata> }
impl PackageRegistryClient for MockRegistryClient { ... }
```
This avoids real HTTP calls in tests while testing the full lookup pipeline.
