## Codebase Patterns
- Repository pattern: Trait in `repository/mod.rs`, impl in `repository/{name}_repository.rs`, struct wrapping `Arc<Mutex<Connection>>` with `new()` + `conn()` helper, `row_to_{name}()` private mapper function
- Migrations: `V{N}__{description}.sql` in `crates/seshat-storage/migrations/`, compiled at build time by `refinery::embed_migrations!("migrations")`
- Tests: `Database::open(":memory:")` for in-memory DB, test helper creates repo instance, no external dependencies
- Wiring: New repo modules declared in `repository/mod.rs` (mod + pub use), traits and types re-exported from `lib.rs`
- Pre-commit hook: runs `cargo fmt` then `cargo clippy`. If fmt modifies files, commit is rejected ‚ÄĒ stage the formatted files and retry
- Import ordering: `rusqlite::{params, Connection}` (alphabetical) is the canonical `cargo fmt` order
- DB path layout: root DB at `repos/{project_name}.db`, submodule DBs at `repos/{project_name}/{mount_path}.db`
- ScanConfig uses `exclude_submodules` (not include_submodules) ‚ÄĒ default false means submodules ARE scanned
- Root discovery ALWAYS excludes submodule dirs (they get their own DBs); `exclude_submodules` controls whether separate scans happen
- `detect_submodule_paths()` is pub in `seshat-scanner::discovery` and re-exported from `seshat-scanner::lib.rs`
- ScanProgress enum is in `seshat-scanner::orchestrator`, re-exported from `seshat-scanner::lib.rs`; new submodule variants added in US-003
- `get_submodule_commit_hash()` uses `gix::open()` + `head_commit()` ‚ÄĒ works with both `.git` files (submodule pointers) and `.git` directories
- CLI progress handling in `scan.rs` uses `RefCell<HashMap<String, ProgressBar>>` for per-submodule spinners keyed by mount path
- Submodule scan flow in `scan.rs::run_scan()`: detect submodules ‚Üí scan each into own DB ‚Üí run root scan ‚Üí update submodules table + repo_metadata in root DB
- `resolve_submodule_db_path()` no longer has `#[allow(dead_code)]` ‚ÄĒ it's used by the submodule scan flow
- Submodule scan reuses `scan_project_with_progress()` with the submodule's abs path and dedicated DB ‚ÄĒ no new orchestrator logic needed
- After scanning each submodule: run convention detectors, persist conventions, write repo_metadata (parent_project, mount_path, file_count, convention_count, last_scan_time)
- Root DB submodules table: upsert pattern (try update first, insert if NotFound); stale entries removed when mount_path no longer in .gitmodules
- Root DB repo_metadata: project_name, file_count, convention_count, last_scan_time written after root scan completes
- Change detection: before scanning a submodule, compare `get_submodule_commit_hash()` with stored `commit_hash` from root DB's `submodules` table via `find_by_path()`; skip scan if both are Some and equal
- Skipped (up-to-date) submodules are still tracked in `scanned_submodules` Vec so they aren't deleted from the root DB during cleanup
- Rust edition 2024: `ref` in pattern bindings is implicit when matching on references ‚ÄĒ don't use `Some(ref x)` when matching on `&Option`, use `Some(x)` instead
- Parallel submodule scanning: uses `std::thread::scope` (not rayon) with `indicatif::MultiProgress` for thread-safe concurrent spinner lines; each thread opens its own DB connection
- Pre-filter pattern: change detection + initialization checks run on the main thread (fast), only actual scans are parallelized via `std::thread::scope`
- MCP ErrorCode enum in `envelope.rs`: add new variants to both the enum definition AND the `as_str()` match + the 2 test arrays (`all_error_codes_serialize_screaming_snake` and `error_code_as_str_matches_serde`)
- `ProjectConnection` struct in `scope.rs`: holds `Arc<Mutex<Connection>>` + `name: String` + `branch: String` ‚ÄĒ the building block for multi-DB MCP routing
- `resolve_scope()` priority: explicit scope > file_path prefix (longest wins) > default root; short name fallback uses last path segment with ambiguity check
- `scope.rs` is a pure logic module ‚ÄĒ no McpServer dependency, fully testable with in-memory DBs
- McpServer struct: `root: ProjectConnection` + `submodules: HashMap<String, ProjectConnection>` + `mount_paths: Vec<String>` ‚ÄĒ replaces the old `conn`/`repo_name`/`branch` fields
- McpServer::new() takes `ServerConfig`, `ProjectConnection` (root), `HashMap<String, ProjectConnection>` (submodules); empty HashMap = single-project backward-compatible mode
- All 5 tool handlers: `resolve_scope(scope, file_path)` at the top of each handler, use returned `(conn, name, branch, scope_name)` for DB access and envelope
- `ResponseEnvelope::success()` takes an explicit `scope` parameter (no longer hardcoded to `"root"`)
- Tool handler `fn handle()` functions take `scope_name: &str` parameter ‚ÄĒ the resolved scope name goes into the response envelope
- `start_stdio()` and `start_stdio_with_shutdown()` accept `ProjectConnection` + `HashMap` instead of raw `conn`/`repo_name`/`branch`
- `serve.rs` loads submodules table from root DB, opens each submodule DB via `resolve_submodule_db_path()`, builds `HashMap<String, ProjectConnection>`, passes to `McpServer::new()`
- `serve.rs` gracefully handles missing submodule DBs: logs warning to stderr, skips that submodule, continues with others
- `print_startup()` shows submodule count and sorted names when submodules are present
