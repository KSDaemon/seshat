## Codebase Patterns
- Edition 2024, Rust 1.85+. `let-else`, `..=` ranges, and similar modern syntax are fine.
- `seshat-graph/src/dependencies.rs` already has a `SuffixIndex` (O(1) reverse-suffix lookup) and an `infer_package_root` heuristic for cross-crate guards. Reuse instead of rolling new ones.
- IR `path` is stored as an absolute `PathBuf`; `target_path` arguments to public APIs are typically relative — always normalise via `normalize_path` + `suffix_matches_at_boundary` before comparing.
- For new pure-graph utilities (BFS, adjacency builders), construct synthetic `HashMap<String, Vec<…>>` directly in unit tests; no DB / IR roundtrip needed. The repo-level `test_helpers::{test_conn, insert_ir}` is for integration-style tests.
- Rust style: snake_case fns/params, PascalCase types, `expect("message")` over `unwrap()` in tests, doc comments on every `pub` item with `///`.
- `DependentEntry` and similar serde types use `#[serde(default)]` on newly-added fields for forward compatibility within the same schema version.
- Clippy is run as `cargo clippy --all-targets -- -D warnings` and CI gates on it — no warnings tolerated.
- `DependentEntry` / `DependencyEntry` JSON field is `file_path` (NOT `path`). Tests indexing into the response envelope must use `dep["file_path"]`, since `DependencyData.target` is the only top-level `path`/`target` string.
- MCP `Edit replace_all` over a fragment that ends in `}));` matches every request struct in `server.rs`, not just the one you intend. Always anchor the `old_string` to a unique sibling line (e.g. the `path:` value) when bulk-rewriting in a multi-tool integration test file.
- `crate::test_helpers` in `crates/seshat-graph/src/lib.rs` is `pub(crate)`, so integration tests under `crates/seshat-graph/tests/` cannot import `test_conn`/`insert_ir` directly. Replicate them inline in the integration test file (5-10 LOC each); do not change the visibility of `test_helpers` just to share two helpers — it would leak test-only API into the public crate surface.
- `DependencyData.truncated` reflects **IR-load** truncation (the `LIMIT` cap when reading `files_ir`), NOT the BFS `MAX_DEPENDENTS` cap. The BFS `TransitiveResult.truncated` flag is currently dropped at `dependencies.rs:329-332` (only `result.entries` is captured). To assert that BFS truncated, check `dependents.len() == MAX_DEPENDENTS` — exposing BFS truncation through the public API is a separate concern not required by US-005.
- Bulk-inserting many IR rows in tests: open a single rusqlite `Transaction` and reuse one `Statement::prepare`d `INSERT` across all rows (~30× faster than `insert_ir`-per-row in default test profile — 90 ms for 3000 rows vs ~3 s).
- IR types (`Export`, `TypeDef`, `Function`) are constructed in MANY places — not just parsers. Detector tests, graph tests, MCP tool tests, and server integration tests all build `ProjectFile` fixtures by hand. When adding a required field (no `#[serde(default)]` shortcut), expect ~30 callsites across `seshat-detectors/`, `seshat-graph/`, `seshat-mcp/`, `seshat-scanner/`, plus the storage roundtrip fixtures. A small Python script that finds the `<TypeName> { ... line: N, ... }` block and inserts `end_line: N,` after `line:` (preserving indentation, skipping blocks that already have `end_line:`) was ~30 LOC and rewrote 9 files in a single pass.
- For schema bumps (e.g. `IR_SCHEMA_VERSION` 7→8): always add a guard test (`current_schema_version_is_v8`) that pins the constant to its expected literal value AND that a fresh serialize emits that byte. Without it, an accidental future bump silently passes the version-prefix tests because they all reference `IR_SCHEMA_VERSION` as a variable.
- Tree-sitter parsers expose start/end positions through `node.start_position().row + 1` / `node.end_position().row + 1` (1-based line numbers). For `export_statement` and Rust `struct_item`/`enum_item`/etc., this gives the full source range of the declaration — perfect for hunk-intersection logic. The existing `Function::end_line` (rust_parser.rs:456) is the canonical pattern; mirror it everywhere.
- `extract_exported_lexical` in `parser/mod.rs` is shared between TS and JS. Adding a parameter there fans out to BOTH parsers — so `end_line` flows through the export_statement node uniformly. The function already has `#[allow(clippy::too_many_arguments)]` style trade-offs; one more positional arg is fine.
- `gix::diff::blob` re-exports `imara_diff::*`, so `gix::diff::blob::{Algorithm, diff, intern::InternedInput}` are all reachable from `seshat-graph` without adding a direct `gix-diff`/`imara-diff` dependency. The closure form of `Sink` (`impl<T: FnMut(Range<u32>, Range<u32>)> Sink for T`) is the easiest collector — capture a `&mut Vec<Hunk>` in the closure and ignore the `()` return of `diff(...)`. imara_diff line ranges are 0-based half-open; convert to 1-based half-open with `+1` on both `start` and `end` (an empty `[s, s)` 0-based range stays empty after `+1`, which is what we want for pure insertion/deletion sites).
- Clippy enforces `clippy::type-complexity` under `-D warnings`. Public function signatures returning nested generics like `Result<Option<(Vec<u8>, Vec<u8>)>, GraphError>` will fail the lint. The fix is a `pub type BlobPair = (Vec<u8>, Vec<u8>);` alias that preserves the AC's tuple shape; do not reach for `#[allow(clippy::type_complexity)]` unless aliasing is actually impossible.
- For the `seshat-graph` blob/diff layer, prefer reading bytes via `repo.find_object(oid)?.data.clone()` rather than the streaming readers — the data is already in memory and the size cap (`MAX_DIFF_FILE_SIZE = 5 MiB`) is enforced cheaply on `obj.data.len()` before any further work. `gix::ObjectId` derives `Copy`, so blob IDs flow freely through `Option<gix::ObjectId>` fields without `clone()` calls; use `.copied()` when extracting from `HashMap<String, gix::ObjectId>::get` rather than `.cloned()`.
- The "binary blob" heuristic git itself uses (NUL byte in first 8 KiB) is the right one for `seshat-graph`'s `read_blob_pair`. Returning `Ok(None)` (rather than `Err`) for binary OR oversized OR missing-on-disk lets the caller treat all three as "fall back to `Hunk::ALL`" — keep the public surface consistent so callers don't need a 3-way match.
- `crates/seshat-mcp/src/call_logger_keys.rs` is the single source of truth for response-envelope JSON keys read by `call_logger::*_result()` extractors. Every tool that has a result summary should have a `pub mod <tool>` with `pub const DATA_*` constants — `dependencies`, `validate_approach`, `diff_impact`, etc. all follow this pattern. Adding the module BEFORE touching the corresponding `*_result()` function makes the diff inside the extractor uniform and avoids "magic string" review objections.
- For `call_logger::*_result()` extractors, the canonical defensive pattern is `.get(KEY).and_then(|v| v.as_<type>()).unwrap_or_else(|| { tracing::debug!("...: key '{}' missing or not a <type>", KEY); <safe_default> })`. Every field — array length, scalar count, status string, bool — uses this shape. Don't introduce ad-hoc `.unwrap_or(0)` shortcuts; they break the uniform debug instrumentation that the file relies on.
- For per-symbol response data on diff_impact (changed_lines, direct_dependent_count, etc.), the canonical surface in `tools/diff_impact.rs::generate_next_steps` uses helper functions like `format_changed_lines` that emit a leading separator (e.g. `" at lines 42-58"`) and return an empty string for the no-content case. The caller's format string becomes `"{name} touched{lines_phrase} with ..."` — no branching, no awkward whitespace, no dangling prepositions. Same shape applies to any future optional clause.
- When extending an `*_result()` summary with a new field sourced from response data, prefer adding the field to the response `Data` struct (e.g. `DiffImpactData.total_hunks`) over deriving it from sibling fields in the call_logger. Derived approximations (sum of per-entry sub-counts) overcount when one source object touches multiple entries. The exact count belongs in the producer, not the observer.
- Criterion benches go in `crates/<crate>/benches/<name>.rs` and require a `[[bench]] name = "..." harness = false` block in that crate's Cargo.toml. `criterion` lives in `[workspace.dependencies]` (Testing section) with `default-features = false, features = ["cargo_bench_support"]` to skip the unused html/plotters chain, and individual crates pull it via `criterion.workspace = true` in their `[dev-dependencies]`. There is no `[profile.bench]` override — criterion uses the default release profile with the workspace's `strip = "debuginfo"` baseline. Runnable via `cargo bench -p <crate> --bench <name>` (add `-- --quick` for a fast smoke run that still produces median stats).
- For perf benches that drive `map_diff_impact` / `query_dependencies` / similar IR-aware code paths, the pattern is: build the synthetic state ONCE in a `Setup` struct (tempdir + git repo + in-memory DB + working-tree mods + DiffImpactRequest), warm up by calling the function once (pays the IR-cache + gix repo discovery cost), then run a single timed reference call with `std::time::Instant::now()/elapsed()` to enforce a hard `assert!(elapsed_ms < BUDGET)` CI gate, AND drive criterion's `bench_function`+`b.iter` for the median statistic in the standard report. Both surfaces document the budget — one fails loud, the other surfaces gradual drift.
- Bench fixtures must lock their AC shape (file count, hunk count, expected symbol count) via an `assert_setup_shape()` call before timing kicks in. Without it, an accidental fixture regression (e.g. a forgotten mod, a misplaced line number) silently shrinks the workload and the perf assert passes on a no-op. The shape assertion belongs OUTSIDE the timed region — it pays for itself in how many "the bench is fast but tests nothing" landmines it catches.

    `gix-diff-0.52.1/src/blob/mod.rs:6`. Glob re-exports DO carry pub
    modules through (verified at compile time, not just spec text).

    diffs: `impl<T: FnMut(Range<u32>, Range<u32>)> Sink for T` has

    5 MiB this is fine — just `clone()` the bytes out. Don't bother
    with the streaming readers (`gix-object::Object::data` etc.)
    until you have a reason to: the size cap on `obj.data.len()`
    fires before any expensive work. If MAX_DIFF_FILE_SIZE ever has
    to grow past ~50 MiB, revisit and switch to a chunked reader.
  - clippy's `type-complexity` lint catches public-API tuples buried
    inside `Result<Option<...>, _>`. The fix is a one-line `pub type`
    alias (e.g. `pub type BlobPair = (Vec<u8>, Vec<u8>);`) — do NOT
    reach for `#[allow(clippy::type_complexity)]` unless the alias
    materially obscures the intent. The alias also makes the public
    documentation read better (`Result<Option<BlobPair>, GraphError>`
    vs the nested generic).
  - For the binary heuristic, `bytes[..bytes.len().min(PROBE_LEN)]`
    is safer than `bytes[..PROBE_LEN]`. The former works on blobs of
    any size including empty ones; the latter panics on anything
    smaller than 8 KiB. The empty case returns `false` (nothing to
    probe), which is the correct semantics — an empty blob should not
    be classified as binary.
  - Returning `Ok(None)` (rather than `Err`) for "can't diff this
    pair" lets US-009 funnel binary, oversized, AND missing-on-disk
    files through a single fall-back path
    (`read_blob_pair(...).unwrap_or(None).map_or_else(|| Hunk::ALL,
    diff_blobs_to_hunks)` or similar). Distinguishing those three
    cases at the API layer would push the fall-back logic into every
    caller without adding any value.
  - The `#[tracing::instrument(skip_all, fields(repo_path = ...))]`
    attribute on `enumerate_changes_with_blobs` mirrors the prior
    `get_changed_files` attribute. Keeping both spans is fine — the
    wrapper just emits a leaf span that immediately delegates to the
    enriched function. If span volume ever becomes a noise problem,
    drop the wrapper's `#[instrument]` and rely on the inner span.

    setup MUST live OUTSIDE the timed region — `criterion`'s
    `b.iter(...)` only times the closure, but if the closure
    re-creates the tempdir or re-stages the diff each iteration
    you measure setup cost, not the function's hot-path. The
    `Setup` struct held by reference inside the bench closure is
    the right pattern.
  - The on-disk file content does NOT have to be parseable by
    tree-sitter for `compute_affected_symbols` to work — the
    hunk-intersection logic only cares about (a) the diff
    between base and modified bytes (computed by imara_diff) and
    (b) the `[line, end_line]` ranges in the IR. So a synthetic
    fixture can use `\n`-delimited TypeScript-shaped text with
    hand-built IR ranges and skip the parser entirely. This
    keeps the bench setup independent of parser changes.
  - `affected_symbols.len()` came back as **49**, not 50, because
    `file_049` is at the tail of the import chain — no upstream
    importers, so its modified `func_049_0` has zero direct
    dependents and US-009's `direct.is_empty() skip` rule
    correctly filters it out. The `assert_setup_shape()` check
    asserts `is_empty()` (not `== 50`) so this is locked
    deliberately, not by accident. If the import chain ever
    becomes a closed loop, both endpoints become flaggable and
    the count rises to 50 — the assertion still holds.
  - `total_hunks` is the cleanest single-number proxy for
    "diff-impact workload size" — it's deterministic from the
    fixture (single-hunk files contribute 1, multi-hunk contribute
    2), independent of how many symbols intersect each hunk, and
    surfaces fixture drift early. Always assert on it in any
    future diff_impact bench fixture.
  - `clippy::single_char_add_str` fires on `s.push_str("}")` and
    similar literal-single-char calls — use `s.push('}')`. This
    is a strict-mode-only lint (allowed by default, denied here
    via `-D warnings`). Worth a sweep before committing any new
    fixture string-builder code.
