## Codebase Patterns
- Workspace: Rust 2024 edition, MSRV 1.85, members under `crates/seshat-*/`. Each crate uses `version.workspace = true` + `[workspace.dependencies]` for shared deps.
- `dirs` crate (v6) is already a workspace dep — use it for `home_dir()` resolution rather than reading `$HOME`/`%USERPROFILE%` manually.
- Error convention: each crate has `src/error.rs` with a `thiserror::Error` enum (e.g. `CliError` in `crates/seshat-cli/src/error.rs`).
- Logging convention: `tracing::warn!` with structured key=value fields (e.g. `tracing::warn!(entry = %raw, "...")`), not `format!`-style strings.
- Module wiring: `crates/seshat-cli/src/lib.rs` declares each module with a `///` doc comment then `pub mod <name>;`. Re-exports follow the module declarations.
- Tests live INLINE in `#[cfg(test)] mod tests` at the bottom of each `.rs` file (the dedicated `tests/` dirs are integration tests).
- Canonical test-temp lib: `tempfile::TempDir`. `tempfile` is in workspace deps (regular and dev).
- macOS gotcha: `std::env::temp_dir()` returns `/var/folders/...`; `/var` is in our macOS denylist. So almost every absolute path on macOS is "dangerous" by default — design tests to either work with that fact or use the `is_dangerous_inner` test entry that takes a controlled `builtin: &[PathBuf]` slice.
- Clippy gate: `cargo clippy --all-targets -- -D warnings` must pass workspace-wide. Run `cargo clippy -p <crate> --all-targets -- -D warnings` for crate-level fast feedback.
- The `.ralph/prd.json` `passes` flag is the source of truth for story completion; update it AFTER tests + clippy + commit succeed.

## 2026-05-06 (EEST) - US-001
Added per-OS dangerous-cwd denylist module so US-003 (and future watcher gating) can refuse to auto-scan from `$HOME`, `~/Library`, drive roots, etc.

**Implemented:**
- New module `crates/seshat-cli/src/dangerous_path.rs` exporting `pub fn is_dangerous_cwd(path: &Path, additional: &[String]) -> bool`.
- Test-only entry point `pub(crate) fn is_dangerous_cwd_with_home(path, additional, home: Option<&Path>) -> bool` for injecting a fake home directory in unit tests.
- Per-OS `builtin_denylist(home)` gated by `cfg(target_os)`:
  - macOS: $HOME plus Library/Documents/Downloads/Desktop/Pictures/Movies/Music/Public/.config/.cache; absolute /, /Users, /Applications, /System, /Library, /private, /tmp, /var, /usr, /etc, /opt.
  - Linux: $HOME, /, /home, /etc, /var, /tmp, /usr, /opt, /root, /proc, /sys, /dev, plus XDG_CONFIG_HOME / XDG_CACHE_HOME / XDG_DATA_HOME with `~/.config` / `~/.cache` / `~/.local/share` fallbacks when env vars are unset.
  - Windows: %USERPROFILE% (+ Documents/Downloads/Desktop), %APPDATA%, %LOCALAPPDATA%, %TEMP%, C:\Windows, C:\Program Files, C:\Program Files (x86), C:\ProgramData, plus all 26 drive roots (`A:\` … `Z:\`).
- Path matching:
  - Both candidate and entries canonicalized via `std::fs::canonicalize` (symlinks resolved); fall back to `path.to_path_buf()` when canonicalize fails (e.g. when path doesn't exist).
  - `path_matches(candidate, entry)` uses `Path::starts_with` so `/var2` does not match `/var` (component-wise prefix check).
  - On macOS/Windows: case-insensitive via `to_string_lossy().to_lowercase()`. On Linux: byte-exact.
- Behavior:
  - Built-in entries that don't resolve on this machine are silently skipped (`push_canonical` ignores `Err`).
  - Relative entries in `additional` are skipped with a structured `tracing::warn!` log.
- Module wired into `crates/seshat-cli/src/lib.rs` with a `///` doc comment.

**Files changed:**
- `crates/seshat-cli/src/dangerous_path.rs` (new, ~340 lines including tests)
- `crates/seshat-cli/src/lib.rs` (added `pub mod dangerous_path;`)

**Tests (17 inline unit tests, all passing):**

**Learnings for future iterations:**
- Don't try to test "this path is NOT dangerous" with a real tempdir on macOS — `/var/folders/...` is under `/var`, which is in the macOS denylist. Use the `is_dangerous_inner(canonical, additional, builtin)` shared helper with an empty/controlled builtin slice, OR cfg-gate such tests for Linux only.
- Rust 2024 edition makes `std::env::set_var` / `remove_var` unsafe AND parallel-test-unsafe. Avoid env mutation in unit tests; instead expose the env-aware logic at a higher boundary (here, `builtin_denylist(home)` accepts the resolved home as a parameter).
- `Path::starts_with` is component-wise: `Path::new("/var2").starts_with("/var")` is `false` because the first non-root component "var2" is not equal to "var". This gives us free protection against the `/var` vs `/var2` prefix-string false-positive.
- For US-003, `is_dangerous_cwd` (path, additional) is the public entry point. `additional` should be wired from `ScanConfig::additional_denylist_paths` once US-002 lands. The function returns `bool` — pair it with `find_git_root(&cwd).is_none()` to decide refusal.
- Workspace clippy is fast on warm cache (~1.5s for incremental). Always run the full `cargo clippy --all-targets -- -D warnings` before declaring a story passing — running just the touched crate misses cross-crate warnings.

## 2026-05-06 (EEST) - US-003
Wired the dangerous-cwd guardrail (US-001) and configurable denylist (US-002) into `seshat serve`. P1 closed: the auto-scan path now refuses to run from a denylisted cwd that isn't inside a git repo, and the `--repo` opt-out emits a non-fatal multi-line stderr warn. This is the user-visible change that prevents the original 91.8 GB recursive-walk leak class for accidental `seshat serve` invocations from `$HOME`, `~/Library`, drive roots, etc.

**Implemented:**
- New `CliError::DangerousCwd { path: PathBuf, hint: String }` variant in `crates/seshat-cli/src/error.rs` with a `#[error(...)]` Display that produces a multi-line message (offending path + denylist explanation + three concrete suggestions). Confirmed `thiserror` v2 formats `{path}` for `PathBuf` via `Path::display()` — same trick used by the existing `IoWithPath` variant.
- Decomposed the resolve-serve logic in `crates/seshat-cli/src/db.rs` into pure helpers so the decision logic is unit-testable without monkey-patching `cwd`/`home`:
  - `build_dangerous_cwd_hint() -> String` — three-suggestion hint body (cd / `seshat scan` / `seshat serve --repo`).
  - `build_repo_override_warning(project_root) -> String` — multi-line ⚠️-prefixed stderr warn for the `--repo` opt-out.
  - `check_serve_dangerous_cwd(explicit_repo, additional, cwd, home) -> Result<(), CliError>` — refusal gate. Returns `Ok(())` when `explicit_repo.is_some()`, when cwd isn't dangerous, or when `find_git_root(cwd).is_some()`. Otherwise returns `Err(DangerousCwd)`.
  - `check_repo_override_dangerous(explicit_repo, additional, project_root, home) -> Option<String>` — non-fatal warning gate. Mirrors the same "dangerous && no git root" condition, but applied to the resolved `project_root` and only when `explicit_repo.is_some()`. Returns `None` (silent) otherwise.
- `resolve_serve_db_or_project_root(explicit_repo: Option<&Path>, additional_denylist_paths: &[String])` — signature extended with the denylist arg. Flow: refusal check on cwd → existing `resolve_project` → override warn on `project_root` → return `ServeTarget::ExistingDb | AutoScan`. The override warn is emitted via `eprintln!` (process side effect kept separate from the pure helper).
- Call site in `crates/seshat-cli/src/serve.rs:449` updated to pass `&config.scan.additional_denylist_paths`. AppConfig already loads scan.additional_denylist_paths from `seshat.toml` (US-002 plumbing), so no further wiring needed.

**Learnings for future iterations:**
- The original PRD criterion "explicit_repo=Some(safe) → proceeds with no warn" is impossible to test reliably with raw paths on macOS, because `/` is in the macOS builtin denylist and `path_matches` uses `Path::starts_with` — so EVERY absolute macOS path matches `/`. The only way for `is_dangerous_cwd_with_home` to return `false` for an absolute path is via a path that fails canonicalize AND doesn't share the first component with any builtin entry (rare). The "safe" semantic the PRD really wanted is "has a git root upstream", so I extended `check_repo_override_dangerous` with the `find_git_root.is_some()` short-circuit (mirroring the refusal gate). This makes "safe" = "the user pointed `--repo` at a real project" — clear and OS-portable.
- `thiserror` v2's `#[error("{path}")]` on a `PathBuf` field works (uses `Path::display()` under the hood) — same trick `IoWithPath` already uses. No need to pre-format the path as a String in the constructor.
- Decomposing into pure decision helpers (`check_serve_dangerous_cwd`, `check_repo_override_dangerous`) avoided needing to capture stderr or `std::env::set_current_dir` in unit tests. The thin wrapper `resolve_serve_db_or_project_root` does the I/O (`current_dir()`, `dirs::home_dir()`, `eprintln!`); pure helpers do the decisions. This is the testable version of "side effects at the edge".
- For US-005 (integration tests), the existing seshat-bin integration tests already use `assert_cmd::Command` and `tempfile::TempDir`. They are currently broken at the parent commit due to the V11 migration issue — the integration test author may need to investigate that before adding new serve_guardrails.rs tests, or scope around it.
- Clippy `question_mark` lint fires on `if x.is_none() { return None; }` and wants `x?;` — which short-circuits propagation. Watch for this when porting C-style early-returns.
