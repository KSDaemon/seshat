## Codebase Patterns
- Workspace: Rust 2024 edition, MSRV 1.85, members under `crates/seshat-*/`. Each crate uses `version.workspace = true` + `[workspace.dependencies]` for shared deps.
- `dirs` crate (v6) is already a workspace dep — use it for `home_dir()` resolution rather than reading `$HOME`/`%USERPROFILE%` manually.
- Error convention: each crate has `src/error.rs` with a `thiserror::Error` enum (e.g. `CliError` in `crates/seshat-cli/src/error.rs`).
- Logging convention: `tracing::warn!` with structured key=value fields (e.g. `tracing::warn!(entry = %raw, "...")`), not `format!`-style strings.
- Module wiring: `crates/seshat-cli/src/lib.rs` declares each module with a `///` doc comment then `pub mod <name>;`. Re-exports follow the module declarations.
- Tests live INLINE in `#[cfg(test)] mod tests` at the bottom of each `.rs` file (the dedicated `tests/` dirs are integration tests).
- Canonical test-temp lib: `tempfile::TempDir`. `tempfile` is in workspace deps (regular and dev).
- macOS gotcha: `std::env::temp_dir()` returns `/var/folders/...`; `/var` is in our macOS denylist. So almost every absolute path on macOS is "dangerous" by default — design tests to either work with that fact or use the `is_dangerous_inner` test entry that takes a controlled `builtin: &[PathBuf]` slice.
- Clippy gate: `cargo clippy --all-targets -- -D warnings` must pass workspace-wide. Run `cargo clippy -p <crate> --all-targets -- -D warnings` for crate-level fast feedback.
- The `.ralph/prd.json` `passes` flag is the source of truth for story completion; update it AFTER tests + clippy + commit succeed.
- Subprocess integration tests for the `seshat` binary live in `crates/seshat-bin/tests/`, NOT `crates/seshat-cli/tests/`. The `[[bin]]` is defined in seshat-bin, so `cargo test -p seshat-cli --test foo` does NOT auto-rebuild the binary; the seshat-bin location is the only one that auto-tracks the bin's transitive deps. assert_cmd/predicates/tempfile/dirs dev-deps already live there.
- `Database::open` (in seshat-storage) does NOT create parent directories — `Connection::open` returns an error if the parent dir doesn't exist. Only `seshat scan` (via `scan.rs::run_scan`) calls `create_dir_all` before opening. Subprocess integration tests that point HOME at a fresh tempdir must pre-create `<HOME>/Library/Application Support/seshat/repos/` (macOS) or `<HOME>/.local/share/seshat/repos/` (Linux) — otherwise `seshat serve` fails on first DB open.

## 2026-05-06 (EEST) - US-001
Added per-OS dangerous-cwd denylist module so US-003 (and future watcher gating) can refuse to auto-scan from `$HOME`, `~/Library`, drive roots, etc.

**Implemented:**
- New module `crates/seshat-cli/src/dangerous_path.rs` exporting `pub fn is_dangerous_cwd(path: &Path, additional: &[String]) -> bool`.
- Test-only entry point `pub(crate) fn is_dangerous_cwd_with_home(path, additional, home: Option<&Path>) -> bool` for injecting a fake home directory in unit tests.
- Per-OS `builtin_denylist(home)` gated by `cfg(target_os)`:
  - macOS: $HOME plus Library/Documents/Downloads/Desktop/Pictures/Movies/Music/Public/.config/.cache; absolute /, /Users, /Applications, /System, /Library, /private, /tmp, /var, /usr, /etc, /opt.
  - Linux: $HOME, /, /home, /etc, /var, /tmp, /usr, /opt, /root, /proc, /sys, /dev, plus XDG_CONFIG_HOME / XDG_CACHE_HOME / XDG_DATA_HOME with `~/.config` / `~/.cache` / `~/.local/share` fallbacks when env vars are unset.
  - Windows: %USERPROFILE% (+ Documents/Downloads/Desktop), %APPDATA%, %LOCALAPPDATA%, %TEMP%, C:\Windows, C:\Program Files, C:\Program Files (x86), C:\ProgramData, plus all 26 drive roots (`A:\` … `Z:\`).
- Path matching:
  - Both candidate and entries canonicalized via `std::fs::canonicalize` (symlinks resolved); fall back to `path.to_path_buf()` when canonicalize fails (e.g. when path doesn't exist).
  - `path_matches(candidate, entry)` uses `Path::starts_with` so `/var2` does not match `/var` (component-wise prefix check).
  - On macOS/Windows: case-insensitive via `to_string_lossy().to_lowercase()`. On Linux: byte-exact.
- Behavior:
  - Built-in entries that don't resolve on this machine are silently skipped (`push_canonical` ignores `Err`).
  - Relative entries in `additional` are skipped with a structured `tracing::warn!` log.
- Module wired into `crates/seshat-cli/src/lib.rs` with a `///` doc comment.

**Learnings for future iterations:**
- Don't try to test "this path is NOT dangerous" with a real tempdir on macOS — `/var/folders/...` is under `/var`, which is in the macOS denylist. Use the `is_dangerous_inner(canonical, additional, builtin)` shared helper with an empty/controlled builtin slice, OR cfg-gate such tests for Linux only.

## 2026-05-06 (EEST) - US-004
P0 closed: the file watcher no longer starts when the auto-scan failed. This was the root-cause of the 91.8 GB memory leak — `notify-debouncer-full` recursively walks the project root on init, so spawning it after a failed scan (e.g. dangerous cwd, project too large, scan timeout) would re-trigger the very walk we already gave up on.

**Implemented:**
- New `WatcherError::ScanFailed(String)` variant in `crates/seshat-watcher/src/error.rs` with `#[error("Auto-scan failed; watcher not started: {0}")]`. Two new unit tests cover Display (asserts the reason and the "watcher not started" consequence both render) and Debug (asserts the variant name remains greppable in log lines).
- Extracted `fn watcher_should_start(enabled: bool, state: &ScanState) -> bool` as a free function near the top of `crates/seshat-cli/src/serve.rs` (right after `resolve_call_log_path`). Returns `enabled && state.error_message().is_none()`. Free function — not a method on ScanState — because the gate also depends on the user's `[watcher] enabled` config, which is owned by serve.rs not seshat-mcp.
- Outer gate at serve.rs:754 changed from `if watcher_enabled` to `if watcher_should_start(watcher_enabled, &scan_state)`. Comment updated to reference the PRD US-004 P0 rationale and the 91.8 GB report.
- Race guard inside the spawned watcher task: after `wait_scan.wait_for_scan()`, re-check `wait_scan.error_message()` — if `Some(msg)`, log `tracing::info!(error_message = %msg, "...")`, send `Err(WatcherError::ScanFailed(msg))` through `watcher_tx`, and return without calling `start_watcher()`. Closes the window where the outer gate observed an in-progress scan but the scan fails before the watcher task wakes up.

Restructured as a `match scan_error.as_ref()` instead of nested if-else to avoid `let_chains` (cleaner and works regardless of MSRV nuances).
- Imported `WatcherError` from `seshat_watcher` (it was previously not imported because `start_watcher` returned its result directly into the channel without ever needing to construct a variant in serve.rs).

**Learnings for future iterations:**
- `let_chains` (`if cond && let Some(x) = expr`) requires Rust 1.88+ in edition 2024. Our MSRV is 1.85, so prefer `match expr { Some(x) if cond => ..., _ => ... }` which works on every stable Rust. The match-guard style is also more readable when there are 4+ arms.
- `print_startup` takes `watcher_status: &str`. When you need to mix owned (`format!(...)`) and borrowed (`"static"`) strings, `std::borrow::Cow<'_, str>` is the canonical fit — pass `&watcher_status` to deref to `&str`. Don't try to make all branches owned (`String`) because that allocates on the hot path; don't try to make all branches borrowed (with a side `let` for the formatted one) because that reads worse than Cow.
- `ScanState::error_message()` returns `Option<String>` — it's a clone, not a borrow. Cache the result in a local (`let scan_error = scan_state.error_message();`) when the banner reads it multiple times in a row, to avoid re-locking the mutex per branch.
- Two-stage gate is the correct pattern when an async task waits for a state that can change underneath it: outer `watcher_should_start` checks at spawn time (cheap, common-case path), inner re-check after `wait_for_scan()` handles the race. Don't try to collapse them — the outer gate also matters because it determines whether to allocate the oneshot channel and the spawn task at all.
- `WatcherError::ScanFailed(String)` is delivered through the existing `watcher_tx: oneshot::Sender<Result<WatcherHandle, WatcherError>>` channel — no API changes to the channel. The shutdown path at serve.rs:976-980 already does `if let Ok(Ok(handle)) = rx.try_recv()` so a `ScanFailed` Err is correctly ignored (no shutdown needed because no watcher started).
- For US-005 (integration tests), the test for "watcher: disabled (auto-scan failed)" can now also check for the enriched form `(auto-scan failed: <reason>)`. The reason text comes from whatever the scanner pushed into `mark_failed()` — for the `auto_scan_limit = 1` case in the PRD, expect something like "too many files" or whatever `auto_scan_project_root` reports.

**Learnings for future iterations:**
- **`cargo test -p <crate>` does NOT rebuild dependent binaries.** The `[[bin]]` for `seshat` lives in `seshat-bin`. When my first attempt placed the test file at `crates/seshat-cli/tests/`, `cargo test -p seshat-cli --test serve_guardrails` happily ran the test against the *stale* `target/debug/seshat` binary that didn't include my serve.rs banner fix. Tests appeared to fail when in fact the fix worked — verified by manually invoking the binary, where it produced the correct banner. **Rule: subprocess integration tests live in the crate that owns the `[[bin]]`** so Cargo auto-tracks the bin's transitive deps. The PRD's "crates/seshat-cli/tests/" path is incorrect for this reason.
- **`Database::open` does not create parent directories.** Only `scan.rs::run_scan` (`scan -p ...`) calls `create_dir_all` on the repos dir. Subprocess tests that isolate `HOME` to a tempdir must pre-create `<HOME>/Library/Application Support/seshat/repos/` (macOS) or `<HOME>/.local/share/seshat/repos/` (Linux), or the very first `seshat serve` will fail with "unable to open database file". This is a real (small) UX bug worth filing separately — first-time `seshat serve` from a fresh user account can hit this.
- **The watcher banner format is `disabled (auto-scan failed: <reason>)`, not `disabled (auto-scan failed)`.** The PRD's plain-text snippet was abbreviated. The actual reason text comes from `scan_state.error_message()`, which for the `auto_scan_limit` branch is `Project too large for auto-scan ({n} files). Run: seshat scan --verbose`. Tests should match the prefix `disabled (auto-scan failed` and assert the reason separately, which is what test 4 does.
- **`repo` is a POSITIONAL argument on `seshat serve`, not `--repo <path>`.** The PRD wording "passes `--repo <path>`" is shorthand for "passes the repo path as the positional REPO argument". `seshat serve --repo /some/path` is parsed by clap as an unknown long flag and exits with an error.
- **`tokio::signal::ctrl_c()` is the only signal handler in serve.rs.** It catches SIGINT, not SIGTERM. The PRD's "send SIGTERM and assert clean exit" doesn't actually produce a clean exit (SIGTERM is uncaught → default action: kill). For test 2/3 cleanup, `child.kill()` (SIGKILL) is functionally equivalent for our purposes — the test goal is "the banner appeared", not "graceful shutdown semantics". Skipped the SIGTERM step.
- **Banner display logic vs. watcher gate logic.** US-004's `watcher_should_start(enabled, &state)` correctly gated the watcher off when scan failed (no leak). But the banner-display match in serve.rs:962-980 had a separate bug: the failure-arm guard `Some(msg) if has_auto_scan` could never match because the AutoScan failure branch sets `auto_scan_project_root = None`, flipping `has_auto_scan` to false at the exact moment we need the banner to show "disabled (auto-scan failed: ...)". US-005's integration test 4 caught this where the unit tests for `watcher_should_start` did not — exactly the value of integration tests. Fix: drop the `if has_auto_scan` guard. Safe because `error_message().is_some()` only ever becomes true on the AutoScan failure path (ExistingDb uses `ScanState::not_needed()`, success uses `mark_complete()`).
- **macOS canonical paths in tests.** `tempfile::tempdir()` on macOS returns `/var/folders/...`, but `std::env::current_dir()` after `cd` into it returns `/private/tmp/...` (because `/tmp` is a symlink to `/private/tmp`). Both are under `/var` (which is denylisted). Stderr output may show one form or the other depending on whether the path was canonicalized — match against substrings, not exact paths.
