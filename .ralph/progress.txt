## Codebase Patterns
- `query_code_pattern` enrichment passes (call-sites, dependent_files) live in
  `crates/seshat-graph/src/code_pattern.rs` as separate functions invoked
  AFTER the merged keyword + vector result set is finalised.  Each
  enrichment is best-effort: a SQL failure downgrades to empty rather
  than aborting the whole query.  When extending the per-pattern response
  shape, add another enrichment step in the same place rather than
  weaving the new lookup into the keyword/vector search paths.
- Read-side SQL probes against `symbol_definitions` / `symbol_imports`
  live directly in `seshat-graph::code_pattern` (e.g.
  `search_symbol_definitions`, `enrich_with_dependent_files`) — they
  hold the connection lock via `crate::lock_conn` and use the same
  V13 `(branch_id, name)` / `(branch_id, imported_name)` indexes the
  write side already maintains.  The `SymbolIndexRepository` trait
  stays write-/lifecycle-focused (replace_file, delete_file,
  delete_branch, counts).
- Migrations live in `crates/seshat-storage/migrations/V<N>__<name>.sql`,
  embedded via `refinery::embed_migrations!`. Refinery's SQL files cannot
  deserialize postcard blobs — for backfill that needs Rust, perform it in
  `Database::open` immediately after `migrations::runner().run(...)`, gated
  on a "table empty?" check so it stays idempotent across reopens.
- IR snippets must be built the same way wherever they are produced. Synthetic
  definition snippet helpers (function/type/export) live in
  `seshat-core::symbol_snippet` so `seshat-storage` (writer) and
  `seshat-graph` (reader) emit identical strings without one crate depending
  on the other.
- Repository traits live in `crates/seshat-storage/src/repository/mod.rs`;
  Sqlite impls live in sibling files. Re-export everything via
  `crates/seshat-storage/src/lib.rs` so downstream crates only import from the
  crate root.
- Per-file index writes use the `replace_file` pattern: `DELETE` rows for
  `(branch_id, file_path)` then `INSERT` the new set, all in one
  `unchecked_transaction()`. Keeps re-scans / hot-tier updates idempotent.
- Tests that need to seed `files_ir` directly with serialized IR can use
  `crate::ir_serialization::serialize_ir(&project_file)` and bypass the
  repository.
- Cross-table writes that must commit atomically (e.g., `files_ir` + symbol
  index) live as a single method on the owning repository
  (`FileIRRepository::upsert_with_symbol_index`) that opens its own
  `unchecked_transaction()` and runs every statement inside it.  Sibling
  modules expose `pub(super)` SQL helpers operating on a `&Transaction` so
  the writer doesn't duplicate the per-table DELETE/INSERT SQL.
- Parser convention: `Import.names` stores defining (rightmost) names, never
  local aliases.  TS/JS already do this via the first identifier of
  `import_specifier`; Rust and Python parsers normalise top-level
  `use foo::Bar as Baz` and `import x as y` / `from m import x as y` to keep
  the IR consistent and downstream code (symbol-index, detectors) free of
  alias-stripping logic.
- Branch lifecycle ownership: every per-branch table must be touched by
  `BranchRepository::create_snapshot` (copy) and `delete_branch` (drop).
  After adding a new per-branch table, audit both methods — missing one
  produces silent state drift (e.g. symbol-index that doesn't survive
  branch switches, or orphan rows that accumulate across recreate cycles).
- Per-crate test helpers seed BOTH halves of a derived index in one call.
  After US-002 made `files_ir` and `symbol_definitions` share a single
  write transaction (`upsert_with_symbol_index`), tests that used to call
  the lower-level `INSERT INTO files_ir ...` directly leave the symbol
  index empty.  `seshat-graph`'s and `seshat-mcp`'s `insert_ir` helpers
  (and `seshat-mcp::server::insert_ir_for_server`) now also call
  `SqliteSymbolIndexRepository::replace_file(...)` so a single fixture
  call lands a complete row set.  When you add another per-file table to
  this DB family, update the helpers in lock-step.
- SQL `LIKE` patterns need escaping for `_` (single-char wildcard) and
  `%` (any-length wildcard).  Use `LIKE ? ESCAPE '\\'` plus a helper that
  prefixes `\\` before `_`, `%`, and `\\` itself.  Without escaping,
  `do_thing` would also match `doXthing` — fixed in
  `crates/seshat-graph/src/code_pattern.rs::escape_like_pattern`.
- Lazy IR load: code paths that load all of a branch's serialized IR
  should branch on "do we actually need it?".  `query_code_pattern_*`
  now skips `load_branch_ir` entirely when the keyword probe is empty AND
  no embedding provider is configured — saves a full deserialization
  pass on misses.
- Blast-radius classification is a single shared helper:
  `seshat_graph::dependencies::classify_blast_radius(count: usize) ->
  BlastRadius` with `< 5 / 5..=20 / > 20` thresholds.  Both file-level
  (`query_dependencies`) and symbol-level (`query_code_pattern`) outputs
  funnel through it, so the labels can't drift between tools.  When
  adding a new per-X risk signal, call this helper rather than inlining
  a comparison — keep the thresholds in one place.
- MCP tool descriptions live in TWO places that must stay in sync:
  (a) the `#[tool(description="...")]` attribute on each handler method in
  `crates/seshat-mcp/src/server.rs` (one-paragraph summary shown in the
  tools listing) and (b) the request struct doc comment in
  `crates/seshat-mcp/src/tools/<tool>.rs` (the JsonSchema description
  embedded in every tool invocation).  When you add/change response fields,
  update BOTH — clients reading the schema and clients reading the listing
  will otherwise see different stories.  `next_steps` content can also be
  pinned via the metadata in the handler, but `handle_response_envelope_
  structure`-style tests only assert the array exists, so adding new
  next_steps requires a dedicated assertion test on the suggestion string.
- Call-site enrichment in `seshat-graph::code_pattern` aggregates by file
  rather than per-call.  `enrich_with_call_sites` walks each pattern's
  matching `function_calls` once, groups into a
  `HashMap<String, FileCallAggregate>` (count + ascending line numbers +
  min-line snippet), converts to public `CallSiteFileAggregate { file,
  site_count, lines: Vec<u32>, first_snippet }`, sorts by `site_count
  desc, file asc`, and truncates to `MAX_CALL_SITE_FILES_PER_PATTERN`.
  `total_call_sites` is summed during the walk so it stays uncapped even
  when the displayed Vec is truncated.  `first_snippet` always runs
  through `seshat_core::truncate_snippet` so the JSON payload stays
  bounded.  Note for future expansions of the per-pattern response: keep
  the cap on the *file* dimension, not on lines within a file.
- Atomic `files_ir` + symbol-index DELETEs live on `FileIRRepository::
  delete_with_symbol_index` (sibling of `upsert_with_symbol_index`).
  Opens its own `unchecked_transaction`, runs `DELETE FROM files_ir`,
  then `pub(super)` `delete_definitions` / `delete_imports` against the
  same `&Transaction`, commits or rolls back the entire trio.  Used by
  the watcher hot tier (`process_file_delete`), the scanner orchestrator
  (incremental file-removal sweep), and `incremental_sync_blocking`
  (branch-switch sync for files that exist in old HEAD but not new HEAD).
  Three call sites, one atomic helper — no caller composes `delete_by_path`
  + `symbol_index_repo.delete_file` directly any more.  Sweeps orphan
  symbol-index rows even when files_ir already missed (defensive).
