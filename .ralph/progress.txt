## Codebase Patterns
- Edition 2024, Rust 1.85+. `let-else`, `..=` ranges, and similar modern syntax are fine.
- `seshat-graph/src/dependencies.rs` already has a `SuffixIndex` (O(1) reverse-suffix lookup) and an `infer_package_root` heuristic for cross-crate guards. Reuse instead of rolling new ones.
- IR `path` is stored as an absolute `PathBuf`; `target_path` arguments to public APIs are typically relative вАФ always normalise via `normalize_path` + `suffix_matches_at_boundary` before comparing.
- For new pure-graph utilities (BFS, adjacency builders), construct synthetic `HashMap<String, Vec<вА¶>>` directly in unit tests; no DB / IR roundtrip needed. The repo-level `test_helpers::{test_conn, insert_ir}` is for integration-style tests.
- Rust style: snake_case fns/params, PascalCase types, `expect("message")` over `unwrap()` in tests, doc comments on every `pub` item with `///`.
- `DependentEntry` and similar serde types use `#[serde(default)]` on newly-added fields for forward compatibility within the same schema version.
- Clippy is run as `cargo clippy --all-targets -- -D warnings` and CI gates on it вАФ no warnings tolerated.
- `DependentEntry` / `DependencyEntry` JSON field is `file_path` (NOT `path`). Tests indexing into the response envelope must use `dep["file_path"]`, since `DependencyData.target` is the only top-level `path`/`target` string.
- MCP `Edit replace_all` over a fragment that ends in `}));` matches every request struct in `server.rs`, not just the one you intend. Always anchor the `old_string` to a unique sibling line (e.g. the `path:` value) when bulk-rewriting in a multi-tool integration test file.
- `crate::test_helpers` in `crates/seshat-graph/src/lib.rs` is `pub(crate)`, so integration tests under `crates/seshat-graph/tests/` cannot import `test_conn`/`insert_ir` directly. Replicate them inline in the integration test file (5-10 LOC each); do not change the visibility of `test_helpers` just to share two helpers вАФ it would leak test-only API into the public crate surface.
- `DependencyData.truncated` reflects **IR-load** truncation (the `LIMIT` cap when reading `files_ir`), NOT the BFS `MAX_DEPENDENTS` cap. The BFS `TransitiveResult.truncated` flag is currently dropped at `dependencies.rs:329-332` (only `result.entries` is captured). To assert that BFS truncated, check `dependents.len() == MAX_DEPENDENTS` вАФ exposing BFS truncation through the public API is a separate concern not required by US-005.
- Bulk-inserting many IR rows in tests: open a single rusqlite `Transaction` and reuse one `Statement::prepare`d `INSERT` across all rows (~30√Ч faster than `insert_ir`-per-row in default test profile вАФ 90 ms for 3000 rows vs ~3 s).
- IR types (`Export`, `TypeDef`, `Function`) are constructed in MANY places вАФ not just parsers. Detector tests, graph tests, MCP tool tests, and server integration tests all build `ProjectFile` fixtures by hand. When adding a required field (no `#[serde(default)]` shortcut), expect ~30 callsites across `seshat-detectors/`, `seshat-graph/`, `seshat-mcp/`, `seshat-scanner/`, plus the storage roundtrip fixtures. A small Python script that finds the `<TypeName> { ... line: N, ... }` block and inserts `end_line: N,` after `line:` (preserving indentation, skipping blocks that already have `end_line:`) was ~30 LOC and rewrote 9 files in a single pass.
- For schema bumps (e.g. `IR_SCHEMA_VERSION` 7вЖТ8): always add a guard test (`current_schema_version_is_v8`) that pins the constant to its expected literal value AND that a fresh serialize emits that byte. Without it, an accidental future bump silently passes the version-prefix tests because they all reference `IR_SCHEMA_VERSION` as a variable.
- Tree-sitter parsers expose start/end positions through `node.start_position().row + 1` / `node.end_position().row + 1` (1-based line numbers). For `export_statement` and Rust `struct_item`/`enum_item`/etc., this gives the full source range of the declaration вАФ perfect for hunk-intersection logic. The existing `Function::end_line` (rust_parser.rs:456) is the canonical pattern; mirror it everywhere.
- `extract_exported_lexical` in `parser/mod.rs` is shared between TS and JS. Adding a parameter there fans out to BOTH parsers вАФ so `end_line` flows through the export_statement node uniformly. The function already has `#[allow(clippy::too_many_arguments)]` style trade-offs; one more positional arg is fine.
- `gix::diff::blob` re-exports `imara_diff::*`, so `gix::diff::blob::{Algorithm, diff, intern::InternedInput}` are all reachable from `seshat-graph` without adding a direct `gix-diff`/`imara-diff` dependency. The closure form of `Sink` (`impl<T: FnMut(Range<u32>, Range<u32>)> Sink for T`) is the easiest collector вАФ capture a `&mut Vec<Hunk>` in the closure and ignore the `()` return of `diff(...)`. imara_diff line ranges are 0-based half-open; convert to 1-based half-open with `+1` on both `start` and `end` (an empty `[s, s)` 0-based range stays empty after `+1`, which is what we want for pure insertion/deletion sites).
- Clippy enforces `clippy::type-complexity` under `-D warnings`. Public function signatures returning nested generics like `Result<Option<(Vec<u8>, Vec<u8>)>, GraphError>` will fail the lint. The fix is a `pub type BlobPair = (Vec<u8>, Vec<u8>);` alias that preserves the AC's tuple shape; do not reach for `#[allow(clippy::type_complexity)]` unless aliasing is actually impossible.
- For the `seshat-graph` blob/diff layer, prefer reading bytes via `repo.find_object(oid)?.data.clone()` rather than the streaming readers вАФ the data is already in memory and the size cap (`MAX_DIFF_FILE_SIZE = 5 MiB`) is enforced cheaply on `obj.data.len()` before any further work. `gix::ObjectId` derives `Copy`, so blob IDs flow freely through `Option<gix::ObjectId>` fields without `clone()` calls; use `.copied()` when extracting from `HashMap<String, gix::ObjectId>::get` rather than `.cloned()`.
- The "binary blob" heuristic git itself uses (NUL byte in first 8 KiB) is the right one for `seshat-graph`'s `read_blob_pair`. Returning `Ok(None)` (rather than `Err`) for binary OR oversized OR missing-on-disk lets the caller treat all three as "fall back to `Hunk::ALL`" вАФ keep the public surface consistent so callers don't need a 3-way match.

- **Learnings for future iterations:**
  - `DEFAULT_TRANSITIVE_DEPTH` belongs in `seshat-graph`, not
    `seshat-mcp`. The graph layer needs it for
    `compute_affected_symbols`, and the MCP tool layer can re-export
    via `pub const ... = GRAPH_DEFAULT_TRANSITIVE_DEPTH;`. Putting
    the canonical value in the lower crate avoids the circular
    dependency (graph cannot import from MCP) and gives the rest of
    the workspace a single source of truth.
  - For tests that exercise import resolution, the file extension
    matters вАФ `blob.bin` does NOT resolve via the suffix index
    (only `.rs`/`.ts`/`.py` etc. are tried), so a binary-blob test
    needs the file at e.g. `src/blob.rs` with NUL bytes injected
    into otherwise text-shaped content. The `is_binary_blob`
    heuristic is content-based (NUL byte in first 8 KiB), not
    extension-based, so this works without changing the resolver.
  - Refactoring `compute_affected_symbols` to take
    `&[ChangedFileWithBlobs]` (instead of `&[ChangedFile]`) lets the
    caller call `enumerate_changes_with_blobs` once and avoid
    re-walking the working tree from inside the symbol-resolution
    pass. The orchestration cost was ~2 LOC in `map_diff_impact`
    (mirror down to `ChangedFile` for the response data and for
    `compute_convention_risks`); the win is that the with-blobs
    pass is the only place the gix repo is opened.
  - Pure-deletion hunks have empty new ranges `[s, s)` and the
    naive `[start, end)` containment check `n in [s, s)` returns
    false for every line. The right rule is: a deletion at position
    `s` "touches" both new lines `s-1` (the line before the gap) and
    `s` (the line after); both adjacent symbols stay flagged. This
    is encoded once in `Hunk::touches_new_line`; the per-symbol
    range filter inlines the same logic so a symbol whose body ends
    right before a deletion is flagged via `s == line || s == end_line + 1`.
  - For per-symbol transitive blast radius, the BFS is file-level вАФ
    transitive entries have empty `import_names` (per US-001) so we
    cannot tell which transitive chain originated from which direct
    import within the changed file. The pragmatic rule used here is:
    if any direct importer of the symbol exists, fold ALL transitive
    entries (depth >= 2) into the count. This over-approximates but
    matches the "blast radius" intent вАФ the same file-level
    over-approximation applied by `query_dependencies` itself when
    multiple symbols share a target.
  - `Hunk::ALL` and the inclusive-intersection formula naturally
    collapse to `(line, end_line)` for every symbol вАФ no special
    casing required at the binary/oversized/added-file branch.
    Falling back to `vec![Hunk::ALL]` from a single
    `read_blob_pair() -> None` arm is sufficient; the per-symbol
    code path runs unchanged.
  - The MCP `tools/diff_impact.rs` test fixture (`affected` helper)
    is the only construction site outside the graph crate. Updating
    it with the two new fields was a 2-line change. There is no
    equivalent server.rs or call_logger fixture that constructs
    `AffectedSymbol` directly вАФ `call_logger::diff_impact_result`
    only inspects the JSON-serialised envelope.
