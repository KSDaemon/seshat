## Codebase Patterns
- Migrations are SQL files in `crates/seshat-storage/migrations/VN__name.sql`, embedded at compile time via `refinery::embed_migrations!("migrations")` in `crates/seshat-storage/src/db.rs`. There is **no** `crates/seshat-storage/tests/migrations.rs` file — migration tests live in `crates/seshat-storage/src/db.rs::tests` alongside `Database::open`.
- `Database::open` enables `PRAGMA foreign_keys = ON` at line 64 of `crates/seshat-storage/src/db.rs`, so any FK with `ON DELETE CASCADE` is actually enforced (don't assume otherwise). Consequence for tests: a child table with a FK to `branches(branch_id)` requires inserting the parent `branches` row first, even in setup helpers.
- Existing migration schema reference: `V11__branches_table.sql` defines `branches(branch_id TEXT PRIMARY KEY, last_scanned_commit, last_scanned_at, snapshot_source, created_at)`. New branch-scoped tables FK to `branches(branch_id)`.
- Repository pattern (per `repo_metadata_repository.rs`): each repo file holds the SQLite impl struct; the `*Repository` trait lives in `repository/mod.rs` (NOT in the impl file). Each impl file `pub use`s its `Sqlite*` struct via `repository/mod.rs`, and the trait is re-exported from `lib.rs`. UPSERT uses `INSERT … ON CONFLICT(key) DO UPDATE SET value = excluded.value`. Constructor takes `Arc<Mutex<Connection>>`; locking goes through the shared `lock_conn` helper. Public methods carry `#[tracing::instrument(skip(self))]`.
- Test helper for on-disk DBs: `TempDir::new("name")` from `db.rs::tests` (defined inline ~line 235) — use it whenever a test must reopen the same DB to exercise idempotency.
