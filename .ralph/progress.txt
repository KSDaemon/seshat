## Codebase Patterns
- Convention nodes in the `nodes` table use `ext_data.source = "auto_detected"` or `"user"` — filter with `json_extract(ext_data, '$.source')` in SQL queries
- Workspace dependencies are declared in root Cargo.toml `[workspace.dependencies]`, then crates reference them via `workspace = true`
- Config structs in seshat-core/src/config.rs use `#[serde(default, rename_all = "snake_case")]` and have explicit `Default` impls with unit tests
- seshat.example.toml documents all config fields as comments with defaults shown — aspirational fields may exist there before the Rust struct has them
- Pre-commit hook runs cargo clippy + cargo fmt; if it modifies files (e.g. Cargo.lock), the commit fails and needs retry with those files staged
- seshat-mcp crate has NO direct storage access — queries go through seshat-graph
- The binary entrypoint is seshat-bin which calls seshat-cli::run()
- DB path for scanned projects: `dirs::data_dir()/seshat/repos/{project_name}.db`
- rmcp's `RunningService::waiting()` takes ownership — cannot use in multiple select! branches; use `cancel()` or structure control flow differently
- CLI commands that need async create their own `tokio::runtime::Runtime` since `run()` is sync
- MCP tool responses use `ResponseEnvelope<T>` (success) and `ErrorEnvelope` (error) from seshat-mcp/src/envelope.rs — all tools wrap results in these envelopes for a uniform JSON schema
- Duration measurement: pass `Instant::now()` at tool entry → `ResponseEnvelope::success(..., start)` computes elapsed ms
