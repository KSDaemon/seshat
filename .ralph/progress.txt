## Codebase Patterns
- Migrations are SQL files in `crates/seshat-storage/migrations/VN__name.sql`, embedded at compile time via `refinery::embed_migrations!("migrations")` in `crates/seshat-storage/src/db.rs`. There is **no** `crates/seshat-storage/tests/migrations.rs` file вАФ migration tests live in `crates/seshat-storage/src/db.rs::tests` alongside `Database::open`.
- `Database::open` enables `PRAGMA foreign_keys = ON` at line 64 of `crates/seshat-storage/src/db.rs`, so any FK with `ON DELETE CASCADE` is actually enforced (don't assume otherwise). Consequence for tests: a child table with a FK to `branches(branch_id)` requires inserting the parent `branches` row first, even in setup helpers.
- Existing migration schema reference: `V11__branches_table.sql` defines `branches(branch_id TEXT PRIMARY KEY, last_scanned_commit, last_scanned_at, snapshot_source, created_at)`. New branch-scoped tables FK to `branches(branch_id)`.
- Repository pattern (per `repo_metadata_repository.rs`): each repo file holds the SQLite impl struct; the `*Repository` trait lives in `repository/mod.rs` (NOT in the impl file). Each impl file `pub use`s its `Sqlite*` struct via `repository/mod.rs`, and the trait is re-exported from `lib.rs`. UPSERT uses `INSERT вА¶ ON CONFLICT(key) DO UPDATE SET value = excluded.value`. Constructor takes `Arc<Mutex<Connection>>`; locking goes through the shared `lock_conn` helper. Public methods carry `#[tracing::instrument(skip(self))]`.
- Test helper for on-disk DBs: `TempDir::new("name")` from `db.rs::tests` (defined inline ~line 235) вАФ use it whenever a test must reopen the same DB to exercise idempotency.
- `BranchId` is `pub struct BranchId(pub String)` in `seshat-core/src/ids.rs`. To pass it to repos that take `&str` (e.g. `BranchMetadataRepository`, `EmbeddingRepository`), use `&branch.0` вАФ no helper method needed.
- The watcher hot tier (`crates/seshat-watcher/src/hot_tier.rs`) does NOT invoke `scan_project` or write `workspace_crates` itself вАФ it only refreshes per-file IR. The full scan path is the single owner of `branch_metadata.workspace_crates` writes; mirror-changes for incremental re-scans are unnecessary.
- The graph crate's shared test fixtures live in `crates/seshat-graph/src/lib.rs::test_helpers` (NOT in a `tests/` directory or a standalone `test_helpers.rs` file). `test_conn()` returns an in-memory DB with all migrations applied; `insert_ir(conn, branch_id, file)` writes `files_ir` + symbol-index rows but does NOT seed `branches` вАФ any helper that writes to a branch-FK-bearing table (e.g. `branch_metadata` from V14) must `INSERT OR IGNORE INTO branches(branch_id)` first.
- `SqliteBranchRepository::create_snapshot` (crates/seshat-storage/src/repository/branch_repository.rs) is the single place where a new branch's data is duplicated from a source branch. Any new per-branch table introduced after V11 must add a matching `INSERT вА¶ SELECT` inside the same transaction вАФ otherwise the new branch will start with that table empty even when its files_ir mirrors the source. Existing copies: nodes, edges, files_ir, symbol_definitions, symbol_imports, branch_metadata (V14, FW-5).
- The `SqliteBranchRepository.conn` field is private but visible inside the `tests` module in the same file (Rust private-field visibility). Tests in branch_repository.rs already use `branch_repo.conn.lock().unwrap()` for raw-SQL assertions вАФ reuse this pattern when a new test needs to verify columns the public repo API doesn't expose (e.g. `branch_metadata.updated_at`).
