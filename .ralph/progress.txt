## Codebase Patterns
- Migrations live in `crates/seshat-storage/migrations/` and are auto-discovered by `refinery::embed_migrations!("migrations")` in `crates/seshat-storage/src/db.rs` ‚Äî adding a new `V<n>__name.sql` file is enough; no registry to update.
- Repository pattern: each entity gets a trait in `crates/seshat-storage/src/repository/mod.rs` and a `SqliteXxxRepository` struct (with `Arc<Mutex<Connection>>`) in its own `xxx_repository.rs` sibling file. Re-export the trait + struct from `mod.rs` and from `crates/seshat-storage/src/lib.rs` for downstream consumers.
- `lock_conn(&self.conn)` in `mod.rs` is the shared helper for acquiring the connection lock ‚Äî use it instead of inlining the lock/`map_err` boilerplate.
- For chunked `IN (?,?,?)` bulk lookups, build placeholders via `(1..=chunk.len()).map(|i| format!("?{i}")).collect::<Vec<_>>().join(", ")` and convert `&[&str]` to `Vec<&dyn rusqlite::types::ToSql>` per chunk. PRD-specified chunk size is 500 (well under SQLite's default `SQLITE_MAX_VARIABLE_NUMBER`).
- For SQL string-mapped enums (e.g. `DecisionState`/`DecisionNature`/`DecisionWeight`) implement `as_sql_str(&self) -> &'static str` and `from_sql_str(&str) -> Result<Self, StorageError>` instead of `Display`/`FromStr` ‚Äî this keeps the SQL contract explicit and isolates DB serialisation from any user-facing formatting that might evolve later.
- `rusqlite::Connection::query_row` short-circuits on `QueryReturnedNoRows` ‚Äî match it to convert into `Ok(None)` for nullable lookups; otherwise propagate `StorageError::from(e)`.
- The PRD source-of-truth lives in `.ralph/tasks/prd-merge-aware-decisions.md`. The `.ralph/prd.json` userStories are the operational view; the markdown contains the full schema, queries, and rationale.
- Pre-existing flaky test in `crates/seshat-cli/src/init/tests.rs` (`opencode_global_config_dir_empty_xdg_falls_back_to_home`) ‚Äî env-var (`XDG_CONFIG_HOME`) test isolation issue independent of any change. Re-run on its own to confirm green.
- Branch lifecycle: post-US-003, branches must be EXPLICITLY registered in the `branches` table to show up in `list_branches`. Three entry points keep the registry in sync ‚Äî `ensure_branch_exists` (idempotent), `set_last_scanned_commit` (UPSERT), `create_snapshot` (registers source via INSERT OR IGNORE + target with `snapshot_source`). `switch_branch` implicitly registers, and `scan_project_with_progress` calls `ensure_branch_exists` at the top of every scan. `delete_branch` removes the registry row LAST inside the cleanup transaction (after edges/nodes/files_ir) so a partial failure doesn't orphan data behind a missing sentinel.
- For SQLite UPSERTs the canonical pattern is `INSERT INTO t (...) VALUES (...) ON CONFLICT(<pk>) DO UPDATE SET col = excluded.col` ‚Äî using `excluded` (not literal placeholders) lets you reuse the values without rebinding parameters.
- `unixepoch()` resolution is per-second; tests asserting that two consecutive timestamps differ need `std::thread::sleep(Duration::from_millis(1100))` between writes (‚â•1.1s, not 1.0s, so the second tick is safely past the boundary on slow CI hosts).
- `rusqlite::query_row` returning `Result<Option<String>, _>` via `row.get(0)` correctly unwraps NULL columns into `Ok(None)` ‚Äî but missing rows still come back as `QueryReturnedNoRows`, which must be matched separately. So the canonical "nullable column for a possibly-missing row" handler has *two* `Ok` arms: `Ok(value)` (row found, value is `Option<T>`) and `Err(QueryReturnedNoRows) => Ok(None)`.
- For "exactly N statements were issued" tests, enable rusqlite's `trace` feature (workspace `Cargo.toml`) and use `Connection::trace_v2(TraceEventCodes::SQLITE_TRACE_STMT, Some(fn_pointer))` inside the test. The callback is a `fn` pointer (not a closure) so state must be parked in a `thread_local! { static FOO: RefCell<Vec<String>> }` that the function reads/writes. Always disable tracing again at the end of the test (`trace_v2(.., None)`) so a parallel test on the same OS thread doesn't inherit a stale callback.
- When a function wraps `bulk_fetch_*` plus a write transaction, do the bulk fetch BEFORE acquiring the write-side `lock_conn` guard. `SqliteDecisionRepository::get_by_hashes` calls `lock_conn` itself, so calling it while the persist guard is held would deadlock. Pre-compute outside, store in a `HashSet`, then take the lock and do the transaction.
- When a downstream story's AC says "rejections live in `decisions` now", the obsolete tests that pin the OLD behaviour (e.g. `persist_conventions_skips_user_rejected` asserting a `user_rejected=1` node survives) must be REMOVED, not made to pass. Replace them with regression tests that pin the NEW invariant (e.g. `persist_conventions_no_longer_preserves_user_rejected_nodes` asserting that node IS now deleted). Old behaviour ‚âÝ new behaviour, even if the test name still parses.

**Learnings for future iterations:**
- `query_row` returns `rusqlite::Error::QueryReturnedNoRows` for an empty result ‚Äî match it explicitly to translate "row absent" into `Ok(None)` instead of letting it bubble up as a hard error.
- `query_map` callbacks are `FnMut(&Row) -> rusqlite::Result<T>`. To return a typed `StorageError` for inner failures (enum-parse, JSON-decode), wrap the inner value in `rusqlite::Result<Result<Decision, StorageError>>` and unwrap the double `?` at the call site (`row??`). Trying to convert directly to `StorageError` inside the callback fails because `rusqlite::Error::FromSqlConversionFailure` requires `Box<dyn Error>` and erases the `StorageError` variant.
- The PRD reference signature `fn upsert(&self, d: Decision)` consumes by value, but the trait was already tightened to `&Decision` in US-001. US-008 will need bulk UPSERT-skip behaviour, so consuming would have forced clones at every call site ‚Äî `&Decision` was the right call.
- For TEXT PRIMARY KEYs, SQLite's `EXPLAIN QUERY PLAN` reports `SEARCH decisions USING INDEX sqlite_autoindex_decisions_1 (description_hash=?)`. The `SEARCH ... USING INDEX` substring is the load-bearing part ‚Äî `assert!(plan.contains("SEARCH") && !plan.contains("SCAN"))` is sufficient and stable across SQLite versions.
- Inline format args (`format!("?{i}")` instead of `format!("?{}", i)`) are the clippy-clean style in this repo ‚Äî `cargo clippy --all-targets -- -D warnings` enforces it.
- `delete` idempotency: per the AC, calling `delete` on a non-existent hash must not error. Don't reach for `StorageError::NotFound` here ‚Äî `conn.execute(...)?; Ok(())` is the right shape, with the deleted row count silently dropped.
- `params!` macro can't be used on heterogeneous `&[&dyn ToSql]` for variadic IN-clauses ‚Äî collect into `Vec<&dyn rusqlite::types::ToSql>` and pass `.as_slice()` to `query_map`.

**Learnings for future iterations:**
- The PRD AC for US-003 doesn't explicitly say `switch_branch` should ensure existence or that `delete_branch` should clean up the branches row, but BOTH are necessary to keep `list_branches` semantically coherent: switching to a branch that doesn't show up in the list is a footgun, and deleting a branch that leaves a stale registry row is a memory leak. Implemented both as natural side-effects.
- `scan_project_with_progress` lives in `seshat-scanner`, not `seshat-cli`. It already imports `seshat_storage`, so adding `ensure_branch_exists` here was a 2-line change (one import addition, one call). This is the right layer because every scan path (CLI `scan`, serve background sync, watcher rescan, MCP refresh) eventually goes through this function ‚Äî registering once at the orchestrator level avoids scattering the call across N callers.
- The PRD AC mentions `create_snapshot(source, target) also INSERTs into branches with snapshot_source = source.0`. The natural reading is "INSERT only the target row". But the source must also be registered, otherwise after `create_snapshot(main, feature)`, `list_branches()` returns `[feature]` only, breaking GC tests in `seshat-cli/src/db.rs` that assert `main` is in the list. The fix is `INSERT OR IGNORE` for source + `ON CONFLICT DO UPDATE SET snapshot_source = excluded.snapshot_source` for target ‚Äî both within the existing `unchecked_transaction` so the snapshot is atomic.
- `unixepoch()` is per-second resolution. The test `set_last_scanned_commit_bumps_last_scanned_at` needs `sleep(1100ms)` (NOT 1000ms) between writes ‚Äî at exactly the second boundary the second write can land in the same epoch second on fast hardware.

So the canonical pattern for "nullable column on a possibly-missing row" needs both arms: `Ok(value) => Ok(value)` (row exists; value already wraps the nullable column) AND `Err(QueryReturnedNoRows) => Ok(None)`. Forgetting the second arm makes `get_last_scanned_commit` panic with a generic SQL error on unknown branches instead of returning `Ok(None)`.
- The `_var_name` underscore prefix on default-impl trait method parameters in `mod.rs` was needed only to silence the unused-arg lint while the bodies were `unimplemented!()`. **Learnings for future iterations:**
- `rusqlite::trace::trace_v2` takes `Option<fn(TraceEvent<'_>)>` ‚Äî a fn POINTER, not a closure. State must live in a `thread_local!` because the callback can't capture. Pattern: `thread_local! { static CAPTURED_SQL: RefCell<Vec<String>> = const { RefCell::new(Vec::new()) }; }` plus a free function `fn trace_capture(event: TraceEvent<'_>)`. Always disable tracing again at the end of the test (`trace_v2(.., None::<fn(_)>)` works without explicit type because `None` infers from `Option<fn(TraceEvent<'_>)>`); cargo test reuses worker threads, so a stray callback would leak across tests.
- `rusqlite 0.37`'s `TraceEvent::Stmt` is a TUPLE variant `Stmt(StmtRef<'s>, &'s str)`, not `Stmt { sql, .. }` (different from older docs and trace_v2 in older C-bindings). Pattern-match accordingly.
- When wrapping `bulk_fetch_*` + a write transaction around the same `Arc<Mutex<Connection>>`, ALWAYS perform the bulk fetch BEFORE the `lock_conn` guard for the transaction. The repository methods call `lock_conn` themselves, so a fetch-while-holding-the-lock would deadlock. The pattern that works: `let result = repo.get_by_hashes(&hash_refs)?;` followed by `let guard = lock_conn(conn)?;` and the BEGIN/INSERT/COMMIT block uses `result` without re-querying.
- When AC says "the exception is removed", the obsolete "exception works" test is NOT salvageable by parameter tweaks ‚Äî REMOVE it and add a regression test that pins the NEW invariant (e.g. "node with `user_rejected=1` IS now deleted"). Mixing old- and new-behaviour assertions in the same test guarantees future churn.
- For an AC like "1 SELECT, not N", a behaviour-only test (verify outputs) is insufficient; the regression guard must actually count SQL statements. Wiring `trace_v2` once is worth the upfront cost ‚Äî N+1 regressions are otherwise indistinguishable from correct behaviour at the assertion level.
- For tests that need a stable SQL substring filter, prefer `to_uppercase()` then `contains("SELECT")` + `contains("FROM DECISIONS")` over regex. Cheap, no extra deps, and the existing chunked `get_by_hashes` SQL renders deterministically as `SELECT ... FROM decisions WHERE description_hash IN (?1, ?2, ...)`.
- `aggregated.iter().zip(hashes.iter())` (where `hashes` is the precomputed `Vec<String>`) lets the INSERT loop access both the convention struct and the matching hash without recomputing ‚Äî and without rebuilding a `HashMap<String, &AggregatedConvention>`. Index-aligned vectors are fine here because the producer (the same function) controls both.
