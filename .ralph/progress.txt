## Codebase Patterns
- `crates/seshat-cli/src/update.rs::extract_binary` expects archives whose **top-level directory** is `seshat-{target}-v{version}` (matches `expected_dir = format!("seshat-{target}-v{version}")` in tests around line 1198). Any change to `release.yml`'s archive layout must keep that contract.
- `release.yml` currently has a single `Package binary` step (one `ARCHIVE_NAME=` assignment, ~line 96) used for both `.tar.gz` (Unix) and `.zip` (Windows) packaging — the AC referencing "line 52" in the PRD is a stale pointer; treat it as "the ARCHIVE_NAME assignment in the build-binaries job".
- The release workflow trigger is `tags: ['v*']`, so `${GITHUB_REF_NAME}` inside that workflow is always a tag like `v1.2.3` — safe to use directly in archive names; matches the `v{version}` portion that `extract_binary` expects.
- For workflow YAML lint, use `actionlint` (already installed at `/opt/homebrew/bin/actionlint`); pure `python3 -c "import yaml"` is unavailable on this host.
- Workspace MSRV is `rust-version = "1.85"` (root `Cargo.toml`). When picking dependency versions, choose the highest stable version whose published `rust-version` is ≤ 1.85 — e.g. `zip = "7"` (rust=1.83) is OK, but `zip = "8"` (rust=1.88) silently raises the project's effective MSRV. Use `cargo info <crate>@<ver>` or `crates.io/api/v1/crates/<crate>/versions` to check rust-version per release before adding a dep.
- New dep imports use the canonical crates.io spelling (with dashes): `self-replace.workspace = true`. Cargo automatically maps the dash-named crate to the underscore Rust identifier (`use self_replace`) — no `package = "..."` rename needed. Same pattern as the existing `tracing-subscriber.workspace = true` / `use tracing_subscriber`.
- For dual-archive (Unix `.tar.gz` + Windows `.zip`) self-update, the workspace already has `flate2 = "1"` + `tar = "0.4"` for tar; pair with `zip = { version = "7", default-features = false, features = ["deflate"] }`. The `deflate` feature on zip 7.x pulls `zopfli` (encode) + `flate2/zlib-rs` (decode) — both light, both already aligned with our existing `flate2`.
- Target-string convention for windows-vs-unix dispatch in `update.rs`: branch on `target.ends_with("windows-msvc")` rather than `cfg!(target_os="windows")`. `current_target()` is the single source of truth for which triple we're shipping — gating on its return value lets unit tests pass arbitrary target strings (e.g. force "x86_64-pc-windows-msvc" on a macOS host) and keeps build-host/runtime-target separated. This is what `find_binary_asset` and `fetch_checksum_for_asset` use.
- `current_target()` only adds positive arms for *supported* triples; the trailing `_ => "unsupported"` default catches every unsupported (arch, os) pair. So aarch64+windows, riscv, wasm, etc. all fall through to "unsupported" without explicit arms — only add an arm when adding real support.
- Archive extraction in `update.rs` dispatches on `archive_path` filename suffix (`.zip` → `extract_zip`, otherwise → `extract_tar_gz`). Compose the expected on-disk binary leaf via `format!("seshat{}", std::env::consts::EXE_SUFFIX)` — never hard-code `"seshat"` or `"seshat.exe"` separately; `EXE_SUFFIX` is the OS-correct empty-or-`.exe` shim and keeps Unix/Windows code paths converged.
- Zip safety constraints in `extract_zip` mirror the tar path: (1) skip empty entry names, (2) skip entries with any `..` component, (3) `enclosed_name()` returns `None` for traversal-prone names — treat that as a skip, (4) `canonicalize` the joined destination and require it to start with `dest_dir`. Items 3 and 4 are defense-in-depth duplicates — keep both, since `enclosed_name` rejects literal `..` while canonicalize rejects symlink-based escapes.
- Cross-platform zip fixture builder for tests: `fn build_zip_archive(entries: &[(&str, &[u8])]) -> Vec<u8>` using `zip::ZipWriter` with `SimpleFileOptions::default().compression_method(zip::CompressionMethod::Deflated)`. Treat names ending with `/` as `add_directory`, others as `start_file` + `write_all`. This runs on macOS/Linux/Windows hosts so windows-target zip extraction tests don't need `cfg(windows)` gating.
- `replace_binary` in `update.rs` calls `self_replace::self_replace(new_binary)` — this **replaces the currently running executable** (uses `current_exe()` internally). Consequence: you cannot exercise `replace_binary` directly from a `cargo test` unit test, because doing so would overwrite the live test binary. Test the error-mapping helper (`map_replace_error`) with synthetic `io::Error` values instead. The `target_exe` argument to `replace_binary` is now used **only for error display** (the `Permission denied updating {target}…` line) — `self_replace` derives the real target from `current_exe()` itself.
- Windows vs Unix permission-denied hint: `map_replace_error` uses `#[cfg(windows)] let hint = "Try running as Administrator."; #[cfg(not(windows))] let hint = "Try: sudo seshat update";`. The CliError `reason` string also branches on `cfg!(windows)` (`"permission denied; try running as Administrator"` vs `"permission denied; try sudo seshat update"`) so callers/tests can introspect the platform-specific hint without parsing stderr. When changing the hint text, update both the eprintln string AND the `reason` string in lockstep.
- `cfg!(target_os = "windows")` early-returns inside `run_self_update` are obsolete now that `self_replace` handles Windows. The Homebrew probe is still platform-conditional but lives in `detect_install_method()` (which returns `Direct` on Windows). Don't reintroduce a top-of-`run_self_update` Windows guard — it's a regression vector for the self-update flow.
- `self_replace::self_delete_outside_path(p)` is **NOT** a `.exe.old`-cleanup helper, despite what its name suggests. In self_replace 1.5.0 it unconditionally schedules deletion of the running executable on **both** Unix (calls `unix::self_delete(&exe)`, ignoring `p` via `let _ = p;`) and Windows (calls `windows::schedule_self_deletion_on_shutdown` which renames current_exe out of its install dir). The `p` argument is a fallback hint for where NOT to put temp files during uninstall flows that wipe a folder, **not** a "skip if inside this path" check. Calling it on every startup would brick the binary. Verified by reading `~/.cargo/registry/src/index.crates.io-*/self-replace-1.5.0/src/{lib.rs,unix.rs,windows.rs}` directly. If a future PRD or AC asks for this call, push back — the API doesn't do what its name implies.
- self_replace 1.5's `self_replace()` already cleans up after itself: on Windows it renames current_exe to `<system_temp>/<random>.__relocated__.exe` and spawns a `<system_temp>/<random>.__selfdelete__.exe` helper that waits for the parent process to exit, then `DeleteFileW`s the relocated binary (then dies via DELETE_ON_CLOSE). Net result: no `seshat.exe.old` left in the install dir under normal flow. Defensive `<current_exe>.old` cleanup at startup (US-006's `cleanup_stale_old_binary`) is purely a guard for manual / recovery / future-version scenarios where a `.old` might appear.
- Pattern for `cfg(windows)`-only helpers with testable inner core: expose `pub fn cleanup_stale_old_binary()` as the call-site API (cfg(windows)-gated body, no-op on Unix), and a private `#[cfg(windows)] fn cleanup_stale_old_binary_at(path: &Path)` that takes an explicit path argument. Tests on Windows exercise the path-injected variant against a tempdir; tests on Unix call the public wrapper and assert no panic. Avoids needing to fake `std::env::current_exe()` from a unit test.
- To append a literal suffix to an existing path (e.g. `seshat.exe` → `seshat.exe.old`), do **not** use `Path::with_extension("exe.old")` — that **replaces** the last extension segment (`seshat.exe` → `seshat.old`, dropping the `.exe`). The portable, OS-correct approach: `let mut s: OsString = path.as_os_str().to_owned(); s.push(".old"); PathBuf::from(s)`. Works regardless of whether the input has `.exe`, `.tar.gz`, or no extension.
- Pattern for end-to-end-style coverage of the `run_self_update` chain *without* invoking `replace_binary` (which corrupts the cargo-test binary via `self_replace::self_replace` → `current_exe()`): compose `build_zip_archive` (or tar equivalent) + `Sha256::new`/`update`/`finalize` → hex + `verify_sha256(archive_path, hex)` + `extract_binary(archive_path, dest, version)` + optional `preflight_check`. This covers extension-dispatch (`extract_binary`), zip/tar safety guards, `EXE_SUFFIX`-aware path composition, SHA-256 verify (both match and mismatch), and the staging-temp-dir cleanup contract — every windows-msvc code path *except* the OS-replace step. The replace step is verified at the helper level (`map_replace_error` with synthetic `io::Error`) and at runtime via Windows CI under US-008.
- On Windows, `std::process::Command::new(invalid_pe_file).output()` returns `Err(io::Error)` (CreateProcessW raises ERROR_BAD_EXE_FORMAT for non-PE bytes). This is the cleanest way to trigger `preflight_check`'s `.map_err` cleanup branch from a test — write 13 bytes of `b"not a PE file"` to `seshat.exe` and call `preflight_check`. No need for a real failing-on-`--version` stub binary, which would require cross-compiling something or shipping an embedded PE blob.
- For "no observable side effect" assertions where stderr can't be cheaply captured (no FD 2 dup), lock on the cache *file content* before+after rather than parsing eprintln. The fresh-cache fast path in `check_and_print_update_notice_inner` deliberately does NOT rewrite the cache (no network, no fresh data), so byte equality of cache content is a proxy for "fast path was taken, no rewrite or fetch happened".
- When a PRD AC asserts the existence of test infrastructure ("reuse existing mock-server helpers", "use the existing X pattern"), grep before trusting. PRD-as-source-of-truth fails when the PRD was drafted from an outdated mental model. US-006 (self_delete_outside_path) and US-007 (mock-server helpers) are both instances of this same anti-pattern. Verify with `Grep` first; deviate with documented rationale when the premise is wrong.

- **PRD AC was broken; deviated with a safe wrapper.** The PRD asked for `let _ = self_replace::self_delete_outside_path(env!("CARGO_PKG_NAME"));` in `lib.rs::run()`. Reading `~/.cargo/registry/src/index.crates.io-*/self-replace-1.5.0/src/{lib.rs,unix.rs,windows.rs}` confirmed this would brick the binary on every invocation:
  - On Unix, `self_delete_outside_path(p)` calls `unix::self_delete(&exe)` ignoring `p` (`let _ = p;`) — which `canonicalize()` + `fs::remove_file()`s the running executable.
  - On Windows, it calls `windows::schedule_self_deletion_on_shutdown` which renames current_exe out of its install dir and spawns a `.__selfdelete__.exe` helper to delete it post-shutdown. The `p` arg is just a fallback hint for where NOT to put temp files (uninstall-folder protection), **not** a "skip if inside path" check. The docstring (lib.rs:156-160) confirms: `"This is equivalent to self_delete on Unix, but it instructs the deletion logic to not place temporary files in the given path…"`.
  - Also: self_replace 1.5's own `self_replace()` already cleans up its relocated old binary via the spawned `.__selfdelete__.exe` helper (windows.rs:280-287), so there is no `seshat.exe.old` left in the install dir for the PRD's cleanup call to chase. The PRD's leftover-file model doesn't match the crate's actual behavior.
- Implemented a safe semantic replacement that satisfies the user-story intent (cleanup leftover .old in install dir) without the destructive API:

- **Learnings for future iterations:**
  - `self_replace::self_delete_outside_path` is a misleadingly-named API: it's a self-deletion helper for **uninstall** flows, not a leftover-file cleanup helper. Despite the name, the path argument is just an instruction about where NOT to place temp files during the deletion (used when wiping a folder); it is not a "skip if exe is inside this path" gate. If a future PRD asks for it as a cleanup primitive, push back — the API is destructive on every platform and ignores its argument's intent on Unix.
  - When a PRD AC specifies a buggy/destructive call but the user-story intent is clear (cleanup behavior), prefer a safe semantic replacement over silent literal compliance. Document the deviation thoroughly in: (1) the commit message, (2) the PRD `notes` field, (3) progress.txt with full rationale and source-pointer evidence. The PRD's literal text is sometimes wrong; the user story description usually captures the real intent.

Tests on Windows exercise the path-injected variant against a tempdir; tests on Unix call the public wrapper and assert it doesn't panic. This avoids the unsolvable "fake `current_exe()` from inside a cargo test" problem.
