## Codebase Patterns
- Migrations live in `crates/seshat-storage/migrations/` and are auto-discovered by `refinery::embed_migrations!("migrations")` in `crates/seshat-storage/src/db.rs` — adding a new `V<n>__name.sql` file is enough; no registry to update.
- Repository pattern: each entity gets a trait in `crates/seshat-storage/src/repository/mod.rs` and a `SqliteXxxRepository` struct (with `Arc<Mutex<Connection>>`) in its own `xxx_repository.rs` sibling file. Re-export the trait + struct from `mod.rs` and from `crates/seshat-storage/src/lib.rs` for downstream consumers.
- `lock_conn(&self.conn)` in `mod.rs` is the shared helper for acquiring the connection lock — use it instead of inlining the lock/`map_err` boilerplate.
- For chunked `IN (?,?,?)` bulk lookups, build placeholders via `(1..=chunk.len()).map(|i| format!("?{i}")).collect::<Vec<_>>().join(", ")` and convert `&[&str]` to `Vec<&dyn rusqlite::types::ToSql>` per chunk. PRD-specified chunk size is 500 (well under SQLite's default `SQLITE_MAX_VARIABLE_NUMBER`).
- For SQL string-mapped enums (e.g. `DecisionState`/`DecisionNature`/`DecisionWeight`) implement `as_sql_str(&self) -> &'static str` and `from_sql_str(&str) -> Result<Self, StorageError>` instead of `Display`/`FromStr` — this keeps the SQL contract explicit and isolates DB serialisation from any user-facing formatting that might evolve later.
- `rusqlite::Connection::query_row` short-circuits on `QueryReturnedNoRows` — match it to convert into `Ok(None)` for nullable lookups; otherwise propagate `StorageError::from(e)`.
- The PRD source-of-truth lives in `.ralph/tasks/prd-merge-aware-decisions.md`. The `.ralph/prd.json` userStories are the operational view; the markdown contains the full schema, queries, and rationale.
- Pre-existing flaky test in `crates/seshat-cli/src/init/tests.rs` (`opencode_global_config_dir_empty_xdg_falls_back_to_home`) — env-var (`XDG_CONFIG_HOME`) test isolation issue independent of any change. Re-run on its own to confirm green.
- Branch lifecycle: post-US-003, branches must be EXPLICITLY registered in the `branches` table to show up in `list_branches`. Three entry points keep the registry in sync — `ensure_branch_exists` (idempotent), `set_last_scanned_commit` (UPSERT), `create_snapshot` (registers source via INSERT OR IGNORE + target with `snapshot_source`). `switch_branch` implicitly registers, and `scan_project_with_progress` calls `ensure_branch_exists` at the top of every scan. `delete_branch` removes the registry row LAST inside the cleanup transaction (after edges/nodes/files_ir) so a partial failure doesn't orphan data behind a missing sentinel.
- For SQLite UPSERTs the canonical pattern is `INSERT INTO t (...) VALUES (...) ON CONFLICT(<pk>) DO UPDATE SET col = excluded.col` — using `excluded` (not literal placeholders) lets you reuse the values without rebinding parameters.
- `unixepoch()` resolution is per-second; tests asserting that two consecutive timestamps differ need `std::thread::sleep(Duration::from_millis(1100))` between writes (≥1.1s, not 1.0s, so the second tick is safely past the boundary on slow CI hosts).
- `rusqlite::query_row` returning `Result<Option<String>, _>` via `row.get(0)` correctly unwraps NULL columns into `Ok(None)` — but missing rows still come back as `QueryReturnedNoRows`, which must be matched separately. So the canonical "nullable column for a possibly-missing row" handler has *two* `Ok` arms: `Ok(value)` (row found, value is `Option<T>`) and `Err(QueryReturnedNoRows) => Ok(None)`.
- For "exactly N statements were issued" tests, enable rusqlite's `trace` feature (workspace `Cargo.toml`) and use `Connection::trace_v2(TraceEventCodes::SQLITE_TRACE_STMT, Some(fn_pointer))` inside the test. The callback is a `fn` pointer (not a closure) so state must be parked in a `thread_local! { static FOO: RefCell<Vec<String>> }` that the function reads/writes. Always disable tracing again at the end of the test (`trace_v2(.., None)`) so a parallel test on the same OS thread doesn't inherit a stale callback.
- When a function wraps `bulk_fetch_*` plus a write transaction, do the bulk fetch BEFORE acquiring the write-side `lock_conn` guard. `SqliteDecisionRepository::get_by_hashes` calls `lock_conn` itself, so calling it while the persist guard is held would deadlock. Pre-compute outside, store in a `HashSet`, then take the lock and do the transaction.
- When a downstream story's AC says "rejections live in `decisions` now", the obsolete tests that pin the OLD behaviour (e.g. `persist_conventions_skips_user_rejected` asserting a `user_rejected=1` node survives) must be REMOVED, not made to pass. Replace them with regression tests that pin the NEW invariant (e.g. `persist_conventions_no_longer_preserves_user_rejected_nodes` asserting that node IS now deleted). Old behaviour ≠ new behaviour, even if the test name still parses.
- For functions that batch multiple writes inside an `apply_*` transaction envelope (BEGIN at the top, COMMIT at the bottom, helpers in between), the canonical pattern is: instantiate `SqliteXxxRepository::new(conn.clone())` INSIDE each helper rather than threading a long-lived repo through the call graph. The `Arc::clone` is cheap, transaction state lives on the `Connection` (not the `Mutex`), and dropping the per-helper lock guard between calls is safe — the open SQLite transaction persists. Helpers stay independently testable this way.
- When a refactor drops fields from helper-function signatures but leaves the `enum` variant shape intact for source compat, prefer the `..` rest pattern at the dispatch match arm (`Variant { used, .. } => helper(used)`) over underscore-prefixing the unused params on the helper. Rest patterns survive future field churn without churning helper signatures, and keep the helper's own argument list semantically clean.
- Tests asserting "old behaviour gone" (e.g. `count(user-source nodes) == 0`) and "new behaviour present" (e.g. `decisions.get_by_hash(...)` returns a row) are NOT redundant. The first pins the negative invariant ("nothing leaked into the legacy table"); the second pins the positive invariant ("durable state was written to the new table"). Keep both — they fail independently, and one passing while the other fails is a real bug class (e.g. dual-write code that forgot to remove the legacy write).
- SQLite's `EXPLAIN QUERY PLAN` uses the FROM-clause **alias**, not the raw table name. If you `LEFT JOIN decisions d`, the plan reads `SEARCH d USING ...` — `assert!(plan.contains("SEARCH decisions"))` fails. Always assert against the alias the production SQL uses, AND check for the underlying index name (`sqlite_autoindex_decisions_1` for V12's PK auto-index) to pin both halves of the contract.
- LEFT JOIN anti-join (`LEFT JOIN b ON b.fk = a.pk WHERE b.<col> IS NULL`) for "rows in A with no match in B": pick a column on the right side that is BOTH the PK AND `NOT NULL` for the WHERE predicate. With V12's `decisions.description_hash TEXT NOT NULL PRIMARY KEY`, `WHERE d.description_hash IS NULL` is the canonical "no match" predicate — unmatched LEFT JOIN rows have NULL on every d-side column, but a nullable column risks false positives.
- The TUI's `setup_empty_db` test fixture in `crates/seshat-cli/src/tui/mod.rs` does NOT run the migration framework — it's a hand-written `CREATE TABLE` batch that pre-dates V11/V12. When a production SQL change references a new table, the fixture must be patched too. Symptom: query compiles fine, works against `Database::open(":memory:")` (which DOES run migrations), but fails with "no such table" only in legacy-fixture tests. Always grep for `CREATE TABLE nodes` (or whatever the fixture initialises) when adding a join to a new table.
- For `EXPLAIN QUERY PLAN` against a JOIN query, `query_row` only sees the FIRST plan line and misses the join step that's usually load-bearing. Use `query_map` to collect ALL rows, join them with `\n`, and assert against the joined string with `.contains()`.

**Learnings for future iterations:**
- `rusqlite::trace::trace_v2` takes `Option<fn(TraceEvent<'_>)>` — a fn POINTER, not a closure. State must live in a `thread_local!` because the callback can't capture. Pattern: `thread_local! { static CAPTURED_SQL: RefCell<Vec<String>> = const { RefCell::new(Vec::new()) }; }` plus a free function `fn trace_capture(event: TraceEvent<'_>)`. Always disable tracing again at the end of the test (`trace_v2(.., None::<fn(_)>)` works without explicit type because `None` infers from `Option<fn(TraceEvent<'_>)>`); cargo test reuses worker threads, so a stray callback would leak across tests.
- `rusqlite 0.37`'s `TraceEvent::Stmt` is a TUPLE variant `Stmt(StmtRef<'s>, &'s str)`, not `Stmt { sql, .. }` (different from older docs and trace_v2 in older C-bindings). Pattern-match accordingly.
- When wrapping `bulk_fetch_*` + a write transaction around the same `Arc<Mutex<Connection>>`, ALWAYS perform the bulk fetch BEFORE the `lock_conn` guard for the transaction. The repository methods call `lock_conn` themselves, so a fetch-while-holding-the-lock would deadlock. The pattern that works: `let result = repo.get_by_hashes(&hash_refs)?;` followed by `let guard = lock_conn(conn)?;` and the BEGIN/INSERT/COMMIT block uses `result` without re-querying.
- When AC says "the exception is removed", the obsolete "exception works" test is NOT salvageable by parameter tweaks — REMOVE it and add a regression test that pins the NEW invariant (e.g. "node with `user_rejected=1` IS now deleted"). Mixing old- and new-behaviour assertions in the same test guarantees future churn.
- For an AC like "1 SELECT, not N", a behaviour-only test (verify outputs) is insufficient; the regression guard must actually count SQL statements. Wiring `trace_v2` once is worth the upfront cost — N+1 regressions are otherwise indistinguishable from correct behaviour at the assertion level.
- For tests that need a stable SQL substring filter, prefer `to_uppercase()` then `contains("SELECT")` + `contains("FROM DECISIONS")` over regex. Cheap, no extra deps, and the existing chunked `get_by_hashes` SQL renders deterministically as `SELECT ... FROM decisions WHERE description_hash IN (?1, ?2, ...)`.
- `aggregated.iter().zip(hashes.iter())` (where `hashes` is the precomputed `Vec<String>`) lets the INSERT loop access both the convention struct and the matching hash without recomputing — and without rebuilding a `HashMap<String, &AggregatedConvention>`. Index-aligned vectors are fine here because the producer (the same function) controls both.

**Learnings for future iterations:**
- SQLite's `EXPLAIN QUERY PLAN` uses the FROM-clause **alias**, not the raw table name. If the FROM clause aliases `decisions` as `d`, the plan reads `SEARCH d USING ...` — `assert!(plan.contains("SEARCH decisions"))` will fail. Always assert against the alias the production SQL uses, OR include both forms in the OR clause if the production SQL might be reformatted.
- For LEFT JOIN "anti-join" patterns (`WHERE d.<col> IS NULL` to find rows with no match), pick a column that is BOTH the PK on the right side AND NOT NULL. `d.description_hash` works here — V12 declares it `TEXT NOT NULL PRIMARY KEY`. If you pick a nullable column you might get false positives (real matches that happen to have a NULL in that column). The PRD AC's `d.id IS NULL` was a leftover from an earlier draft where decisions had a synthetic `id` column; the natural translation to V12 is `d.description_hash IS NULL`.
- When a function uses a const SQL fragment like `SQL_NOT_REMOVED` (which references `ext_data` unqualified) inside a multi-table FROM clause, the unqualified reference is fine ONLY as long as the column name is unique across the joined tables. `ext_data` exists only on `nodes` (not on `decisions`), so `ext_data` resolves unambiguously even with the `nodes n LEFT JOIN decisions d` FROM clause. If a future migration adds `ext_data` to `decisions`, this becomes ambiguous and you'd need to qualify it as `n.ext_data` everywhere or split SQL_NOT_REMOVED into a `with_alias("n")` form.
- The TUI's `setup_empty_db` test fixture in `tui/mod.rs` does NOT run the migration framework — it's a hand-written `CREATE TABLE` batch that pre-dates V11/V12. So adding a column or table referenced by production SQL requires patching THIS fixture too. Symptom: a query that compiles fine and works against `Database::open(":memory:")` (which DOES run migrations) fails with "no such table" only in the legacy fixture's tests. Always grep for `CREATE TABLE nodes` (or whatever the fixture initialises) when adding a join to a new table.
- For unit tests that use `EXPLAIN QUERY PLAN`, `query_map` returns multiple rows (one per join step). Collect all rows with `.query_map().unwrap().map(|r| r.unwrap()).collect()` and join them with `\n` to inspect the full plan, then assert against the joined string with `.contains()` — a single-row `query_row` only sees the first plan line and misses the join step that's usually load-bearing.
- `LEFT JOIN d ON d.<pk> = n.<pk>` + `WHERE d.<pk> IS NULL` is the canonical "anti-join" and SQLite optimises it well when the right side has a PK auto-index. The alternative `WHERE n.<pk> NOT IN (SELECT <pk> FROM d)` typically requires a second scan or temp B-tree and is slower at scale. EXPLAIN QUERY PLAN confirms the JOIN form gets `SEARCH d USING COVERING INDEX sqlite_autoindex_decisions_1 (description_hash=?) LEFT-JOIN` — no temp B-tree on the join itself (the `USE TEMP B-TREE FOR ORDER BY` is for the outer `ORDER BY n.confidence DESC`, unavoidable without a confidence index).
- The LEFT JOIN's `ON d.description_hash = n.description_hash` matches NULL == NULL as *unknown*, NOT *true*. So an auto-detected node with `description_hash = NULL` (pre-V8 legacy row) gets NULL on the d side regardless of whether any decision row has NULL hash. The WHERE `d.description_hash IS NULL` keeps it. This is the desired behaviour: legacy NULL-hash nodes can't match any decision (decisions PK is NOT NULL), so they shouldn't be filtered out. The defensive `query_conventions_for_review_keeps_undecided_with_null_hash` test pins this — without it, a future "fix" that switches to `INNER JOIN ... d.description_hash IS NULL` (which is unsatisfiable) or `WHERE n.description_hash IS NOT NULL AND d.description_hash IS NULL` would silently drop legacy rows.
- When the PRD AC says `WHERE d.id IS NULL` but the actual schema has no `id` column, don't blindly add an `id` column to the schema — translate the AC to the closest equivalent in the actual schema. The intent is "no matching decision row", and `d.<PK> IS NULL` expresses that with one fewer column. Document the deviation in the commit message and PRD notes so future readers don't think the AC was ignored.
