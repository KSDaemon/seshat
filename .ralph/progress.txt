## Codebase Patterns
- Migrations live in `crates/seshat-storage/migrations/` and are auto-discovered by `refinery::embed_migrations!("migrations")` in `crates/seshat-storage/src/db.rs` — adding a new `V<n>__name.sql` file is enough; no registry to update.
- Repository pattern: each entity gets a trait in `crates/seshat-storage/src/repository/mod.rs` and a `SqliteXxxRepository` struct (with `Arc<Mutex<Connection>>`) in its own `xxx_repository.rs` sibling file. Re-export the trait + struct from `mod.rs` and from `crates/seshat-storage/src/lib.rs` for downstream consumers.
- `lock_conn(&self.conn)` in `mod.rs` is the shared helper for acquiring the connection lock — use it instead of inlining the lock/`map_err` boilerplate.
- For chunked `IN (?,?,?)` bulk lookups, build placeholders via `(1..=chunk.len()).map(|i| format!("?{i}")).collect::<Vec<_>>().join(", ")` and convert `&[&str]` to `Vec<&dyn rusqlite::types::ToSql>` per chunk. PRD-specified chunk size is 500 (well under SQLite's default `SQLITE_MAX_VARIABLE_NUMBER`).
- For SQL string-mapped enums (e.g. `DecisionState`/`DecisionNature`/`DecisionWeight`) implement `as_sql_str(&self) -> &'static str` and `from_sql_str(&str) -> Result<Self, StorageError>` instead of `Display`/`FromStr` — this keeps the SQL contract explicit and isolates DB serialisation from any user-facing formatting that might evolve later.
- `rusqlite::Connection::query_row` short-circuits on `QueryReturnedNoRows` — match it to convert into `Ok(None)` for nullable lookups; otherwise propagate `StorageError::from(e)`.
- The PRD source-of-truth lives in `.ralph/tasks/prd-merge-aware-decisions.md`. The `.ralph/prd.json` userStories are the operational view; the markdown contains the full schema, queries, and rationale.
- Pre-existing flaky test in `crates/seshat-cli/src/init/tests.rs` (`opencode_global_config_dir_empty_xdg_falls_back_to_home`) — env-var (`XDG_CONFIG_HOME`) test isolation issue independent of any change. Re-run on its own to confirm green.
- Branch lifecycle: post-US-003, branches must be EXPLICITLY registered in the `branches` table to show up in `list_branches`. Three entry points keep the registry in sync — `ensure_branch_exists` (idempotent), `set_last_scanned_commit` (UPSERT), `create_snapshot` (registers source via INSERT OR IGNORE + target with `snapshot_source`). `switch_branch` implicitly registers, and `scan_project_with_progress` calls `ensure_branch_exists` at the top of every scan. `delete_branch` removes the registry row LAST inside the cleanup transaction (after edges/nodes/files_ir) so a partial failure doesn't orphan data behind a missing sentinel.
- For SQLite UPSERTs the canonical pattern is `INSERT INTO t (...) VALUES (...) ON CONFLICT(<pk>) DO UPDATE SET col = excluded.col` — using `excluded` (not literal placeholders) lets you reuse the values without rebinding parameters.
- `unixepoch()` resolution is per-second; tests asserting that two consecutive timestamps differ need `std::thread::sleep(Duration::from_millis(1100))` between writes (≥1.1s, not 1.0s, so the second tick is safely past the boundary on slow CI hosts).
- `rusqlite::query_row` returning `Result<Option<String>, _>` via `row.get(0)` correctly unwraps NULL columns into `Ok(None)` — but missing rows still come back as `QueryReturnedNoRows`, which must be matched separately. So the canonical "nullable column for a possibly-missing row" handler has *two* `Ok` arms: `Ok(value)` (row found, value is `Option<T>`) and `Err(QueryReturnedNoRows) => Ok(None)`.
- For "exactly N statements were issued" tests, enable rusqlite's `trace` feature (workspace `Cargo.toml`) and use `Connection::trace_v2(TraceEventCodes::SQLITE_TRACE_STMT, Some(fn_pointer))` inside the test. The callback is a `fn` pointer (not a closure) so state must be parked in a `thread_local! { static FOO: RefCell<Vec<String>> }` that the function reads/writes. Always disable tracing again at the end of the test (`trace_v2(.., None)`) so a parallel test on the same OS thread doesn't inherit a stale callback.
- When a function wraps `bulk_fetch_*` plus a write transaction, do the bulk fetch BEFORE acquiring the write-side `lock_conn` guard. `SqliteDecisionRepository::get_by_hashes` calls `lock_conn` itself, so calling it while the persist guard is held would deadlock. Pre-compute outside, store in a `HashSet`, then take the lock and do the transaction.
- When a downstream story's AC says "rejections live in `decisions` now", the obsolete tests that pin the OLD behaviour (e.g. `persist_conventions_skips_user_rejected` asserting a `user_rejected=1` node survives) must be REMOVED, not made to pass. Replace them with regression tests that pin the NEW invariant (e.g. `persist_conventions_no_longer_preserves_user_rejected_nodes` asserting that node IS now deleted). Old behaviour ≠ new behaviour, even if the test name still parses.
- For functions that batch multiple writes inside an `apply_*` transaction envelope (BEGIN at the top, COMMIT at the bottom, helpers in between), the canonical pattern is: instantiate `SqliteXxxRepository::new(conn.clone())` INSIDE each helper rather than threading a long-lived repo through the call graph. The `Arc::clone` is cheap, transaction state lives on the `Connection` (not the `Mutex`), and dropping the per-helper lock guard between calls is safe — the open SQLite transaction persists. Helpers stay independently testable this way.
- When a refactor drops fields from helper-function signatures but leaves the `enum` variant shape intact for source compat, prefer the `..` rest pattern at the dispatch match arm (`Variant { used, .. } => helper(used)`) over underscore-prefixing the unused params on the helper. Rest patterns survive future field churn without churning helper signatures, and keep the helper's own argument list semantically clean.
- Tests asserting "old behaviour gone" (e.g. `count(user-source nodes) == 0`) and "new behaviour present" (e.g. `decisions.get_by_hash(...)` returns a row) are NOT redundant. The first pins the negative invariant ("nothing leaked into the legacy table"); the second pins the positive invariant ("durable state was written to the new table"). Keep both — they fail independently, and one passing while the other fails is a real bug class (e.g. dual-write code that forgot to remove the legacy write).
- SQLite's `EXPLAIN QUERY PLAN` uses the FROM-clause **alias**, not the raw table name. If you `LEFT JOIN decisions d`, the plan reads `SEARCH d USING ...` — `assert!(plan.contains("SEARCH decisions"))` fails. Always assert against the alias the production SQL uses, AND check for the underlying index name (`sqlite_autoindex_decisions_1` for V12's PK auto-index) to pin both halves of the contract.
- LEFT JOIN anti-join (`LEFT JOIN b ON b.fk = a.pk WHERE b.<col> IS NULL`) for "rows in A with no match in B": pick a column on the right side that is BOTH the PK AND `NOT NULL` for the WHERE predicate. With V12's `decisions.description_hash TEXT NOT NULL PRIMARY KEY`, `WHERE d.description_hash IS NULL` is the canonical "no match" predicate — unmatched LEFT JOIN rows have NULL on every d-side column, but a nullable column risks false positives.
- The TUI's `setup_empty_db` test fixture in `crates/seshat-cli/src/tui/mod.rs` does NOT run the migration framework — it's a hand-written `CREATE TABLE` batch that pre-dates V11/V12. When a production SQL change references a new table, the fixture must be patched too. Symptom: query compiles fine, works against `Database::open(":memory:")` (which DOES run migrations), but fails with "no such table" only in legacy-fixture tests. Always grep for `CREATE TABLE nodes` (or whatever the fixture initialises) when adding a join to a new table.
- For `EXPLAIN QUERY PLAN` against a JOIN query, `query_row` only sees the FIRST plan line and misses the join step that's usually load-bearing. Use `query_map` to collect ALL rows, join them with `\n`, and assert against the joined string with `.contains()`.
- When a story's AC names specific test files (e.g. "all assertions in `crates/seshat-detectors/tests/snippet_quality.rs`"), restrict the migration scope to those exact files. Do NOT expand to inline `#[cfg(test)] mod tests` blocks in source files — those tests are typically owned by the story that migrates the production function they cover. Strict AC-scoped migration keeps stories independently shippable; broad migration creates a chicken-and-egg with not-yet-migrated production code (e.g. inline tests in `seshat-graph/src/decisions.rs` that pin `ext["source"] == "user"` are the test contract for the legacy `record_decision` function — those migrate alongside US-004's production change, not under US-018's test-cleanup banner).
- Recognise verification-only stories: when a "migrate tests" story comes after several stories that each shipped tests with the production change inline, the migration story may be vacuously satisfied. Audit the named files for the AC patterns; if grep returns zero matches, the story is a checkpoint, not a code change. Document the audit findings in the PRD notes so future readers can verify the conclusion without re-running the analysis.
- When the storage-layer identity changes (e.g. rowid → description_hash), the AC's "no regression in envelope shape" should be read narrowly. It usually applies only to the named tools (here, `query_*`), not to mutation tools whose request schema MUST change to accept the new identifier. Don't over-constrain by trying to keep `id: i64` as the primary identifier across a hash-keyed schema — you'll either invent synthetic-rowid hacks or paint yourself into a corner. The breaking-DB-schema framing of the PRD usually licenses the breaking-MCP-schema change too.
- For shared response types (e.g. `ConventionResult`) that need a NEW identifier without breaking existing consumers: ADD the new field with `#[serde(skip_serializing_if = "String::is_empty")]` (for strings) or `#[serde(skip_serializing_if = "Option::is_none")]` (for options) instead of changing the type of the existing identifier. Old consumers keep working; new consumers see the new field; the wire shape is purely additive.
- For "merge results from FTS5 nodes table + LIKE-search of decisions table" with dedup-by-hash: build the dedup key set from the SMALLER vec (typically decisions, project-wide) and skip elements of the LARGER vec whose hash is in the set. The opposite order (set from larger, skip smaller) accidentally drops the wrong winner when both sources have a row.
- "FTS5-search of every relevant table" is tempting but overkill when one of the tables is small (decisions are user-driven, typically ≪1000 rows). A simple `LOWER(description) LIKE %kw%` AND-joined keyword search runs in microseconds against a few hundred rows and avoids FTS5 schema changes (e.g. `conventions_fts.node_id` is TEXT but always-parsed-as-i64 in `fts.rs` — adding decisions would have required a synthetic-id scheme to coexist). Pick the simple path unless the table is large enough to need FTS5 ranking.
- When migrating FROM a soft-delete contract TO a hard-delete contract, the obsolete `_marks_as_removed` test is NOT salvageable by tweaking assertions — REMOVE it and add a `_hard_deletes_row` test that pins `repo.get_by_hash(...).is_none()` AFTER the delete. Mixing old soft-delete and new hard-delete assertions guarantees future regression risk.
- `update_*` operations on hash-keyed tables: keep the PK invariant when underlying fields change (e.g. description). Document the invariant in the doc comment AND pin it with a regression test (`*_preserves_pk_when_*_changes`). The alternative (delete old + insert new under recomputed hash) silently re-keys rows from under callers holding the old hash. Force callers to remove+re-record explicitly when they want to re-key — make the lifecycle visible.
- Cross-crate "scan completion" hook: when a behaviour must fire at the end of multiple scan paths (`run_scan` in seshat-cli, `background_sync`/`fallback_rescan` in serve.rs, `execute_bulk_rescan` in seshat-watcher), put the helper in `seshat-scanner` (not seshat-cli — the watcher can't depend on cli). seshat-scanner already depends on seshat-storage and gix, so it's the right place for any helper that bridges "git state" and "branches table". The seshat-cli helpers in `db.rs` (`detect_branch`, `find_git_root`) are CLI-side only because they're used by serve startup and dangerous-cwd logic — those don't run in the watcher.
- HEAD-reading helper signature: prefer `fn get_head_commit(path: &Path) -> Option<String>` over `Result<String>` — every call site treats "no git" the same way ("skip silently"), so wrapping the failure in `Result` would force every caller to discriminate between "no .git" and "open failed" only to discard the distinction. `Option<String>` keeps the call sites honest and matches the existing `get_submodule_commit_hash` shape (which is now an alias).
- When wiring a "scan-completion sentinel" (e.g. `last_scanned_commit`) across multiple paths, the helper must be a panic-free, error-swallowing call: storage-write failures get `warn!`-logged but DO NOT propagate. A successful scan that just ran for 30 seconds should not regress because a sentinel write hit a transient SQLite contention. Git-unavailable is a separate, expected branch — `debug!`-logged and silent (column stays NULL per the AC). Two distinct log levels reflect two distinct events: warn = unexpected failure worth investigating; debug = expected condition (no git here).
- AC#N "X invocations triggered by Y do the same" is sometimes ambiguous (e.g. US-009's "run_detection_cycle_sync invocations triggered by warm-tier"). Audit the codebase: `grep` for the named function, list all call sites, and check whether each is "triggered by Y". If the audit shows the AC item is vacuously satisfied (callers were already covered by other AC items), document the finding in PRD notes rather than inventing a new wiring point that would lie about freshness state. The watcher's warm-tier timer task uses `run_detection_cycle` (not `_sync`) and only re-runs detection on existing IR — a "scan completion" sentinel there would mark the project as freshly scanned when no scan happened.
- `cargo fmt --all` may rewrite multi-line `use` statements after sorting added imports. Don't fight it — accept the formatter's choice and re-read the file before subsequent edits if a system-reminder says the file changed.
- When a workspace test (e.g. `serve_starts_and_shows_startup_info`) fails on your branch, ALWAYS verify whether it fails on clean `main` before assuming responsibility. `git stash && cargo test ... && git stash pop` (or run on `main` directly) is a 30-second sanity check that prevents wasted hours debugging unrelated flakes. Document confirmed pre-existing failures in PRD notes and progress.txt so future story owners don't have to re-run the same audit.
- Freshness-check return shape: prefer a 3-variant enum (`UpToDate` / `Stale { old, new }` / `GitUnavailable`) over `Option<String>` for "should we sync?" questions where the *reason* matters. The Stale variant carries the OLD commit (Option<String>: None on a never-scanned branch) so the caller can pass it as a tree-diff hint without re-querying; `GitUnavailable` short-circuits at the call site without conflating "no git" with "no diff" (both would be `None` in the Option-only shape, but they have different downstream semantics — UpToDate is "skip", GitUnavailable is "skip", Stale-with-None is "sync but no hint, fall through to fallback_rescan"). Match arms at the call site stay self-documenting.
- When two earlier-computed values become a duplicate of a later block's local computation (e.g. `sync_root` was inlined inside `if sync_old_branch.is_some()` in serve.rs), lift the value up so all downstream blocks share it. Don't synthesise a third copy for the new feature — that's the path to drift (one update misses one of the three sites). One owner, multiple readers. Verified: `gc_repo_path` and `project_root` do their own computations because they have different semantics (`project_root` uses cwd directly to keep worktree paths intact; `find_git_root` would walk up to the main repo).
- For a feature that needs to spawn a long-running background thread but only conditionally (e.g. `if needs_sync { thread::spawn(...) }`), keep the boolean trigger flat (`let needs_sync = a.is_some() || b.is_some();`) and resolve the parameter inside the same scope (`let old_hint = a.or_else(|| b);`). Don't nest two `if let Some(...)` blocks — the conditions are independent triggers, and flattening makes both paths visible at the same indent level. The thread-spawn machinery (sync_in_progress flag, ClearOnDrop guard, Database::open guard) is identical regardless of which trigger fired, so writing it once with the resolved hint as input is the least redundant shape.
- For long-running entry points like `run_serve` that can't be exercised end-to-end in tests (they wait on Ctrl+C / serve stdio forever), refactor the in-startup decision logic into a pure helper exposed via the public API. Test the helper directly. Document in the test that the helper's return value is what the entry point keys off (so a reader knows the test pins the AC at the right layer). Don't try to spawn run_serve in a test thread and assert side effects — that path is brittle (timeouts, races, stdio mocking) and leads to flaky tests.
- A "test the trigger condition" integration test is a legitimate stand-in for "test the triggered behaviour" when the trigger and the spawn are connected by a single boolean and the spawn machinery is tested independently elsewhere. Lock the *gate*, not the *spawn*. The PRD's "assert background_sync was triggered" is satisfied by asserting the gate returns Stale — that's necessary AND sufficient for the spawn (`let needs_sync = ... || head_change_hint.is_some(); if needs_sync { thread::spawn(...) }`). The spawn itself is exercised by the existing branch-switch tests and by US-017's broader serve tests.
- When the AC says "old commit hint (used for gix tree-diff if reachable; otherwise fallback_rescan)", recognise that the existing `resolve_branch_tree_paths` already handles BOTH branch refs AND raw commit hashes via `try_find_reference` → `gix::ObjectId::from_hex` fallback. So a freshness-check OLD commit hash can be passed through the same `Option<&str>` parameter that branch switches use — no new code path needed in `background_sync`. Just resolve the hint at the call site (`branch_name.or_else(|| commit_hash)`) and let the existing fallback chain handle the rest.
- Optional progress-callback parameter shape: prefer `Option<&dyn Fn(usize, usize)>` over a generic `<F: Fn>` parameter. The trait-object form lets the call site pass a stack-borrowed closure (e.g. one that fetches `AtomicUsize::fetch_add` on a local) without paying the `'static` tax that `Box<dyn Fn>` defaults to. Generics force every caller without a callback to provide a concrete type for the `None` arm — `Option::<fn(usize, usize)>::None` is uglier than the trait-object's natural `None`. The per-call dispatch overhead is irrelevant compared to the I/O-bound work the callback is reporting on.
- Progress callback emit cadence: fire AT THE TOP of each loop iteration (before any `continue` paths), then fire one final `(total, total)` tick AFTER the loop. This keeps the UI honest when iterations short-circuit (skipped files, oid-match early-returns, etc.) — without the top-of-loop emit, a long run of skipped files would freeze the progress display at the last "real" upsert. The final tick snaps the UI to "X / X" so the user sees completion.
- Progress UI throttle interval: `Duration::from_millis(950)` (not 1000) for "1 Hz" emits. Exact 1-second throttling flickers when subsequent calls land just on either side of the boundary; 950 ms is a stable cadence that's strictly faster-than-1-Hz on the worst case but indistinguishable from 1 Hz to the user. Same logic as the `unixepoch()` test sleeps (1100 ms not 1000 ms) at the other end of the spectrum.
- TTY-vs-piped progress branching: detect via `std::io::IsTerminal` on the stream you'll print to (stderr in our case, not stdout — stdout is reserved for MCP/JSON), then emit two flavours: `\rSyncing... Files: X / Y    ` for TTYs (same-line rewrite via carriage return + extra trailing spaces to clear longer previous lines) and one line per emit for piped output (`Syncing files: X / Y`). The two flavours can share the same throttle const but use different printer closures — keep the printer factories private (`tty_progress_printer`/`piped_progress_printer` returning `impl Fn`) and let the entry point pick.
- Testable-helper extraction for "entry point ends in interactive TUI / blocking server": when the function ends in a long-running interactive call (TUI launch, MCP serve loop), pull the upstream gate logic into a pub helper that returns a structured outcome enum (e.g. `ReviewSyncOutcome::Skipped | UpToDate | GitUnavailable | Synced { ... }`). Tests lock the helper, NOT the entry point. The entry point becomes a thin orchestrator: gate → helper → TUI. This is the same pattern US-010 used for `check_branch_freshness` against `run_serve`.
- For 1-extra-argument additions that push past clippy's 7-arg `too_many_arguments` cap: reach for `#[allow(clippy::too_many_arguments)]` on the helper. A Params struct is only worth it when arguments cluster semantically (e.g. all "filter" params or all "config" params) — for `incremental_sync_blocking` the 8 args are root + branches + db + branch_id + 2 configs + callback, none of which group naturally. Future evolution: if the arg list grows further or the helper sprouts a struct-variant call site, revisit.
- "Thin wrapper preserves call shape" pattern: when extracting a private helper into a richer pub(crate) helper that another module needs, keep the existing call shape via a 1-line wrapper. `background_sync` stays as-is for all existing call sites; the new `incremental_sync_blocking` adds a `progress: Option<&dyn Fn>` parameter and is called via `incremental_sync_blocking(..., None)` from the wrapper. Callers within the same module that DON'T need progress (e.g. branch-switch handler in `on_branch_switch`) keep using `background_sync` and don't churn. Total touched lines: 1 wrapper + N callers = N+1, vs the alternative "rewrite every caller to pass `None` explicitly" which is N + N for nothing.
- For non-git fixture tests, ALWAYS pre-flight `assert!(!path.join(".git").exists())` BEFORE the test does anything else. On macOS `tempdir()` lands in `/private/var/folders/.../T/` which is not inside any git repo, so the assertion costs nothing — but a future change that wrote sources into the workspace dir would silently inherit the project's `.git` and the AC assertion would still pass for the wrong reason. The pre-flight surfaces fixture problems as fixture problems, not as silent regressions in production fallback logic.
- Verification-only stories ARE legitimate completion shape when the AC items map cleanly to behaviours implemented across earlier stories. Deliverable: a TEST that pins the merged contract end-to-end PLUS PRD notes that map each AC item to its source-of-truth file:line. Without the per-AC mapping, a future reader has to re-run the audit; with it, they verify the conclusion in 30 seconds. Pattern: numbered list in PRD notes — one bullet per AC, naming the existing file:line range that satisfies it, then naming the new test that locks it.

**Learnings for future iterations:**
- `query_row` returns `rusqlite::Error::QueryReturnedNoRows` for an empty result — match it explicitly to translate "row absent" into `Ok(None)` instead of letting it bubble up as a hard error.
- `query_map` callbacks are `FnMut(&Row) -> rusqlite::Result<T>`. To return a typed `StorageError` for inner failures (enum-parse, JSON-decode), wrap the inner value in `rusqlite::Result<Result<Decision, StorageError>>` and unwrap the double `?` at the call site (`row??`). Trying to convert directly to `StorageError` inside the callback fails because `rusqlite::Error::FromSqlConversionFailure` requires `Box<dyn Error>` and erases the `StorageError` variant.
- The PRD reference signature `fn upsert(&self, d: Decision)` consumes by value, but the trait was already tightened to `&Decision` in US-001. US-008 will need bulk UPSERT-skip behaviour, so consuming would have forced clones at every call site — `&Decision` was the right call.
- For TEXT PRIMARY KEYs, SQLite's `EXPLAIN QUERY PLAN` reports `SEARCH decisions USING INDEX sqlite_autoindex_decisions_1 (description_hash=?)`. The `SEARCH ... USING INDEX` substring is the load-bearing part — `assert!(plan.contains("SEARCH") && !plan.contains("SCAN"))` is sufficient and stable across SQLite versions.
- Inline format args (`format!("?{i}")` instead of `format!("?{}", i)`) are the clippy-clean style in this repo — `cargo clippy --all-targets -- -D warnings` enforces it.
- `delete` idempotency: per the AC, calling `delete` on a non-existent hash must not error. Don't reach for `StorageError::NotFound` here — `conn.execute(...)?; Ok(())` is the right shape, with the deleted row count silently dropped.
- `params!` macro can't be used on heterogeneous `&[&dyn ToSql]` for variadic IN-clauses — collect into `Vec<&dyn rusqlite::types::ToSql>` and pass `.as_slice()` to `query_map`.

- `rusqlite::trace::trace_v2` takes `Option<fn(TraceEvent<'_>)>` — a fn POINTER, not a closure. State must live in a `thread_local!` because the callback can't capture. If the FROM clause aliases `decisions` as `d`, the plan reads `SEARCH d USING ...` — `assert!(plan.contains("SEARCH decisions"))` will fail. - For LEFT JOIN "anti-join" patterns (`WHERE d.<col> IS NULL` to find rows with no match), pick a column that is BOTH the PK on the right side AND NOT NULL. `d.description_hash` works here — V12 declares it `TEXT NOT NULL PRIMARY KEY`. If you pick a nullable column you might get false positives (real matches that happen to have a NULL in that column). - When a function uses a const SQL fragment like `SQL_NOT_REMOVED` (which references `ext_data` unqualified) inside a multi-table FROM clause, the unqualified reference is fine ONLY as long as the column name is unique across the joined tables. `ext_data` exists only on `nodes` (not on `decisions`), so `ext_data` resolves unambiguously even with the `nodes n LEFT JOIN decisions d` FROM clause. So adding a column or table referenced by production SQL requires patching THIS fixture too. Symptom: a query that compiles fine and works against `Database::open(":memory:")` (which DOES run migrations) fails with "no such table" only in the legacy fixture's tests. Always grep for `CREATE TABLE nodes` (or whatever the fixture initialises) when adding a join to a new table.
- For unit tests that use `EXPLAIN QUERY PLAN`, `query_map` returns multiple rows (one per join step). The alternative `WHERE n.<pk> NOT IN (SELECT <pk> FROM d)` typically requires a second scan or temp B-tree and is slower at scale. EXPLAIN QUERY PLAN confirms the JOIN form gets `SEARCH d USING COVERING INDEX sqlite_autoindex_decisions_1 (description_hash=?) LEFT-JOIN` — no temp B-tree on the join itself (the `USE TEMP B-TREE FOR ORDER BY` is for the outer `ORDER BY n.confidence DESC`, unavoidable without a confidence index).

So an auto-detected node with `description_hash = NULL` (pre-V8 legacy row) gets NULL on the d side regardless of whether any decision row has NULL hash. The WHERE `d.description_hash IS NULL` keeps it. This is the desired behaviour: legacy NULL-hash nodes can't match any decision (decisions PK is NOT NULL), so they shouldn't be filtered out. The defensive `query_conventions_for_review_keeps_undecided_with_null_hash` test pins this — without it, a future "fix" that switches to `INNER JOIN ... d.description_hash IS NULL` (which is unsatisfiable) or `WHERE n.description_hash IS NOT NULL AND d.description_hash IS NULL` would silently drop legacy rows.
- When the PRD AC says `WHERE d.id IS NULL` but the actual schema has no `id` column, don't blindly add an `id` column to the schema — translate the AC to the closest equivalent in the actual schema. The intent is "no matching decision row", and `d.<PK> IS NULL` expresses that with one fewer column. **Learnings for future iterations:**
- For "should we sync?" gate decisions where the *reason* affects what to pass downstream, prefer a 3-variant enum (`UpToDate` / `Stale { old, new }` / `GitUnavailable`) over `Option<String>`. The Stale variant carries the OLD commit (Option<String>: None on a never-scanned branch) so the caller can pass it as a tree-diff hint without re-querying. The GitUnavailable variant short-circuits at the call site without conflating "no git here" with "no diff to do" — both would collapse to None in the Option-only shape, but they have distinct downstream semantics. Match arms at the call site stay self-documenting.
- Branch-switch-and-HEAD-change union is one boolean (`needs_sync = a.is_some() || b.is_some()`) plus one resolved hint (`old_hint = a.or_else(|| b)`). Don't nest two `if let Some(...)` blocks for the spawn — the conditions are independent triggers and flattening makes both paths visible at the same indent level. The thread-spawn machinery (sync_in_progress flag, ClearOnDrop guard, Database::open guard) is identical regardless of which trigger fired, so writing it once with the resolved hint as input is the least redundant shape.
- When two earlier-computed values become a duplicate of a later block's local computation (e.g. `sync_root` was inlined inside `if sync_old_branch.is_some()`), lift the value up so all downstream blocks share it. Don't synthesise a third copy for the new feature — that's the path to drift. Verify the lift doesn't accidentally fold in nearby variables that LOOK identical but have semantic differences (`gc_repo_path` could be folded; `project_root` cannot — `project_root` uses cwd directly to keep worktree paths intact, where `find_git_root` would walk up to the main repo).
- For long-running entry points like `run_serve` that can't be exercised end-to-end in tests (they wait on Ctrl+C or serve stdio forever), refactor the in-startup decision logic into a pure helper exposed via the public API of the right crate (here, `seshat-scanner` because it owns the gix bridge), and test the helper directly. Document in the integration test that the helper's return value is what the entry point keys off — so a reader knows the test pins the AC at the right layer. Don't try to spawn `run_serve` in a test thread and assert side effects — that path is brittle (timeouts, races, stdio mocking) and leads to flaky tests.
- "Test the trigger condition" is a legitimate stand-in for "test the triggered behaviour" when the trigger and the spawn are connected by a single boolean and the spawn machinery is tested independently elsewhere. The PRD's "assert background_sync was triggered" is satisfied by asserting the gate returns Stale — that's necessary AND sufficient for the spawn (`let needs_sync = ... || head_change_hint.is_some(); if needs_sync { thread::spawn(...) }`). The spawn itself is exercised by the existing branch-switch tests and by US-017's broader serve tests.
- When the AC says "old commit hint (used for gix tree-diff if reachable; otherwise fallback_rescan)", recognise that `resolve_branch_tree_paths` already handles BOTH branch refs AND raw commit hashes via `try_find_reference` → `gix::ObjectId::from_hex` fallback. So a freshness-check OLD commit hash flows through the same `Option<&str>` parameter that branch switches use — no new code path is needed in `background_sync`. Just resolve the hint at the call site and let the existing fallback chain handle the rest.
- Reading HEAD is cheap, but checking it on every `run_serve` startup is one extra gix open + one extra SQLite SELECT. The total cost (~ms) is negligible relative to the savings (a single stale-cache MCP response that the user has to debug after `git pull`). Don't over-engineer the caching — measure first if it ever becomes a hot path.
- Tracing fields `old_head` and `new_head` truncated via `.chars().take(7).collect::<String>()` (NOT `&hash[..7]`) — the latter panics on non-ASCII inputs. Hash strings are ASCII hex, but the `.chars()` form is robust to future schema changes (e.g. branch IDs with unicode) and is already the canonical `seshat` style for "first N chars" displays.
- Negative-case test for git-unavailable + present-but-stale sentinel: even when a sentinel is recorded for a branch, a fresh `check_branch_freshness` against a directory without `.git` MUST return `GitUnavailable` (not Stale). Otherwise a user who removed `.git` would trigger an unnecessary sync against a sentinel that's no longer meaningful. The unit test `check_branch_freshness_returns_git_unavailable_for_non_git_directory` pins this — without it, a future "optimisation" that read the sentinel before reading HEAD might short-circuit incorrectly.

**Learnings for future iterations:**
- `Box<dyn Fn(usize, usize)>` defaults to `Box<dyn Fn(usize, usize) + 'static>`, so a Box capturing a stack-borrowed reference (e.g. `&AtomicUsize` from the enclosing function) won't compile. Two ways out: (a) use a stack closure directly (`let cb = |...| { ... };` then `Some(&cb)`) — no Box needed, `Option<&dyn Fn>` accepts any lifetime via elision; or (b) annotate the lifetime explicitly (`Box<dyn Fn(...) + 'a>`). The stack-closure form is simpler and was the right call here.
- `prepare_review_sync` captures the `progress_callback` parameter from the outer fn into the internal counted closure. Closures borrow by default, and the `progress_callback: Option<&dyn Fn(...)>` parameter has a lifetime tied to the fn — moving the closure into `Some(&counted_cb)` works because `counted_cb` lives on the same stack frame. No `move` keyword needed.
- For `pub fn` helpers that accept an optional callback, prefer `Option<&dyn Fn(...)>` over `<F: Fn>` generics. The trait-object form lets call sites pass stack-borrowed closures without `'static` constraints, doesn't force callers without a callback to invent a concrete `fn` type for the `None` arm, and the dispatch overhead is irrelevant against I/O-bound work the callback is reporting on.
- When extracting a private helper into a richer pub(crate) helper with a new optional parameter, keep the existing call shape via a 1-line wrapper for in-module callers. `background_sync` stays as-is for everyone in `serve.rs`; the new `incremental_sync_blocking` adds the progress parameter and is called via `incremental_sync_blocking(..., None)` from the wrapper. Callers that don't need progress don't churn.
- TTY detection for progress UI: `std::io::stderr().is_terminal()` (not stdout — stdout is reserved for MCP/JSON output across the workspace, and `seshat review` is no exception). The two flavours (TTY same-line `\r` rewrite + final `done.` line; piped one-line-per-emit) share the same throttle constant but diverge in their printer closures. Pad the rewrite line with trailing spaces (`Files: X / Y    `) to clear longer previous lines — without padding, "Files: 100 / 200" leaves a stray "0" when the next emit prints "Files: 5 / 200".
- Throttle interval for "1 Hz" progress: `Duration::from_millis(950)`, NOT 1000. Exact 1-second throttling flickers when consecutive calls land on either side of the boundary. Same shape as the `unixepoch()` test sleeps (1100 ms not 1000 ms) at the other end of the spectrum — boundary-buffer matters.
- Progress callback emit point: at the TOP of the loop iteration (BEFORE any `continue` paths), then a final `(total, total)` tick AFTER the loop. The top-of-loop emit keeps the UI advancing through skipped-files runs (oid-match early-returns, extension-mismatch continues, exclude-glob skips). Without it, a stretch of 50 skipped files freezes the progress display at the last "real" upsert and the user thinks the sync hung. The final tick snaps the UI to "X / X" so the user sees completion.
- Adding ONE more parameter that pushes a function past clippy's 7-arg `too_many_arguments` cap: reach for `#[allow(clippy::too_many_arguments)]` on the function. A Params struct is only worth the ceremony when arguments cluster semantically (e.g. all "filter" options, all "config" knobs). For `incremental_sync_blocking` the 8 args are root + branches + db + branch_id + 2 configs + callback — none of which group naturally. If the arg list grows further or a struct-variant call site emerges, revisit.
- For test-driven proof that "sync ran AND TUI received fresh data": instead of stubbing the TUI (impractical without a fake terminal), assert against the SAME `Arc<Mutex<Connection>>` handle the sync wrote into. The TUI reads from that handle when launched in production, so a post-sync `files_ir` query that finds the new file IS the test for "fresh data". Keep the assertion at the storage layer — it's stable across TUI evolutions.
- "ADD a new file" is a cleaner integration-test signal than "MODIFY an existing file" for diff sync. Modifying changes the tree-oid (good signal) but the assertion has to find a specific blob hash. Adding a new path is unambiguously a non-skip in the diff (new key in `new_paths`, absent from `old_paths`) and the assertion is a simple `files.iter().any(|(p, _)| p.ends_with("util.rs"))`. Pick the simpler signal when both work.
- Mirror `FreshnessCheck`'s variants in `ReviewSyncOutcome`. Don't collapse them into a single `Result<bool, _>` — the four cases (Skipped vs UpToDate vs GitUnavailable vs Synced) have distinct UX implications: Skipped is "user explicitly opted out"; UpToDate is "we'd have synced if the gate hadn't said pass"; GitUnavailable is "we can't sync but that's expected"; Synced is "we just did real work". Tests that assert `outcome == UpToDate` would silently lose coverage if the production code regressed UpToDate to Skipped.
- For testable-helper extraction from interactive-TUI entry points: the helper's return value should encode every decision the entry point makes downstream. `prepare_review_sync` returns `Synced { progress_emits, ... }` not just `Synced` because the AC requires "progress callback emits at least one update" — putting that count in the outcome lets the integration test assert it without a side-channel atomic. The outcome type is the integration-test contract; design it for the assertions you need, not just for the production code path.

- The integration test helper that mirrors `seshat scan`'s pipeline must call `record_branch_scan_complete` AFTER persist — not before, not in scan_project itself. The seshat-scanner orchestrator only calls `ensure_branch_exists` (US-003); the post-scan freshness sentinel write is wired in CLI's `run_scan` and the serve/watcher sync paths (US-009). A test helper that omits this call would PASS the scan + decide + rescan flow but FAIL to exercise AC#3, leaving a silent gap. Lift the call into the test helper alongside the scan/detect/persist sequence so every test running through the helper exercises all AC layers.
- Clippy 1.94+ (`doc-overindented-list-items`) flags 5+ space continuation indents on doc-list items. Use 3-space continuation for numbered lists (`//! 1. line one\n//!    line two continuation`) — the convention `cargo fmt` doesn't enforce but clippy does. Easy to miss because nothing else in the workspace's doc-comment style hits the threshold; trips on the first multi-line numbered list.
