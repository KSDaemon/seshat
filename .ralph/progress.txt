## Codebase Patterns
- Migrations live in `crates/seshat-storage/migrations/V<N>__<name>.sql`,
  embedded via `refinery::embed_migrations!`. Refinery's SQL files cannot
  deserialize postcard blobs — for backfill that needs Rust, perform it in
  `Database::open` immediately after `migrations::runner().run(...)`, gated
  on a "table empty?" check so it stays idempotent across reopens.
- IR snippets must be built the same way wherever they are produced. Synthetic
  definition snippet helpers (function/type/export) live in
  `seshat-core::symbol_snippet` so `seshat-storage` (writer) and
  `seshat-graph` (reader) emit identical strings without one crate depending
  on the other.
- Repository traits live in `crates/seshat-storage/src/repository/mod.rs`;
  Sqlite impls live in sibling files. Re-export everything via
  `crates/seshat-storage/src/lib.rs` so downstream crates only import from the
  crate root.
- Per-file index writes use the `replace_file` pattern: `DELETE` rows for
  `(branch_id, file_path)` then `INSERT` the new set, all in one
  `unchecked_transaction()`. Keeps re-scans / hot-tier updates idempotent.
- Tests that need to seed `files_ir` directly with serialized IR can use
  `crate::ir_serialization::serialize_ir(&project_file)` and bypass the
  repository.
- Cross-table writes that must commit atomically (e.g., `files_ir` + symbol
  index) live as a single method on the owning repository
  (`FileIRRepository::upsert_with_symbol_index`) that opens its own
  `unchecked_transaction()` and runs every statement inside it.  Sibling
  modules expose `pub(super)` SQL helpers operating on a `&Transaction` so
  the writer doesn't duplicate the per-table DELETE/INSERT SQL.
- Parser convention: `Import.names` stores defining (rightmost) names, never
  local aliases.  TS/JS already do this via the first identifier of
  `import_specifier`; Rust and Python parsers normalise top-level
  `use foo::Bar as Baz` and `import x as y` / `from m import x as y` to keep
  the IR consistent and downstream code (symbol-index, detectors) free of
  alias-stripping logic.
- Branch lifecycle ownership: every per-branch table must be touched by
  `BranchRepository::create_snapshot` (copy) and `delete_branch` (drop).
  After adding a new per-branch table, audit both methods — missing one
  produces silent state drift (e.g. symbol-index that doesn't survive
  branch switches, or orphan rows that accumulate across recreate cycles).

- **Learnings for future iterations:**
  - `BranchRepository::create_snapshot` and `delete_branch` are the
    branch-lifecycle bottleneck.  When the schema grows a new per-branch
    table (V13 added two), both methods must be updated together.  The
    sync path (`incremental_sync_blocking`) handles only changed/added/
    removed files; for *unchanged* files in a freshly snapshotted branch
    the new branch inherits its state from the snapshot copy.  If
    `create_snapshot` doesn't copy a per-branch table, that table will
    be empty for unchanged files until the next full re-scan.
  - `process_file_delete` propagates `NotFound` from `delete_by_path`
    (existing behaviour, untouched here).  In practice the watcher can
    receive `Remove` events for files we never indexed (CSVs, hidden
    files), so warnings will fire — left as-is to keep this change
    minimally scoped to US-003.  If we want to silence them, the orchestrator
    already shows the pattern (`let _ = file_ir_repo.delete_by_path(...)`).
  - For US-004 (return `dependent_files` in `query_code_pattern`): the
    `symbol_imports` table is now reliably populated and kept in sync across
    full scans, hot-tier events, branch switches, and HEAD-change syncs.
    The blast-radius / dependent_files SQL described in US-004's AC ("SELECT
    DISTINCT importer_file FROM symbol_imports WHERE branch_id=? AND
    imported_name=?") will have an authoritative source against any branch
