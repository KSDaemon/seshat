## Codebase Patterns
- Graph functions take `conn: &Arc<Mutex<Connection>>` and `branch_id: &str` as first two params, return `Result<T, GraphError>`
- Use `crate::lock_conn(conn)?` to acquire database connection lock
- Response structs derive `Debug, Clone, Serialize` (not Deserialize)
- IR data is stored as postcard-serialized blobs in `files_ir.ir_data` with 1-byte version prefix
- Use `seshat_storage::deserialize_ir()` to deserialize IR blobs; handle `StaleIR` errors gracefully
- New modules go in `crates/seshat-graph/src/` and get exported from `lib.rs` with `pub mod` + `pub use`
- Convention searching uses `fts::search_conventions()` which returns `Vec<NodeId>`; then load full node data from `nodes` table
- Error mapping pattern: `.map_err(|e| GraphError::Storage(seshat_storage::StorageError::QueryError(format!(...))))`
- Test helpers: `test_conn()` from `crate::test_helpers` returns in-memory DB with all migrations applied
- Pre-commit hook runs `cargo fmt` which may reorder imports; commit after fmt fixes with `git add -A && git commit`
- MCP tool handlers are thin wrappers: validate input → call `seshat_graph` → wrap in `ResponseEnvelope`
- Every MCP request struct MUST have `repo: Option<String>`, `scope: Option<String>`, `file_path: Option<String>` for routing
- Use `impl_tool_request!` macro in `server.rs` for each new request type
- Tool registration pattern: `#[tool(description = "...")]` method calling `self.execute_tool(TOOL, req, |pc, scope_name, req| handler::handle(...))`
- Call logger summaries: add result extractor fn in `call_logger.rs` + match arm in `server.rs::log_tool_call`
- `with_extra()` on `ResponseMetadata` needs `&str` or values that impl `Into<serde_json::Value>` — use `.as_str()` for `&String`
- Use `map_graph_error()` from `envelope.rs` for graph-layer errors — it maps `NodeNotFound`, `NotUserDecision`, `InvalidInput` to proper error codes
- `CodeSnippet` from `seshat_core` has `MAX_SNIPPET_LINES = 20`; code_pattern uses local `MAX_PATTERN_SNIPPET_LINES = 10`
- IR types: `ProjectFile` contains `functions: Vec<Function>`, `types: Vec<TypeDef>`, `exports: Vec<Export>` — all from `seshat_core::ir`
- `load_branch_ir()` in `code_pattern.rs` is `pub(crate)` — reuse it in other graph modules (dependencies, validate_approach) instead of duplicating
- `Import` struct: `module: String, names: Vec<String>, is_type_only: bool, line: usize` — module is the raw import path
- `DependencyUsage` struct: `package: String, import_path: String, line: usize` — external deps from manifest
- For relative import resolution, normalize paths by resolving `.` and `..` components; try common extensions (.ts, .tsx, .js, .rs, .py)
- `GraphError::NodeNotFound` exists for file-not-found cases (maps to NODE_NOT_FOUND at MCP layer)
- FTS5 `search_conventions()` uses AND semantics — all tokens must match; for broader matching, use LIKE-based keyword search with OR
- `query_code_pattern` can be reused internally by other graph modules for duplicate detection — filter by score threshold
- Contradictions are stored in `edges` table with `edge_type = 'contradicts'` and can be queried by source_id/target_id
- Decisions (nature="decision") and observations (nature="observation") are stored in `nodes` table and searchable by keyword LIKE match
- Clippy flags `"value" | _` as wildcard-in-or-patterns — use just `_` for the catch-all match arm
- `ValidateApproachParams` struct lives in `seshat_graph` — use `seshat_graph::ValidateApproachParams` in MCP handlers
- `ureq` v3 API: use `Agent::config_builder()...build().into()` not `Agent::new()`; use `send_json()` (requires `json` feature); use `body_mut().read_json::<T>()` for JSON responses
- `EmbeddingConfig` lives in `seshat_embedding` crate — `seshat-cli` imports it via `use seshat_embedding::EmbeddingConfig`
- `EmbeddingProvider` trait requires `Send + Sync + Debug` for trait object compatibility and `unwrap_err()` in tests
- Provider factory pattern: `create_provider(&EmbeddingConfig) -> Result<Box<dyn EmbeddingProvider>, EmbeddingError>` with unknown provider → `ConfigError`
- Migrations use refinery `embed_migrations!("migrations")` — new SQL files in `crates/seshat-storage/migrations/` with `V{N}__{description}.sql` naming
- Repository pattern: trait in `repository/mod.rs` + SQLite impl in `repository/{name}_repository.rs`; use `lock_conn()` helper
- Embeddings stored as raw f32 bytes (little-endian) in BLOB column — use `f32s_to_bytes()` / `bytes_to_f32s()` from `seshat_storage`
- `EmbeddingRepository` trait + `SqliteEmbeddingRepository` in `seshat-storage` for code embedding persistence
- Embedding generation hooks into scan pipeline via `generate_embeddings()` in `seshat-cli/src/scan.rs` — called after `rebuild_fts_index()`
- Embedding is optional: when `config.embedding` is `None`, zero overhead; provider errors are logged and scan continues
- `unchecked_transaction()` on `rusqlite::Connection` available through `MutexGuard` — use for batched inserts in a single transaction
- `query_code_pattern_with_embeddings()` takes `Option<&dyn EmbeddingProvider>` — the backward-compatible `query_code_pattern()` delegates with `None`
- `cosine_similarity(a, b)` is a pure Rust f32 dot-product function in `code_pattern.rs` — no SQLite extension needed
- Vector search loads embeddings directly from `code_embeddings` table via SQL, not via `EmbeddingRepository` trait (to avoid circular deps)
- Merge strategy: HashMap keyed by `(file_path, name, kind)`, max score wins when same item in both keyword + vector results
- Clippy `type_complexity` fires on `HashMap<(String, String, String), (usize, usize, bool, CodeSnippet)>` — define type aliases `IrLookupKey` and `IrSnippetData`
- When adding `seshat-embedding` as a dep to `seshat-graph`, also add `.workspace = true` in Cargo.toml (workspace dep already declared in root)
