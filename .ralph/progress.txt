## Codebase Patterns
- `ManifestAnalysis` lives in `crates/seshat-scanner/src/manifest.rs` — the scanner crate
- `analyze_manifests()` is the central entry point for manifest processing in the orchestrator
- All `tracing::warn!` calls use structured fields: `path = %path.display(), error = %e`
- Crate/package names always normalise hyphens to underscores (`my-crate` → `my_crate`)
- `ManifestType` variants: `CargoToml`, `PackageJson`, `PyprojectToml`
- `seshat-cli/src/report/mod.rs` has test helpers constructing `ManifestAnalysis` manually — add new fields there too
- `internal_names` extraction functions follow the same pattern: private `fn extract_*(path, content) -> Vec<String>`, deserialise minimal struct, warn+return empty on parse error, normalise hyphens to underscores
- PEP 621 (`[project] name`) takes precedence over Poetry (`[tool.poetry] name`) for name extraction — early return when PEP 621 found
- `load_internal_names(conn, branch_id)` is a public fn in `seshat-graph/src/dependencies.rs` — reads `workspace_crates` JSON from `repo_metadata` via `SqliteRepoMetadataRepository`, returns empty Vec on missing key or invalid JSON
- `SqliteRepoMetadataRepository` and `RepoMetadataRepository` are re-exported from `seshat_storage` at top level
- `is_internal_crate(module, names)` replaces the removed `is_workspace_crate()` — checks `first_module_segment` against the dynamic `&[String]` slice
- `resolve_internal_crate_import` replaces `resolve_workspace_crate_import` — same logic, renamed for generality
