## Codebase Patterns
- `seshat-core` is the shared types crate with zero dependencies on other seshat crates — all shared enums/structs go here
- Re-exports in `lib.rs` flatten commonly used types for downstream ergonomics (e.g., `pub use dependency::DependencyDomain`)
- `seshat-scanner/src/manifest.rs` categorize functions use lowercased, hyphen-to-underscore normalized package names — match arms must use underscores not hyphens
- `seshat-detectors/src/dependency_usage.rs` classify functions use original package names (with hyphens) — match arms use the original crate name form
- `cargo fmt --all` must be run before committing — the project uses the default rustfmt settings
- Tests construct `ProjectFile` structs manually with explicit `language_ir` variants matching the language
- The `DependencyDomain` enum in `seshat-core/src/dependency.rs` is the single source of truth for dependency classification (12 variants), used by both scanner and detectors
- Storage repository pattern: trait in `repository/mod.rs`, SQLite impl in `repository/{entity}_repository.rs`, struct has `conn: Arc<Mutex<Connection>>` + `conn()` helper, free `row_to_{entity}()` function for row mapping
- Migrations use Refinery format: `V{N}__{description}.sql` in `crates/seshat-storage/migrations/`, embedded at compile time via `embed_migrations!("migrations")`
- JSON arrays stored as TEXT in SQLite, serialized/deserialized with serde_json in repository layer
- The `PackageRegistryClient` trait and types live in `seshat-scanner/src/registry/mod.rs`, concrete clients in `registry/{crates_io,npm,pypi}.rs`, storage repo lives in `seshat-storage/src/repository/package_metadata_repository.rs`
- Workspace deps defined in root `Cargo.toml` under `[workspace.dependencies]`, crate Cargo.toml uses `.workspace = true`
- The registry_mapping module in `seshat-scanner/src/registry/registry_mapping.rs` provides the three-tier dependency classification pipeline: (1) SQLite cache, (2) registry API fetch + cache, (3) hardcoded fallback. Use `classify_with_registry()` for full pipeline or individual mapping functions for targeted lookups.
- `Trend` enum lives in `seshat-core/src/knowledge.rs` (Rising/Stable/Declining/Unknown). `compute_trend()` in `seshat-detectors/src/confidence.rs` takes `&[Option<i64>]` file dates + config + now timestamp. P90 = sorted[ceil(N*0.9)-1]. Use `aggregate_findings(findings, config, file_dates, now)` with the `HashMap<String, Option<i64>>` parameter for trends.
- `aggregate_findings()` requires 4 parameters: findings, config, file_dates map, and now timestamp. Pass empty HashMap + 0 for now if trends are not needed.
- `AggregatedConvention::ext_data(&self, existing: Option<&Value>)` builds JSON with `{"trend": "...", "adoption_rate": ...}` merged with existing ext_data for KnowledgeNode construction.
- `ConventionDetector` trait has a default `detect_cross_file(&self, files: &[ProjectFile]) -> Vec<ConventionFinding>` method for cross-file analysis. The pipeline calls it once per detector after all per-file `detect()` calls. Only override if cross-file analysis is needed.
- Import resolution heuristics: Rust (`crate::`/`super::`/`self::` = internal), TS/JS (`./`/`../` = internal), Python (relative `.` or module path matches project file = internal). Use `is_internal_import()` and `import_resolves_to()` in `dependency_usage.rs`.
- `file_module_path()` converts a ProjectFile path to a canonical module path: Rust → `crate::mod::path`, TS/JS → `src/dir/file` (strip ext + /index), Python → `pkg.module` (strip .py + /__init__).
