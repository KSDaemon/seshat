## Codebase Patterns
- Migrations live in `crates/seshat-storage/migrations/` and are auto-discovered by `refinery::embed_migrations!("migrations")` in `crates/seshat-storage/src/db.rs` ‚ÄĒ adding a new `V<n>__name.sql` file is enough; no registry to update.
- Repository pattern: each entity gets a trait in `crates/seshat-storage/src/repository/mod.rs` and a `SqliteXxxRepository` struct (with `Arc<Mutex<Connection>>`) in its own `xxx_repository.rs` sibling file. Re-export the trait + struct from `mod.rs` and from `crates/seshat-storage/src/lib.rs` for downstream consumers.
- `lock_conn(&self.conn)` in `mod.rs` is the shared helper for acquiring the connection lock ‚ÄĒ use it instead of inlining the lock/`map_err` boilerplate.
- For chunked `IN (?,?,?)` bulk lookups, build placeholders via `(1..=chunk.len()).map(|i| format!("?{i}")).collect::<Vec<_>>().join(", ")` and convert `&[&str]` to `Vec<&dyn rusqlite::types::ToSql>` per chunk. PRD-specified chunk size is 500 (well under SQLite's default `SQLITE_MAX_VARIABLE_NUMBER`).
- For SQL string-mapped enums (e.g. `DecisionState`/`DecisionNature`/`DecisionWeight`) implement `as_sql_str(&self) -> &'static str` and `from_sql_str(&str) -> Result<Self, StorageError>` instead of `Display`/`FromStr` ‚ÄĒ this keeps the SQL contract explicit and isolates DB serialisation from any user-facing formatting that might evolve later.
- `rusqlite::Connection::query_row` short-circuits on `QueryReturnedNoRows` ‚ÄĒ match it to convert into `Ok(None)` for nullable lookups; otherwise propagate `StorageError::from(e)`.
- The PRD source-of-truth lives in `.ralph/tasks/prd-merge-aware-decisions.md`. The `.ralph/prd.json` userStories are the operational view; the markdown contains the full schema, queries, and rationale.
- Pre-existing flaky test in `crates/seshat-cli/src/init/tests.rs` (`opencode_global_config_dir_empty_xdg_falls_back_to_home`) ‚ÄĒ env-var (`XDG_CONFIG_HOME`) test isolation issue independent of any change. Re-run on its own to confirm green.
- Branch lifecycle: post-US-003, branches must be EXPLICITLY registered in the `branches` table to show up in `list_branches`. Three entry points keep the registry in sync ‚ÄĒ `ensure_branch_exists` (idempotent), `set_last_scanned_commit` (UPSERT), `create_snapshot` (registers source via INSERT OR IGNORE + target with `snapshot_source`). `switch_branch` implicitly registers, and `scan_project_with_progress` calls `ensure_branch_exists` at the top of every scan. `delete_branch` removes the registry row LAST inside the cleanup transaction (after edges/nodes/files_ir) so a partial failure doesn't orphan data behind a missing sentinel.
- For SQLite UPSERTs the canonical pattern is `INSERT INTO t (...) VALUES (...) ON CONFLICT(<pk>) DO UPDATE SET col = excluded.col` ‚ÄĒ using `excluded` (not literal placeholders) lets you reuse the values without rebinding parameters.
- `unixepoch()` resolution is per-second; tests asserting that two consecutive timestamps differ need `std::thread::sleep(Duration::from_millis(1100))` between writes (‚Č•1.1s, not 1.0s, so the second tick is safely past the boundary on slow CI hosts).
- `rusqlite::query_row` returning `Result<Option<String>, _>` via `row.get(0)` correctly unwraps NULL columns into `Ok(None)` ‚ÄĒ but missing rows still come back as `QueryReturnedNoRows`, which must be matched separately. So the canonical "nullable column for a possibly-missing row" handler has *two* `Ok` arms: `Ok(value)` (row found, value is `Option<T>`) and `Err(QueryReturnedNoRows) => Ok(None)`.
- For "exactly N statements were issued" tests, enable rusqlite's `trace` feature (workspace `Cargo.toml`) and use `Connection::trace_v2(TraceEventCodes::SQLITE_TRACE_STMT, Some(fn_pointer))` inside the test. The callback is a `fn` pointer (not a closure) so state must be parked in a `thread_local! { static FOO: RefCell<Vec<String>> }` that the function reads/writes. Always disable tracing again at the end of the test (`trace_v2(.., None)`) so a parallel test on the same OS thread doesn't inherit a stale callback.
- When a function wraps `bulk_fetch_*` plus a write transaction, do the bulk fetch BEFORE acquiring the write-side `lock_conn` guard. `SqliteDecisionRepository::get_by_hashes` calls `lock_conn` itself, so calling it while the persist guard is held would deadlock. Pre-compute outside, store in a `HashSet`, then take the lock and do the transaction.
- When a downstream story's AC says "rejections live in `decisions` now", the obsolete tests that pin the OLD behaviour (e.g. `persist_conventions_skips_user_rejected` asserting a `user_rejected=1` node survives) must be REMOVED, not made to pass. Replace them with regression tests that pin the NEW invariant (e.g. `persist_conventions_no_longer_preserves_user_rejected_nodes` asserting that node IS now deleted). Old behaviour ‚Č† new behaviour, even if the test name still parses.
- For functions that batch multiple writes inside an `apply_*` transaction envelope (BEGIN at the top, COMMIT at the bottom, helpers in between), the canonical pattern is: instantiate `SqliteXxxRepository::new(conn.clone())` INSIDE each helper rather than threading a long-lived repo through the call graph. The `Arc::clone` is cheap, transaction state lives on the `Connection` (not the `Mutex`), and dropping the per-helper lock guard between calls is safe ‚ÄĒ the open SQLite transaction persists. Helpers stay independently testable this way.
- When a refactor drops fields from helper-function signatures but leaves the `enum` variant shape intact for source compat, prefer the `..` rest pattern at the dispatch match arm (`Variant { used, .. } => helper(used)`) over underscore-prefixing the unused params on the helper. Rest patterns survive future field churn without churning helper signatures, and keep the helper's own argument list semantically clean.
- Tests asserting "old behaviour gone" (e.g. `count(user-source nodes) == 0`) and "new behaviour present" (e.g. `decisions.get_by_hash(...)` returns a row) are NOT redundant. The first pins the negative invariant ("nothing leaked into the legacy table"); the second pins the positive invariant ("durable state was written to the new table"). Keep both ‚ÄĒ they fail independently, and one passing while the other fails is a real bug class (e.g. dual-write code that forgot to remove the legacy write).
- SQLite's `EXPLAIN QUERY PLAN` uses the FROM-clause **alias**, not the raw table name. If you `LEFT JOIN decisions d`, the plan reads `SEARCH d USING ...` ‚ÄĒ `assert!(plan.contains("SEARCH decisions"))` fails. Always assert against the alias the production SQL uses, AND check for the underlying index name (`sqlite_autoindex_decisions_1` for V12's PK auto-index) to pin both halves of the contract.
- LEFT JOIN anti-join (`LEFT JOIN b ON b.fk = a.pk WHERE b.<col> IS NULL`) for "rows in A with no match in B": pick a column on the right side that is BOTH the PK AND `NOT NULL` for the WHERE predicate. With V12's `decisions.description_hash TEXT NOT NULL PRIMARY KEY`, `WHERE d.description_hash IS NULL` is the canonical "no match" predicate ‚ÄĒ unmatched LEFT JOIN rows have NULL on every d-side column, but a nullable column risks false positives.
- The TUI's `setup_empty_db` test fixture in `crates/seshat-cli/src/tui/mod.rs` does NOT run the migration framework ‚ÄĒ it's a hand-written `CREATE TABLE` batch that pre-dates V11/V12. When a production SQL change references a new table, the fixture must be patched too. Symptom: query compiles fine, works against `Database::open(":memory:")` (which DOES run migrations), but fails with "no such table" only in legacy-fixture tests. Always grep for `CREATE TABLE nodes` (or whatever the fixture initialises) when adding a join to a new table.
- For `EXPLAIN QUERY PLAN` against a JOIN query, `query_row` only sees the FIRST plan line and misses the join step that's usually load-bearing. Use `query_map` to collect ALL rows, join them with `\n`, and assert against the joined string with `.contains()`.
- When a story's AC names specific test files (e.g. "all assertions in `crates/seshat-detectors/tests/snippet_quality.rs`"), restrict the migration scope to those exact files. Do NOT expand to inline `#[cfg(test)] mod tests` blocks in source files ‚ÄĒ those tests are typically owned by the story that migrates the production function they cover. Strict AC-scoped migration keeps stories independently shippable; broad migration creates a chicken-and-egg with not-yet-migrated production code (e.g. inline tests in `seshat-graph/src/decisions.rs` that pin `ext["source"] == "user"` are the test contract for the legacy `record_decision` function ‚ÄĒ those migrate alongside US-004's production change, not under US-018's test-cleanup banner).

Audit the named files for the AC patterns; if grep returns zero matches, the story is a checkpoint, not a code change. Document the audit findings in the PRD notes so future readers can verify the conclusion without re-running the analysis.

It usually applies only to the named tools (here, `query_*`), not to mutation tools whose request schema MUST change to accept the new identifier. Don't over-constrain by trying to keep `id: i64` as the primary identifier across a hash-keyed schema ‚ÄĒ you'll either invent synthetic-rowid hacks or paint yourself into a corner. The breaking-DB-schema framing of the PRD usually licenses the breaking-MCP-schema change too.

Old consumers keep working; new consumers see the new field; the wire shape is purely additive.
- For "merge results from FTS5 nodes table + LIKE-search of decisions table" with dedup-by-hash: build the dedup key set from the SMALLER vec (typically decisions, project-wide) and skip elements of the LARGER vec whose hash is in the set. The opposite order (set from larger, skip smaller) accidentally drops the wrong winner when both sources have a row.
- "FTS5-search of every relevant table" is tempting but overkill when one of the tables is small (decisions are user-driven, typically ‚Č™1000 rows). A simple `LOWER(description) LIKE %kw%` AND-joined keyword search runs in microseconds against a few hundred rows and avoids FTS5 schema changes (e.g. `conventions_fts.node_id` is TEXT but always-parsed-as-i64 in `fts.rs` ‚ÄĒ adding decisions would have required a synthetic-id scheme to coexist). Pick the simple path unless the table is large enough to need FTS5 ranking.
- When migrating FROM a soft-delete contract TO a hard-delete contract, the obsolete `_marks_as_removed` test is NOT salvageable by tweaking assertions ‚ÄĒ REMOVE it and add a `_hard_deletes_row` test that pins `repo.get_by_hash(...).is_none()` AFTER the delete. Mixing old soft-delete and new hard-delete assertions guarantees future regression risk.

Document the invariant in the doc comment AND pin it with a regression test (`*_preserves_pk_when_*_changes`). The alternative (delete old + insert new under recomputed hash) silently re-keys rows from under callers holding the old hash. Force callers to remove+re-record explicitly when they want to re-key ‚ÄĒ make the lifecycle visible.

- Freshness-check return shape: prefer a 3-variant enum (`UpToDate` / `Stale { old, new }` / `GitUnavailable`) over `Option<String>` for "should we sync?" questions where the *reason* matters. The Stale variant carries the OLD commit (Option<String>: None on a never-scanned branch) so the caller can pass it as a tree-diff hint without re-querying; `GitUnavailable` short-circuits at the call site without conflating "no git" with "no diff" (both would be `None` in the Option-only shape, but they have different downstream semantics ‚ÄĒ UpToDate is "skip", GitUnavailable is "skip", Stale-with-None is "sync but no hint, fall through to fallback_rescan"). The thread-spawn machinery (sync_in_progress flag, ClearOnDrop guard, Database::open guard) is identical regardless of which trigger fired, so writing it once with the resolved hint as input is the least redundant shape.

Don't try to spawn run_serve in a test thread and assert side effects ‚ÄĒ that path is brittle (timeouts, races, stdio mocking) and leads to flaky tests.

- When the AC says "old commit hint (used for gix tree-diff if reachable; otherwise fallback_rescan)", recognise that the existing `resolve_branch_tree_paths` already handles BOTH branch refs AND raw commit hashes via `try_find_reference` ‚Üí `gix::ObjectId::from_hex` fallback. - Optional progress-callback parameter shape: prefer `Option<&dyn Fn(usize, usize)>` over a generic `<F: Fn>` parameter. The trait-object form lets the call site pass a stack-borrowed closure (e.g. one that fetches `AtomicUsize::fetch_add` on a local) without paying the `'static` tax that `Box<dyn Fn>` defaults to. Generics force every caller without a callback to provide a concrete type for the `None` arm ‚ÄĒ `Option::<fn(usize, usize)>::None` is uglier than the trait-object's natural `None`. The per-call dispatch overhead is irrelevant compared to the I/O-bound work the callback is reporting on.
- Progress callback emit cadence: fire AT THE TOP of each loop iteration (before any `continue` paths), then fire one final `(total, total)` tick AFTER the loop. This keeps the UI honest when iterations short-circuit (skipped files, oid-match early-returns, etc.) ‚ÄĒ without the top-of-loop emit, a long run of skipped files would freeze the progress display at the last "real" upsert. The final tick snaps the UI to "X / X" so the user sees completion.
- Progress UI throttle interval: `Duration::from_millis(950)` (not 1000) for "1 Hz" emits. Exact 1-second throttling flickers when subsequent calls land just on either side of the boundary; 950 ms is a stable cadence that's strictly faster-than-1-Hz on the worst case but indistinguishable from 1 Hz to the user. Same logic as the `unixepoch()` test sleeps (1100 ms not 1000 ms) at the other end of the spectrum.
- TTY-vs-piped progress branching: detect via `std::io::IsTerminal` on the stream you'll print to (stderr in our case, not stdout ‚ÄĒ stdout is reserved for MCP/JSON), then emit two flavours: `\rSyncing... Files: X / Y    ` for TTYs (same-line rewrite via carriage return + extra trailing spaces to clear longer previous lines) and one line per emit for piped output (`Syncing files: X / Y`). The two flavours can share the same throttle const but use different printer closures ‚ÄĒ keep the printer factories private (`tty_progress_printer`/`piped_progress_printer` returning `impl Fn`) and let the entry point pick.
- Testable-helper extraction for "entry point ends in interactive TUI / blocking server": when the function ends in a long-running interactive call (TUI launch, MCP serve loop), pull the upstream gate logic into a pub helper that returns a structured outcome enum (e.g. `ReviewSyncOutcome::Skipped | UpToDate | GitUnavailable | Synced { ... }`). Tests lock the helper, NOT the entry point. The entry point becomes a thin orchestrator: gate ‚Üí helper ‚Üí TUI. This is the same pattern US-010 used for `check_branch_freshness` against `run_serve`.

- "Thin wrapper preserves call shape" pattern: when extracting a private helper into a richer pub(crate) helper that another module needs, keep the existing call shape via a 1-line wrapper. `background_sync` stays as-is for all existing call sites; the new `incremental_sync_blocking` adds a `progress: Option<&dyn Fn>` parameter and is called via `incremental_sync_blocking(..., None)` from the wrapper. Callers within the same module that DON'T need progress (e.g. branch-switch handler in `on_branch_switch`) keep using `background_sync` and don't churn. Total touched lines: 1 wrapper + N callers = N+1, vs the alternative "rewrite every caller to pass `None` explicitly" which is N + N for nothing.
- For non-git fixture tests, ALWAYS pre-flight `assert!(!path.join(".git").exists())` BEFORE the test does anything else. On macOS `tempdir()` lands in `/private/var/folders/.../T/` which is not inside any git repo, so the assertion costs nothing ‚ÄĒ but a future change that wrote sources into the workspace dir would silently inherit the project's `.git` and the AC assertion would still pass for the wrong reason. The pre-flight surfaces fixture problems as fixture problems, not as silent regressions in production fallback logic.
- For wire shapes that need both serialisation (internal ‚Üí JSON) and deserialisation (JSON ‚Üí internal), pair a borrow-only `XxxJson<'a>` (only `Serialize`, zero allocations on export) with an owned `XxxJsonOwned` (`Serialize + Deserialize`, owns its strings). Both MUST share the exact same field names and order so the produced JSON is identical between paths. Anchor the inverse contract with a `decision_json_owned_into_decision_round_trips_via_export_format`-style defensive test: serialise the borrow form, deserialise into the owned form, convert back, assert `PartialEq` against the original. If a future field is added to one but not the other, that test fails first and surfaces the drift before any user-visible round-trip starts losing data.
- For "import a set of records with a conflict policy" CLI shapes, return a structured `ImportSummary { total, inserted, updated, skipped }` from the import seam. The CLI renders a summary line; tests assert against the four counters individually (so a regression that conflates "updated" with "skipped" fails loudly even when totals are right). Strict mode MUST pre-flight all incoming hashes via the existing `get_by_hashes` bulk lookup BEFORE any writes happen ‚ÄĒ collect every conflicting hash (sorted, deduplicated) into the error message. The alternative ("abort on first conflict during the write loop") risks partial writes and gives the user only one of N actionable hashes per run.
- For "latest decided_at wins" tie-breaking semantics, use STRICT greater-than (`>`), not greater-or-equal (`>=`). Equal timestamps preserve the incumbent ‚ÄĒ this means a round-trip of unchanged data (export ‚Üí import same DB) is a no-op (`skipped += 1` not `updated += 1`), no `updated_at` churn, and no spurious writes. The "latest wins" framing reads as "newer overwrites older"; equal isn't "newer", so equal preserves. Pin both halves with separate tests (`*_skips_when_existing_is_newer` AND `*_skips_on_equal_decided_at`) so a future refactor that flips to `>=` fails the equal-case test even if the older-case test still passes.
- For round-trip integration tests (export ‚Üí wipe ‚Üí import ‚Üí assert table identical), do BOTH a fresh-DB variant (export from DB-A, import into clean DB-B, assert) AND an in-place-wipe variant (export from DB, DELETE every row, import back, assert). The fresh-DB variant catches "import wiped the wrong fields"; the in-place-wipe variant catches "wipe missed something" or "import accidentally preserved residue from the pre-wipe state". A "table identical" assertion via `Decision`'s derived `PartialEq` (every field, including `decided_at` and `updated_at`) is the strongest available ‚ÄĒ any field drift caused by a future schema migration or serde change fails the round-trip test immediately.
- Verification-only stories ARE legitimate completion shape when the AC items map cleanly to behaviours implemented across earlier stories. Deliverable: a TEST that pins the merged contract end-to-end PLUS PRD notes that map each AC item to its source-of-truth file:line. - For interactive CLI confirmation prompts (`[y/N]`-style), inject the I/O streams via generic `W: Write, R: BufRead` parameters on a small helper instead of reaching directly for `std::io::stdin().lock()` inside the function body. The orchestrator entry-point passes the real stdio handles. Same shape as US-011's progress callback abstraction (`Option<&dyn Fn>`), just for input rather than output.
- For "lookup by full ID or unambiguous prefix" CLI subcommands, prefer a single prefix-filter pass over the full list rather than a two-step "exact match first, prefix search on miss" pattern. A full hash naturally matches exactly one row (PK uniqueness), so the prefix path subsumes both cases without an extra code branch. Min-prefix-length is enforced UP-FRONT before the lookup (not as a disambiguation aid only kicking in on multi-match) ‚ÄĒ even a uniquely-matching short prefix is rejected, because the rule is a CLI-level safety rail meant to prevent accidental near-misses ("ab" matching one row today might match three after the next scan). Errors enumerate the matched short hashes on ambiguous prefixes so the user can lengthen by hand without re-running `decisions list`.
- For CLI subcommands that mutate state (delete, edit, rename) AND have a `--yes` non-interactive escape hatch, expose a `pub fn xxx_with_database(&Database, ...) -> Result<T, CliError>` test seam that bypasses both project resolution AND the prompt ‚ÄĒ the same resolve+mutate path the `--yes` branch invokes. Unit tests target the seam via in-memory DB; integration tests target it against real fixtures. Avoids stdin mocking entirely. Compare to US-011's `prepare_review_sync` (testable layer for an interactive TUI orchestrator).
- Multi-branch git fixtures need `seshat.db` in `.gitignore` BEFORE the initial commit. Without it, scanning on branch A creates/modifies `seshat.db` in the working tree; scanning on branch B does the same; the next `git checkout` between them aborts with "Your local changes to the following files would be overwritten by checkout: seshat.db". The Russian/translated git error is easy to miss without piping `Stdio::piped()` on stderr in the helper. `.gitignore` matches real-project usage anyway (Seshat artifacts are never under VCS) so this is the realistic fixture shape, not a test-only workaround. Pattern: `fs::write(path.join(".gitignore"), "seshat.db\nseshat.db-*\n")` immediately after `git init`, BEFORE `git add .`.
- For test git helpers, capture both stdout AND stderr via `Stdio::piped()` + `Command::output()` and include both in the panic message (`String::from_utf8_lossy`). The default `Stdio::null()` shape hides git error messages ‚ÄĒ failures surface only as "git ['checkout', 'main'] failed" with no diagnostic. Trade: 0 perf cost (helper runs once per test), full debuggability. Localised git output (Russian, Japanese, German) goes through `from_utf8_lossy` cleanly.

Lock the same primitives the production code uses (`check_branch_freshness`, `detect_branch`) so a regression in those primitives breaks the test even if the gate's local logic is unchanged.
- Mirroring helpers must reproduce subtle production contracts EXACTLY. The serve gate's `head_change_hint = old_commit` (not `Some(new_commit)`) means a `Stale { old_commit: None, .. }` does NOT fire the head-change leg ‚ÄĒ only the branch-switch leg can fire for a never-scanned branch. When mirroring, lift the production code line-for-line into the helper's docstring, then implement against the docstring ‚ÄĒ not against your intuition of "what the gate probably does".
- For documentation-only stories (ADR + README + CHANGELOG + smoke-tests) on a code repo with no markdown linter configured, the "lints pass" AC is vacuously satisfied; explicitly document the absence of `.markdownlint*` / `.mdformat*` files in the PRD notes so a future reader doesn't second-guess. **Learnings for future iterations:**
- SQLite's `EXPLAIN QUERY PLAN` uses the FROM-clause **alias**, not the raw table name. If the FROM clause aliases `decisions` as `d`, the plan reads `SEARCH d USING ...` ‚ÄĒ `assert!(plan.contains("SEARCH decisions"))` will fail. Always assert against the alias the production SQL uses, OR include both forms in the OR clause if the production SQL might be reformatted.
- For LEFT JOIN "anti-join" patterns (`WHERE d.<col> IS NULL` to find rows with no match), pick a column that is BOTH the PK on the right side AND NOT NULL. `d.description_hash` works here ‚ÄĒ V12 declares it `TEXT NOT NULL PRIMARY KEY`. If you pick a nullable column you might get false positives (real matches that happen to have a NULL in that column). The PRD AC's `d.id IS NULL` was a leftover from an earlier draft where decisions had a synthetic `id` column; the natural translation to V12 is `d.description_hash IS NULL`.
- When a function uses a const SQL fragment like `SQL_NOT_REMOVED` (which references `ext_data` unqualified) inside a multi-table FROM clause, the unqualified reference is fine ONLY as long as the column name is unique across the joined tables. `ext_data` exists only on `nodes` (not on `decisions`), so `ext_data` resolves unambiguously even with the `nodes n LEFT JOIN decisions d` FROM clause. If a future migration adds `ext_data` to `decisions`, this becomes ambiguous and you'd need to qualify it as `n.ext_data` everywhere or split SQL_NOT_REMOVED into a `with_alias("n")` form.
- The TUI's `setup_empty_db` test fixture in `tui/mod.rs` does NOT run the migration framework ‚ÄĒ it's a hand-written `CREATE TABLE` batch that pre-dates V11/V12. So adding a column or table referenced by production SQL requires patching THIS fixture too. Symptom: a query that compiles fine and works against `Database::open(":memory:")` (which DOES run migrations) fails with "no such table" only in the legacy fixture's tests. Always grep for `CREATE TABLE nodes` (or whatever the fixture initialises) when adding a join to a new table.
- For unit tests that use `EXPLAIN QUERY PLAN`, `query_map` returns multiple rows (one per join step). Collect all rows with `.query_map().unwrap().map(|r| r.unwrap()).collect()` and join them with `\n` to inspect the full plan, then assert against the joined string with `.contains()` ‚ÄĒ a single-row `query_row` only sees the first plan line and misses the join step that's usually load-bearing.
- `LEFT JOIN d ON d.<pk> = n.<pk>` + `WHERE d.<pk> IS NULL` is the canonical "anti-join" and SQLite optimises it well when the right side has a PK auto-index. The alternative `WHERE n.<pk> NOT IN (SELECT <pk> FROM d)` typically requires a second scan or temp B-tree and is slower at scale. EXPLAIN QUERY PLAN confirms the JOIN form gets `SEARCH d USING COVERING INDEX sqlite_autoindex_decisions_1 (description_hash=?) LEFT-JOIN` ‚ÄĒ no temp B-tree on the join itself (the `USE TEMP B-TREE FOR ORDER BY` is for the outer `ORDER BY n.confidence DESC`, unavoidable without a confidence index).
- The LEFT JOIN's `ON d.description_hash = n.description_hash` matches NULL == NULL as *unknown*, NOT *true*. So an auto-detected node with `description_hash = NULL` (pre-V8 legacy row) gets NULL on the d side regardless of whether any decision row has NULL hash. The WHERE `d.description_hash IS NULL` keeps it. This is the desired behaviour: legacy NULL-hash nodes can't match any decision (decisions PK is NOT NULL), so they shouldn't be filtered out. The defensive `query_conventions_for_review_keeps_undecided_with_null_hash` test pins this ‚ÄĒ without it, a future "fix" that switches to `INNER JOIN ... d.description_hash IS NULL` (which is unsatisfiable) or `WHERE n.description_hash IS NOT NULL AND d.description_hash IS NULL` would silently drop legacy rows.
- When the PRD AC says `WHERE d.id IS NULL` but the actual schema has no `id` column, don't blindly add an `id` column to the schema ‚ÄĒ translate the AC to the closest equivalent in the actual schema. The intent is "no matching decision row", and `d.<PK> IS NULL` expresses that with one fewer column. Document the deviation in the commit message and PRD notes so future readers don't think the AC was ignored.

**Learnings for future iterations:**
- When the same behaviour must fire at the end of multiple scan paths owned by different crates, put the helper in `seshat-scanner` ‚ÄĒ not `seshat-cli`. A 30-second scan that just succeeded should not regress to a hard failure because writing a single sentinel row hit a transient SQLite contention. Storage errors get `warn!`-logged for ops visibility; git-unavailable is `debug!`-logged because it's an expected branch (no warning fatigue for users running outside git). Two distinct levels reflect two distinct events.
- AC items phrased as "X invocations triggered by Y" can be ambiguous when X is called from multiple callers and "Y" is one of them. Audit the callers (workspace-wide grep for the function name), enumerate them, and check each against the trigger description. If the AC item turns out to be already satisfied by callers covered under other AC items, document the audit in PRD notes ‚ÄĒ don't invent a new wiring point that would lie about state. The warm-tier in this codebase doesn't actually re-scan the filesystem; it re-runs detection on existing IR. A "scan completion" sentinel there would be a falsehood.

**Learnings for future iterations:**
- For "should we sync?" gate decisions where the *reason* affects what to pass downstream, prefer a 3-variant enum (`UpToDate` / `Stale { old, new }` / `GitUnavailable`) over `Option<String>`. The Stale variant carries the OLD commit (Option<String>: None on a never-scanned branch) so the caller can pass it as a tree-diff hint without re-querying. The GitUnavailable variant short-circuits at the call site without conflating "no git here" with "no diff to do" ‚ÄĒ both would collapse to None in the Option-only shape, but they have distinct downstream semantics. Match arms at the call site stay self-documenting.
- Branch-switch-and-HEAD-change union is one boolean (`needs_sync = a.is_some() || b.is_some()`) plus one resolved hint (`old_hint = a.or_else(|| b)`). Don't nest two `if let Some(...)` blocks for the spawn ‚ÄĒ the conditions are independent triggers and flattening makes both paths visible at the same indent level. The thread-spawn machinery (sync_in_progress flag, ClearOnDrop guard, Database::open guard) is identical regardless of which trigger fired, so writing it once with the resolved hint as input is the least redundant shape.
- When two earlier-computed values become a duplicate of a later block's local computation (e.g. `sync_root` was inlined inside `if sync_old_branch.is_some()`), lift the value up so all downstream blocks share it. Don't synthesise a third copy for the new feature ‚ÄĒ that's the path to drift. Verify the lift doesn't accidentally fold in nearby variables that LOOK identical but have semantic differences (`gc_repo_path` could be folded; `project_root` cannot ‚ÄĒ `project_root` uses cwd directly to keep worktree paths intact, where `find_git_root` would walk up to the main repo).
- For long-running entry points like `run_serve` that can't be exercised end-to-end in tests (they wait on Ctrl+C or serve stdio forever), refactor the in-startup decision logic into a pure helper exposed via the public API of the right crate (here, `seshat-scanner` because it owns the gix bridge), and test the helper directly. Document in the integration test that the helper's return value is what the entry point keys off ‚ÄĒ so a reader knows the test pins the AC at the right layer. Don't try to spawn `run_serve` in a test thread and assert side effects ‚ÄĒ that path is brittle (timeouts, races, stdio mocking) and leads to flaky tests.
- "Test the trigger condition" is a legitimate stand-in for "test the triggered behaviour" when the trigger and the spawn are connected by a single boolean and the spawn machinery is tested independently elsewhere. The PRD's "assert background_sync was triggered" is satisfied by asserting the gate returns Stale ‚ÄĒ that's necessary AND sufficient for the spawn (`let needs_sync = ... || head_change_hint.is_some(); if needs_sync { thread::spawn(...) }`). - When the AC says "old commit hint (used for gix tree-diff if reachable; otherwise fallback_rescan)", recognise that `resolve_branch_tree_paths` already handles BOTH branch refs AND raw commit hashes via `try_find_reference` ‚Üí `gix::ObjectId::from_hex` fallback. So a freshness-check OLD commit hash flows through the same `Option<&str>` parameter that branch switches use ‚ÄĒ no new code path is needed in `background_sync`. Just resolve the hint at the call site and let the existing fallback chain handle the rest.
- Reading HEAD is cheap, but checking it on every `run_serve` startup is one extra gix open + one extra SQLite SELECT. The total cost (~ms) is negligible relative to the savings (a single stale-cache MCP response that the user has to debug after `git pull`). Don't over-engineer the caching ‚ÄĒ measure first if it ever becomes a hot path.
- Tracing fields `old_head` and `new_head` truncated via `.chars().take(7).collect::<String>()` (NOT `&hash[..7]`) ‚ÄĒ the latter panics on non-ASCII inputs. Hash strings are ASCII hex, but the `.chars()` form is robust to future schema changes (e.g. branch IDs with unicode) and is already the canonical `seshat` style for "first N chars" displays.
- Negative-case test for git-unavailable + present-but-stale sentinel: even when a sentinel is recorded for a branch, a fresh `check_branch_freshness` against a directory without `.git` MUST return `GitUnavailable` (not Stale). **Learnings for future iterations:**
- `Box<dyn Fn(usize, usize)>` defaults to `Box<dyn Fn(usize, usize) + 'static>`, so a Box capturing a stack-borrowed reference (e.g. `&AtomicUsize` from the enclosing function) won't compile. Two ways out: (a) use a stack closure directly (`let cb = |...| { ... };` then `Some(&cb)`) ‚ÄĒ no Box needed, `Option<&dyn Fn>` accepts any lifetime via elision; or (b) annotate the lifetime explicitly (`Box<dyn Fn(...) + 'a>`). The stack-closure form is simpler and was the right call here.
- `prepare_review_sync` captures the `progress_callback` parameter from the outer fn into the internal counted closure. Closures borrow by default, and the `progress_callback: Option<&dyn Fn(...)>` parameter has a lifetime tied to the fn ‚ÄĒ moving the closure into `Some(&counted_cb)` works because `counted_cb` lives on the same stack frame. No `move` keyword needed.
- For `pub fn` helpers that accept an optional callback, prefer `Option<&dyn Fn(...)>` over `<F: Fn>` generics. The trait-object form lets call sites pass stack-borrowed closures without `'static` constraints, doesn't force callers without a callback to invent a concrete `fn` type for the `None` arm, and the dispatch overhead is irrelevant against I/O-bound work the callback is reporting on.
- When extracting a private helper into a richer pub(crate) helper with a new optional parameter, keep the existing call shape via a 1-line wrapper for in-module callers. `background_sync` stays as-is for everyone in `serve.rs`; the new `incremental_sync_blocking` adds the progress parameter and is called via `incremental_sync_blocking(..., None)` from the wrapper. Callers that don't need progress don't churn.
- TTY detection for progress UI: `std::io::stderr().is_terminal()` (not stdout ‚ÄĒ stdout is reserved for MCP/JSON output across the workspace, and `seshat review` is no exception). The two flavours (TTY same-line `\r` rewrite + final `done.` line; piped one-line-per-emit) share the same throttle constant but diverge in their printer closures. Pad the rewrite line with trailing spaces (`Files: X / Y    `) to clear longer previous lines ‚ÄĒ without padding, "Files: 100 / 200" leaves a stray "0" when the next emit prints "Files: 5 / 200".
- Throttle interval for "1 Hz" progress: `Duration::from_millis(950)`, NOT 1000. Exact 1-second throttling flickers when consecutive calls land on either side of the boundary. Same shape as the `unixepoch()` test sleeps (1100 ms not 1000 ms) at the other end of the spectrum ‚ÄĒ boundary-buffer matters.
- Progress callback emit point: at the TOP of the loop iteration (BEFORE any `continue` paths), then a final `(total, total)` tick AFTER the loop. The top-of-loop emit keeps the UI advancing through skipped-files runs (oid-match early-returns, extension-mismatch continues, exclude-glob skips). Without it, a stretch of 50 skipped files freezes the progress display at the last "real" upsert and the user thinks the sync hung. The final tick snaps the UI to "X / X" so the user sees completion.
- Adding ONE more parameter that pushes a function past clippy's 7-arg `too_many_arguments` cap: reach for `#[allow(clippy::too_many_arguments)]` on the function. A Params struct is only worth the ceremony when arguments cluster semantically (e.g. all "filter" options, all "config" knobs). For `incremental_sync_blocking` the 8 args are root + branches + db + branch_id + 2 configs + callback ‚ÄĒ none of which group naturally. If the arg list grows further or a struct-variant call site emerges, revisit.
- For test-driven proof that "sync ran AND TUI received fresh data": instead of stubbing the TUI (impractical without a fake terminal), assert against the SAME `Arc<Mutex<Connection>>` handle the sync wrote into. The TUI reads from that handle when launched in production, so a post-sync `files_ir` query that finds the new file IS the test for "fresh data". Keep the assertion at the storage layer ‚ÄĒ it's stable across TUI evolutions.
- "ADD a new file" is a cleaner integration-test signal than "MODIFY an existing file" for diff sync. Modifying changes the tree-oid (good signal) but the assertion has to find a specific blob hash. Adding a new path is unambiguously a non-skip in the diff (new key in `new_paths`, absent from `old_paths`) and the assertion is a simple `files.iter().any(|(p, _)| p.ends_with("util.rs"))`. Pick the simpler signal when both work.
- Mirror `FreshnessCheck`'s variants in `ReviewSyncOutcome`. Don't collapse them into a single `Result<bool, _>` ‚ÄĒ the four cases (Skipped vs UpToDate vs GitUnavailable vs Synced) have distinct UX implications: Skipped is "user explicitly opted out"; UpToDate is "we'd have synced if the gate hadn't said pass"; GitUnavailable is "we can't sync but that's expected"; Synced is "we just did real work". Tests that assert `outcome == UpToDate` would silently lose coverage if the production code regressed UpToDate to Skipped.
- For testable-helper extraction from interactive-TUI entry points: the helper's return value should encode every decision the entry point makes downstream. `prepare_review_sync` returns `Synced { progress_emits, ... }` not just `Synced` because the AC requires "progress callback emits at least one update" ‚ÄĒ putting that count in the outcome lets the integration test assert it without a side-channel atomic. The outcome type is the integration-test contract; design it for the assertions you need, not just for the production code path.

**Learnings for future iterations:**
- Borrow-vs-owned JSON DTO pair: `DecisionJson<'a>` (borrow, Serialize-only) for export; `DecisionJsonOwned` (owned, Serialize+Deserialize) for import. The borrow form sidesteps allocations on the hot serialise path (export reads from already-loaded `Decision` rows); the owned form is the only practical shape for `serde_json::from_str` (try to deserialize into `&str` fields and you fight Cow lifetimes for nothing). Both share field names and order so the produced JSON is identical ‚ÄĒ the `decision_json_owned_into_decision_round_trips_via_export_format` defensive test pins this contract: serialise via the borrow form, deserialise into the owned form, convert back, `assert_eq!`. Without the test, a future field added to one but not the other would silently break the round-trip starting at the next export.
- Strict mode pre-flight pattern: when a CLI command has both a "soft" mode (apply policy per-row) and a "strict" mode (abort on any conflict), the strict mode MUST do its check BEFORE any writes, in a single bulk query, and report ALL conflicts (sorted, deduplicated) ‚ÄĒ not just the first. The existing `SqliteDecisionRepository::get_by_hashes` chunked bulk lookup is the right hammer: one SQL roundtrip for up to 500 hashes per chunk, so even thousand-row imports stay fast. The failure message carries every conflicting hash so the user can resolve them all in one pass. The alternative ("abort on first conflict during the write loop") risks partial writes AND gives the user only 1/N actionable hashes per run.
- Equal-timestamp tie-breaker: for "latest decided_at wins" semantics, use STRICT greater-than (`>`), not greater-or-equal (`>=`). Equal-timestamp imports preserve the incumbent ‚Üí a round-trip of unchanged data is a no-op (`skipped += 1`, not `updated += 1`), no `updated_at` churn, no spurious writes. Pin BOTH halves: a `*_skips_when_existing_is_newer` test (older incoming) and a `*_skips_on_equal_decided_at` test. Without the equal-case test, a future refactor that flips `>` to `>=` fails the older-case test only, and the equal-case behaviour silently regresses.
- Round-trip tests: do BOTH a fresh-DB variant (export from DB-A, import into clean DB-B, assert identical) AND an in-place-wipe variant (export from DB, DELETE all rows, import back, assert identical). The fresh-DB variant catches "import wiped the wrong fields"; the in-place-wipe variant catches "import accidentally preserved pre-wipe residue" (a regression where the upsert path reads a stale field from somewhere outside the JSON). The `Decision` derived `PartialEq` across every field ‚ÄĒ including `decided_at` AND `updated_at` ‚ÄĒ is the strongest available "table identical" assertion, and it's free.
- Test seam pair pattern for file-based CLI commands: expose TWO public helpers ‚ÄĒ one string-level (`*_decisions_to_string` / `*_decisions_from_str`) and one file-level (`run_export(&Path)` / `run_import(&Path, bool)`). Tests drive the string-level seam (no I/O fixtures, no tempdirs). The file-level helpers are thin: open file ‚Üí call string-level helper ‚Üí render summary. This keeps the test surface small and focused on policy (conflict resolution, validation), while the file I/O is exercised once by the integration test that needs a real `tempdir()` round-trip.
- For partial-write defence: when a strict-mode failure aborts mid-payload, assert in the test that NONE of the payload's rows landed (not just that the conflicting one didn't change). The `import_decisions_from_str_strict_lists_all_conflicts` test puts a NON-conflicting hash in the same payload as conflicting ones and asserts `repo.get_by_hash(non_conflicting_hash).is_none()` after the failure. Without that assertion, a future refactor that did the strict pre-flight AFTER inserting the non-conflicting row would still pass the conflict-list test.
- `cargo clippy --all-targets` is the right gate (not just `cargo clippy`). Tests live under `--tests`/`--examples` and have their own clippy lints (e.g. `needless_borrows_for_generic_args` flagged a `&repo.join("seshat.db")` only inside the integration test, not in the production code). Run with `--all-targets -- -D warnings` BEFORE committing ‚ÄĒ otherwise CI's stricter gate fails after the local one passed.

**Learnings for future iterations:**
- For doc-only completion stories (ADR + README + CHANGELOG + smoke-tests), the "lints pass" AC may be vacuously satisfied when the repo has no markdown linter configured ‚ÄĒ explicitly document the absence in PRD notes so a future reader doesn't second-guess and re-run a non-existent check. The "no broken internal links" half is checkable mechanically: collect every relative path from each new doc, then `for f in $paths; do test -e "$f" && echo OK || echo MISS; done`. A single typoed path is easy to miss eyeballing 100 lines of prose.
- For doc-only commits, `cargo test --workspace` is a smoke check, not the primary verification ‚ÄĒ the source tree is unchanged so results MUST match HEAD. ALWAYS confirm any failure reproduces with the new files stashed away (`git stash --include-untracked`) before claiming success or re-investigating. Pre-existing flakes documented in earlier progress entries (e.g. `serve_starts_and_shows_startup_info`) reproduce identically and are not the doc author's problem.
- When the AC says "README updated" and no README exists, create one with the AC-mandated content (here: `seshat decisions` reference). Don't try to write a comprehensive project README from scratch ‚ÄĒ keep the new file scoped to install + quick-start + the AC-mandated section + minimal connective tissue. A future feature-rich README is somebody else's story.
- ADRs in this codebase don't have a single template ‚ÄĒ `_bmad-output/planning-artifacts/story-11-1-branch-detection-worktree.md` uses a "Status / Story / Acceptance Criteria" shape, while `_bmad-output/implementation-artifacts/11-2-branch-switch-adr14.md` mirrors the same. For a retrospective ADR (documenting work already shipped), the more familiar "Context / Decision / Consequences / Implementation Notes / References" shape is clearer because it's organising rationale, not driving implementation. Pick the shape that matches the document's purpose, not the loudest precedent.
- CHANGELOG entries for "BREAKING with mechanical recovery" should split Breaking ‚Üí Added ‚Üí Fixed ‚Üí Known Limitations and put the recovery command (`rm <db> && rescan`) in the FIRST sentence of the FIRST Breaking bullet. Users skim CHANGELOGs; if the recovery action is buried under three paragraphs of rationale, it gets missed and the upgrade fails. The rationale belongs in the linked ADR.
- Smoke-test docs that map US-N ‚Üí manual steps benefit from a "Quick everything-still-works sanity check" section at the end (`cargo test --workspace`, `cargo fmt`, `cargo clippy`) as a circuit-breaker: if the automated checks pass, the smoke-tests are mostly verifying user-visible polish (output, prompts, log messages) rather than core correctness. Without the circuit-breaker, a human running smoke-tests after every commit is wasteful ‚ÄĒ they'd be re-deriving the test suite by hand.
- Mapping AC items to source-of-truth file:line ranges in PRD notes (US-018 / US-012 / US-019 pattern) makes verification-only stories AND doc stories trivially auditable. A future reader needs only `test -e <path>` and `grep -n <anchor>` to confirm completion ‚ÄĒ not a re-read of the original prose. Build the audit map as you write, not after.
