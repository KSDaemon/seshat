## Codebase Patterns
- Edition 2024, Rust 1.85+. `let-else`, `..=` ranges, and similar modern syntax are fine.
- `seshat-graph/src/dependencies.rs` already has a `SuffixIndex` (O(1) reverse-suffix lookup) and an `infer_package_root` heuristic for cross-crate guards. Reuse instead of rolling new ones.
- IR `path` is stored as an absolute `PathBuf`; `target_path` arguments to public APIs are typically relative — always normalise via `normalize_path` + `suffix_matches_at_boundary` before comparing.
- For new pure-graph utilities (BFS, adjacency builders), construct synthetic `HashMap<String, Vec<…>>` directly in unit tests; no DB / IR roundtrip needed. The repo-level `test_helpers::{test_conn, insert_ir}` is for integration-style tests.
- Rust style: snake_case fns/params, PascalCase types, `expect("message")` over `unwrap()` in tests, doc comments on every `pub` item with `///`.
- `DependentEntry` and similar serde types use `#[serde(default)]` on newly-added fields for forward compatibility within the same schema version.
- Clippy is run as `cargo clippy --all-targets -- -D warnings` and CI gates on it — no warnings tolerated.
- `DependentEntry` / `DependencyEntry` JSON field is `file_path` (NOT `path`). Tests indexing into the response envelope must use `dep["file_path"]`, since `DependencyData.target` is the only top-level `path`/`target` string.
- MCP `Edit replace_all` over a fragment that ends in `}));` matches every request struct in `server.rs`, not just the one you intend. Always anchor the `old_string` to a unique sibling line (e.g. the `path:` value) when bulk-rewriting in a multi-tool integration test file.

- **Learnings for future iterations:**
  - `DependencyData` derives `Serialize` only (no `Deserialize`).
    Using `#[serde(default = "fn_name")]` on new fields would emit a
    `dead_code` warning because the default function is never called
    by serialize. Plain `#[serde(default)]` is safe and matches the
    existing pattern used by `truncated`/`depth`/`via`.
  - The `direct_count` for blast-radius classification at `depth >= 2`
    must be derived from the BFS result by filtering on
    `entry.depth == 1`. Don't try to call `build_dependents` separately
    — that re-walks the full IR and discards the amortisation benefit
    of the reverse-adjacency map.
  - `query_dependencies_batch` with `depth > 1` should build the
    reverse adjacency **once** and reuse it across targets. Each
    target's `compute_transitive_dependents(target, &reverse, depth)`
    is then O(BFS) instead of O(IR + BFS).
  - `MAX_TRANSITIVE_DEPTH = 10` was chosen as the upper bound. The
    cap matters because BFS at unbounded depth on a dense graph could
    push past `MAX_DEPENDENTS = 500` for almost every node.
  - When updating many test-callsite signatures, a single `perl -i -pe`
    run across the file is faster than dozens of `Edit` calls. The
    pattern is uniform (`query_dependencies(&conn, "main", "...")`)
    so a regex captures it cleanly without affecting `_batch`.

- **Learnings for future iterations:**
  - `DependentEntry` / `DependencyEntry` serialise their file path as
    `file_path`, not `path`. The only `target` / top-level path in the
    response envelope is `data.target`. Tests that index into
    `data.dependents[i]["path"]` will silently `as_str() == None` and
    panic on `unwrap()`. Always check `Serialize` derive output before
    asserting on JSON keys.
  - `Edit replace_all` over a fragment ending in `}));` matches every
    request struct in `server.rs` (around 45 different request types
    end the same way). Always anchor `old_string` to a unique
    sibling line — e.g. the `path:` value or a unique earlier field —
    when bulk-rewriting integration test files with many request types.
    Recovery: `git checkout -- file.rs` and redo with anchored Edits.
  - `map_graph_error` exposes only one "InvalidInput" suggestion
    (`"Check the nature and weight parameter values"`), which is wrong
    for parameters that aren't `nature`/`weight`. When the AC requires
    a domain-specific suggestion, validate at the tool-handler layer
    BEFORE calling into the graph layer and emit the suggestion
    yourself via `ErrorEnvelope::new(... ErrorCode::InvalidInput, msg,
    suggestion)`. The graph-layer validation still runs as a
    defence-in-depth check.
  - When extending an MCP request struct with a new `Option<T>` field,
    `schemars` will emit a schema-level description from the
    `#[schemars(description = ...)]` attribute. Always include both
    the range and the default value in that description so the MCP
    schema (which agents see) is self-explanatory — the
    `#[tool(description=...)]` on the handler covers the semantics,
    but the field-level schemars description is what shows up in
    auto-generated MCP tool docs.
  - The `DEFAULT_TRANSITIVE_DEPTH = 3` ≠ graph-layer
    `QueryDependenciesOptions::default().depth = 1`. The mismatch is
    intentional: the graph-layer default preserves direct-only
    semantics for in-tree callers (validate_approach,
    diff_impact:378), while the MCP-layer default exposes the
    transitive-by-default UX to AI agents. Anyone touching either
    constant needs to think about which surface they're changing.
