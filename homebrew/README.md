# Homebrew Distribution

This directory holds the Homebrew formula template and bootstrap notes for
shipping `seshat` via Homebrew. The end-user install path is:

```bash
brew tap KSDaemon/seshat
brew install seshat
```

## How it works

1. The `Release` workflow builds platform binaries and ships
   `seshat-<triple>.tar.gz` archives (each archive includes a
   `completions/` subfolder with bash/zsh/fish/powershell/elvish scripts
   pre-generated by the binary itself).
2. When the GitHub Release is `published`, the `Homebrew Bump` workflow
   (`.github/workflows/homebrew-bump.yml`) downloads the four Unix-y
   tarballs (no Homebrew on Windows), computes SHA256s, renders
   `homebrew/seshat.rb` with substituted version + SHAs, and pushes the
   result to `KSDaemon/homebrew-seshat` as `Formula/seshat.rb`.
3. End users `brew tap KSDaemon/seshat && brew install seshat` and Homebrew
   picks up the matching arch tarball, drops the binary into `bin/`, and
   wires up shell completions via `bash_completion`/`zsh_completion`/
   `fish_completion` helpers.

## One-time bootstrap

You need to do these steps **once** before the first release that should
appear in the tap.

### 1. Create the public tap repository

Repository name: **`homebrew-seshat`** (Homebrew enforces the
`homebrew-<name>` prefix; users address it without the prefix:
`brew tap KSDaemon/seshat`).

```bash
gh repo create KSDaemon/homebrew-seshat --public \
  --description "Homebrew tap for KSDaemon/seshat"
```

The bump workflow creates `Formula/seshat.rb` automatically on the first
push, so the repo can start empty. Optionally seed it with a README that
points users at the install snippet.

### 2. Create a Personal Access Token for the bump workflow

The default `GITHUB_TOKEN` issued to workflows can only write to the
repository it ran in. To push to a *different* repo (the tap), we need a
token whose scope reaches `KSDaemon/homebrew-seshat`. Two options:

- **Fine-grained PAT** (recommended): create at
  <https://github.com/settings/tokens?type=beta>, restrict to repository
  `KSDaemon/homebrew-seshat`, grant `Contents: Read and write`.
- **Classic PAT**: create at <https://github.com/settings/tokens>,
  scope `repo` (broad — gives access to all your private repos).

Then add it as a secret in the **source** repo (this one, `seshat`) under
**Settings → Secrets and variables → Actions → New repository secret**:

- Name: `HOMEBREW_TAP_TOKEN`
- Value: the PAT from above

Once the secret exists, every release published to this repo will update
the tap. Without the secret the bump workflow logs a warning and skips —
the rest of the release pipeline ships unaffected.

### 3. Cut a release

Tag a commit (`git tag v0.1.2 && git push origin v0.1.2`). The `Release`
workflow builds binaries and uploads them; once it publishes the
GitHub Release, `Homebrew Bump` fires automatically.

Verify locally:

```bash
brew tap KSDaemon/seshat
brew install seshat
seshat --version
```

## Local formula development

Editing `homebrew/seshat.rb` and want to validate without going through a
release cycle?

```bash
# Audit syntactic correctness against Homebrew rules.
brew audit --strict --formula ./homebrew/seshat.rb

# Install from the local file (uses the version + SHA placeholders, so
# this only works after rendering the template manually for an existing
# release tag).
brew install --build-from-source ./homebrew/seshat.rb
```

For a true local end-to-end test, point a temporary tap at a checkout:

```bash
mkdir -p "$(brew --repo)/Library/Taps/ksdaemon"
ln -s "$PWD" "$(brew --repo)/Library/Taps/ksdaemon/homebrew-seshat-local"
brew install ksdaemon/seshat-local/seshat
```

## Re-running a bump manually

If a tap update fails (network blip, secret typo, fresh tap repo) and you
fix the cause, re-trigger via **Actions → Homebrew Bump → Run workflow**
and pass the desired tag (e.g. `v0.1.2`).
